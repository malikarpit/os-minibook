2.1 Process Concepts & Memory Layout

Process
A process is a program in execution — a dynamic entity with its own address space, CPU registers, open files, and OS-managed resources. A program (static binary) becomes a process the moment the OS creates a PCB for it and loads it into memory.

Exam Trick: Stack grows downward (toward lower addresses). Heap grows upward. They meet in the middle if memory is exhausted → stack overflow or malloc returns NULL. SIGKILL cannot be caught; SIGTERM can.

Process Creation — fork() & exec()

C — fork() + exec() pattern
pid_t pid = fork();    // Duplicate current process

if (pid < 0) {
    perror("fork failed");  // Fork failed (resource limit?)
    exit(1);
}
if (pid == 0) {
    // CHILD: fork() returns 0
    execvp("ls", args);    // Replace image with 'ls'; no return on success
    _exit(1);              // Only reached if exec fails
} else {
    // PARENT: fork() returns child's PID (> 0)
    int status;
    waitpid(pid, &status, 0);  // Wait for child; prevents zombie
    printf("Child exited: %d\n", WEXITSTATUS(status));
}

🎓 University
Q: What distinguishes a zombie process from an orphan process?
AnswerA zombie has finished but its parent is still running and hasn't called wait() — the PCB stays in the process table to preserve exit status. An orphan is still running but its parent has terminated — Linux automatically re-parents it to init (PID 1), which calls wait() periodically. Zombies waste PIDs; orphans are handled automatically.

//   → Child of first fork calls second; parent does not → 3 processes

if (fork() == 0) {   // Only CHILD enters
    fork();          // Grandchild created
}

// Process tree: Main(prints) → Child(prints) → Grandchild(prints)
// Answer: 3 prints total
Pattern 4 — Fork in loop with break
for (int i = 0; i < 3; i++) {
    if (fork() == 0) break;  // Child breaks out of loop!
}
printf("hello\n");
// Each child only forks once then breaks.
// Iteration 0: Main forks → Child-1 breaks (1 process extra)
// Iteration 1: Main forks again → Child-2 breaks (2 processes extra)
// Iteration 2: Main forks → Child-3 breaks (3 processes extra)
// + Main itself → Total: 4 processes, 4 prints
🔬 Advanced
🔬 Copy-on-Write (COW): When fork() is called, the child does NOT immediately get copies of all pages. Instead, both parent and child share the same physical pages marked read-only. Only when either process writes to a page does the OS create a private copy for the writer. This makes fork() nearly instantaneous for large processes. CoW is why Redis can fork for background saves without doubling memory immediately.
🔬 ASLR (Address Space Layout Randomisation): Linux randomises the base addresses of stack, heap, and shared libraries at process creation. This prevents attackers from reliably jumping to known addresses in buffer overflow exploits. Disable with echo 0 > /proc/sys/kernel/randomize_va_space (for debugging only).

Context Switch

Context switch = pure overhead. During a context switch, no user work is done — the CPU is fully occupied saving and restoring register state. On modern hardware, a context switch takes ~1–10 μs. High-frequency context switches (tiny RR quantum) can waste 10-30% of CPU time on pure switching overhead.
Context Switch Steps (simplified)
StepAction
1Timer interrupt fires (or process yields/blocks)
2CPU switches to kernel mode
3Save CPU registers of running process into its PCB
4Scheduler selects next process from ready queue
5Load CPU registers from next process's PCB
6Update MMU (TLB flush if different address space)
7Return to user mode — new process runs

C — POSIX pthreads example
#include <pthread.h>
#include <stdio.h>

void* worker(void* arg) {
    int id = *(int*)arg;
    printf("Thread %d running\n", id);
    return NULL;
}

int main() {
    pthread_t t1, t2;
    int id1 = 1, id2 = 2;

    pthread_create(&t1, NULL, worker, &id1);  // Create thread 1
    pthread_create(&t2, NULL, worker, &id2);  // Create thread 2

    pthread_join(t1, NULL);  // Wait for t1 to finish
    pthread_join(t2, NULL);  // Wait for t2 to finish
    return 0;
}
⚡ GATE

Worked GATE-Style Question — Thread Models

Q: A system uses a many-to-one threading model. Thread T1 makes a blocking system call. Which of the following is true?
  1. Only T1 blocks; other threads continue
  2. The entire process blocks
  3. The kernel creates a new kernel thread
  4. T1 is moved to the waiting state while others remain in running
View Answer
Answer: B — In the many-to-one model, all user threads map to a single kernel thread. When that kernel thread makes a blocking system call, the OS blocks the one kernel thread — and since all user threads run through it, the entire process blocks. This is the fundamental weakness of many-to-one.
⚡ GATE Key Points:
  • Linux pthreads = one-to-one (NPTL — Native POSIX Thread Library). Each thread is a kernel-scheduled task.
  • Thread creation is faster than process creation (no address space copy), but context switch overhead is similar.
  • Threads share heap → race conditions possible. Must use mutex/semaphore for shared data.
  • pthread_join() is to threads as wait() is to processes — prevents zombie threads.
🔬 Advanced
🔬 Go Goroutines — M:N Threading Done Right: Go implements M:N threading at the language runtime level. The Go scheduler (GOMAXPROCS-controlled) multiplexes goroutines onto OS threads. When a goroutine makes a blocking system call, the runtime parks it and runs another goroutine on the same OS thread. This gives the appearance of thousands of concurrent goroutines without thousands of OS threads. The Go scheduler uses work-stealing: idle OS threads steal goroutines from busy threads' run queues, achieving excellent load balance with minimal synchronization overhead.
🔬 Thread-Local Storage (TLS): TLS provides per-thread variables that appear global but have a separate copy for each thread. In C: __thread int counter;. In pthreads: pthread_key_create(). Use cases: errno (must be per-thread so concurrent syscalls don't clobber each other), database connection pools, per-thread random seeds.

2.4 IPC — All 5 Mechanisms

Processes are isolated by default (separate address spaces). IPC (Inter-Process Communication) provides controlled ways for processes to exchange data.

🔧
Pipes
Medium speed
Unidirectional byte stream. Anonymous (parent↔child) or named (FIFO, any processes).
⚡
Shared Memory
Fastest (no kernel copy)
Map same physical pages into multiple address spaces. Needs semaphore for sync.
📬
Message Queues
Medium speed
Kernel-managed queue of typed messages. Persists until explicitly deleted.
🔔
Signals
Instant (async)
Asynchronous notification. Small — just a signal number. No data payload.
🌐
Sockets
Flexible (network-capable)
Bidirectional. Works across machines. Unix domain sockets for local IPC.

1. Pipes

C — Pipe (parent→child)
int fd[2];
pipe(fd);           // fd[0]=read end, fd[1]=write end

if (fork() == 0) {  // CHILD
    close(fd[1]);   // close unused write end
    char buf[64];
    ssize_t n = read(fd[0], buf, sizeof(buf) - 1);
    if (n > 0) { buf[n] = '\0'; printf("Child got: %s\n", buf); }
    close(fd[0]);
    _exit(0);
} else {            // PARENT
    close(fd[0]);   // close unused read end
    const char *msg = "Hello from parent";
    write(fd[1], msg, strlen(msg));
    close(fd[1]);   // close write end → child sees EOF
    wait(NULL);
}

GATE IPC Speed Ranking: Shared Memory > Message Queue ≈ Pipes > Sockets. Shared memory wins because after initial setup, data transfer requires NO kernel involvement — processes read/write directly to the mapped pages. All other mechanisms require at least one kernel copy (kernel ↔ user space).
GATE-style Q: Which IPC mechanism requires synchronisation primitives to prevent data races?
(a) Pipe   (b) Message Queue   (c) Shared Memory   (d) Signals
Answer(c) Shared Memory — Pipes and message queues have built-in serialisation (read blocks until data available; write blocks if buffer full). Shared memory has no such protection — two processes can simultaneously read/write the same bytes, causing corruption. You must use semaphores or mutexes to coordinate access.

Why SJF is Optimal (Non-Preemptive): Among all non-preemptive algorithms, SJF minimises average waiting time. Burst Time Prediction — Exponential Averaging

Next Burst Prediction (SJF in practice)
τn+1 = α · tn + (1−α) · τn

τn+1 = predicted next burst  |  tn = actual last burst
τn = previous prediction  |  α ∈ [0,1] (typically 0.5)

α=0 → ignore recent history (use only old prediction)
α=1 → use only most recent burst
α=0.5 → equal weight (most common in GATE)
GATE Trick: SJF can cause starvation — long jobs may never run if short jobs keep arriving. The fix is Aging: gradually increase the priority of waiting processes. SJF/FCFS cannot starve jobs already in the queue for FCFS; SJF can starve long processes if short ones keep arriving.

Priority Inversion — The Full Story

Priority Inversion: A high-priority task H is blocked waiting for a lock held by low-priority task L. Medium-priority task M (which doesn't need the lock) preempts L because M > L. Now H is effectively blocked behind M, even though H > M. This inverts the intended priority ordering.
🚀 Mars Pathfinder Incident (1997):

The Mars Pathfinder spacecraft experienced repeated system resets several days into the mission. Root cause: unbounded priority inversion in VxWorks RTOS. A low-priority meteorological data task held a shared mutex. A high-priority communications task needed that mutex. Multiple medium-priority tasks kept preempting the low-priority task. The high-priority communications task starved long enough to trigger a hardware watchdog timer, which issued a system reset. The fix: enable priority inheritance on the mutex (a one-line VxWorks config change). NASA uploaded the patch to Mars. Resets stopped. A $150M mission nearly failed due to a subtle scheduling bug.

MLFQ — Multilevel Feedback Queue

MLFQ
Multiple queues with different priorities and time quantums. Processes move between queues based on their CPU usage behaviour. CPU-bound jobs sink; I/O-bound interactive jobs stay at top. The go-to algorithm for general-purpose OS schedulers.
🎓 University

How MLFQ Works — The 6 Rules (from OSTEP)

  1. Rule 1: If Priority(A) > Priority(B) → A runs, B doesn't.
  2. Rule 2: If Priority(A) == Priority(B) → A and B run Round Robin.
  3. Rule 3: When a process enters the system → placed at highest priority (Q0).
  4. Rule 4 (Demotion): If a process uses its entire quantum at priority level → demoted one level. Penalises CPU-bound jobs.
  5. Rule 5 (Stay): If a process yields CPU for I/O before quantum expires → stays at same priority. Interactive/I/O jobs stay at top.
  6. Rule 6 (Priority Boost): Periodically move all processes to highest priority — prevents starvation of long jobs.
MLFQ — 3-Queue Example Walkthrough
Queue Q0 (RR, q=8ms)   ← Highest priority — new/interactive jobs
Queue Q1 (RR, q=16ms)  ← Medium — moderate CPU users
Queue Q2 (FCFS)        ← Lowest — long CPU-bound jobs

Scenario: Interactive job (I/O every 5ms) vs. CPU-bound job (no I/O):
  CPU-bound: enters Q0 (q=8ms), uses full quantum → demoted to Q1
             uses Q1 quantum (16ms) fully → demoted to Q2 → runs on FCFS
  Interactive: enters Q0, yields after 5ms (I/O) → stays in Q0!
               Gets quick responses, never demoted.

Priority Boost (every S ms): Both return to Q0 — long job gets CPU
  This prevents CPU-bound job from starving forever in Q2

🔬 Advanced
🔬 MLFQ Anti-Gaming: A process could game MLFQ by intentionally yielding just before its quantum expires, artificially staying in a high-priority queue. Modern MLFQs (like those in OSTEP's description) measure total CPU time at a priority level, not just per-quantum. Once total allotment at a level is used, the process is demoted — regardless of how many voluntary yields it made. This closes the gaming loophole.

Linux CFS — How vruntime Works:
Each process has a virtual runtime (vruntime) = actual CPU time weighted by priority.
CFS always runs the process with the smallest vruntime (leftmost node in Red-Black Tree).
Lower-priority (higher nice) processes accumulate vruntime faster → they wait longer between runs.
Formula: vruntime += delta_exec × (NICE_0_LOAD / weight)
This gives fair CPU distribution — hence "Completely Fair Scheduler."

🔬 SCHED_DEADLINE (EDF): Linux 3.14+ added Earliest Deadline First scheduling. Tasks specify runtime, deadline, and period. The scheduler picks the task with the earliest absolute deadline. 🔬 Work-Stealing Scheduler: Modern parallel runtimes (Go, Java ForkJoin, Cilk) use work-stealing. Each worker thread has its own deque of tasks. When idle, it steals from the tail of another thread's deque. Hot path (local task execution) is lock-free. Achieves excellent load balance with minimal synchronisation overhead.

2.15 Thread Synchronisation Patterns

⚠️ Why Synchronisation Is Needed: Threads share the heap. Without synchronisation, concurrent writes create race conditions — the outcome depends on which thread executes first (non-deterministic). The OS cannot prevent this in user space — the programmer must use mutual exclusion primitives.
C — Mutex (Mutual Exclusion)
#include <pthread.h>

pthread_mutex_t lock = PTHREAD_MUTEX_INITIALIZER;
int shared_counter = 0;

void* increment(void* arg) {
    for (int i = 0; i < 100000; i++) {
        pthread_mutex_lock(&lock);    // Acquire — blocks if held
        shared_counter++;             // Critical section
        pthread_mutex_unlock(&lock);  // Release
    }
    return NULL;
}
// Without mutex: result is unpredictable (race condition)
// With mutex: always exactly 200000
C — Semaphore (counting / binary)
#include <semaphore.h>

sem_t sem;
sem_init(&sem, 0, 1);    // Binary semaphore (initial=1)

void* thread_func(void* arg) {
    sem_wait(&sem);       // P() — decrement; block if 0
    // Critical section
    sem_post(&sem);       // V() — increment; wake waiting thread
    return NULL;
}
// Counting semaphore: sem_init(&sem, 0, N) → allows N concurrent accesses
// Use: limit DB connections to max N threads simultaneously

2.17 Process & Thread Deep Dive

Process Priority Inversion — Complete Example

Priority inversion is a subtle scheduling bug where a high-priority task is effectively blocked by a lower-priority task. This was famously discovered in the Mars Pathfinder mission (1997). The rover experienced unexpected system resets because a low-priority meteorological data collection task held a mutex that a high-priority bus management task needed. A medium-priority communications task kept preempting the low-priority task, keeping the mutex locked and starving the high-priority task — violating real-time deadlines.

Priority Inversion Scenario:
1. L (low priority) acquires mutex M.
2. H (high priority) tries to acquire M — blocks (correct behaviour).
3. K (medium priority) preempts L (since K > L in priority).
4. L is stuck behind K; M stays locked; H waits for M → H waits for K!
5. H (highest) effectively runs at K's (medium) priority — INVERSION!
Fix: Priority inheritance — OS temporarily raises L's priority to H's level while L holds M.
// Demonstrating priority inheritance (POSIX mutex with PI)
#include <pthread.h>

pthread_mutexattr_t attr;
pthread_mutex_t pi_mutex;

int setup_pi_mutex() {
    pthread_mutexattr_init(&attr);
    // Enable Priority Inheritance protocol
    pthread_mutexattr_setprotocol(&attr, PTHREAD_PRIO_INHERIT);
    pthread_mutex_init(&pi_mutex, &attr);
    return 0;
}

// Now when low-priority thread L holds pi_mutex and

// → OS automatically boosts L's priority to H's level

// → H unblocks and runs at full priority

CPU Affinity & NUMA

CPU Affinity is the property of binding a thread or process to run on specific CPU cores. On multi-core systems, migrating a process between cores causes cache pollution (the old core's warm cache is wasted). Setting CPU affinity prevents migration, improving cache performance for latency-sensitive applications like databases and real-time audio.

// Set CPU affinity — bind process to cores 0 and 1 only
#define _GNU_SOURCE
#include <sched.h>
#include <stdio.h>

int main() {
    cpu_set_t mask;
    CPU_ZERO(&mask);      // Clear all CPUs
    CPU_SET(0, &mask);    // Allow CPU 0
    CPU_SET(1, &mask);    // Allow CPU 1
    
    if (sched_setaffinity(0, sizeof(mask), &mask) != 0) {
        perror("sched_setaffinity failed");
        return 1;
    }
    printf("Process now runs only on CPU 0 or 1\n");
    
    // Verify
    cpu_set_t get_mask;
    sched_getaffinity(0, sizeof(get_mask), &get_mask);
    for (int i = 0; i < 4; i++)
        printf("CPU %d: %s\n", i, CPU_ISSET(i, &get_mask) ? "YES" : "NO");
    return 0;
}

2.18 Key Facts, Formulas & Code Bank

Context Switch Frequency: With n processes each running for average burst t, context switches per second ≈ n/(t + c) where c = context switch time. Minimise c to maximise efficiency.
Process States Mnemonic: Never Rush Recklessly — Wait Today → New, Ready, Running, Waiting, Terminated
CRITICAL: Only ONE process can be in Running state per CPU core at any time. Multiple processes can be in Ready and Waiting states simultaneously — those queues can be arbitrarily long.
CPU Efficiency with context switch overhead:
Efficiency = BT / (BT + switch_time) × 100%
If BT=10ms, switch=1ms: Efficiency = 10/11 ≈ 90.9%
If BT=1ms, switch=1ms: Efficiency = 1/2 = 50% — switch overhead dominates!
IPC Performance Ranking (fastest to slowest):
Shared Memory > Memory-Mapped File > Pipe > Message Queue > Socket (loopback) > Network Socket
Deadlock in IPC: Two processes each write to their pipe to the other, but both pipes are full (64KB buffer). Both block waiting for the other to read — deadlock! Solution: non-blocking I/O (O_NONBLOCK) or use select()/poll() to handle multiple FDs.
Thread Overhead vs Process:
Thread creation: ~10µs | Process creation: ~1000µs (100× slower)
Thread context switch: ~1µs (no TLB flush) | Process switch: ~5-10µs
Amdahl's Law — Multithreading Limit:
Speedup = 1 / (S + (1-S)/N) where S=serial fraction, N=threads
If 20% of code is serial: max speedup → 1/0.2 = 5× regardless of N threads
Scheduling Tie-Breaking Rule: When SJF/SRTF has ties (equal burst times), the problem statement should specify. Default assumption for GATE: break ties by arrival time (earlier arrival wins); if simultaneous arrival, use process ID order (P1 before P2).
RR Quantum Selection Rule:
Good quantum q: 80% of CPU bursts should complete within q.
Typical: q = 10ms–100ms. Linux CFS doesn't use fixed quantum — uses vruntime fairness.

// Simulate FCFS scheduling in C
#include <stdio.h>
struct Process { int pid, at, bt, ct, tat, wt; };

void fcfs(struct Process p[], int n) {
    // Sort by arrival time
    for (int i=0; i<n-1; i++)
        for (int j=i+1; j<n; j++)
            if (p[i].at > p[j].at) { struct Process t=p[i]; p[i]=p[j]; p[j]=t; }
    
    int time = 0;
    for (int i=0; i<n; i++) {
        if (time < p[i].at) time = p[i].at;  // CPU idle gap
        p[i].ct = time + p[i].bt;
        p[i].tat = p[i].ct - p[i].at;
        p[i].wt = p[i].tat - p[i].bt;
        time = p[i].ct;
        printf("P%d: CT=%d TAT=%d WT=%d\n", p[i].pid, p[i].ct, p[i].tat, p[i].wt);
    }
}
// Simulate Round Robin scheduling
void round_robin(struct Process p[], int n, int q) {
    int remaining[n], time=0, done=0;
    for (int i=0; i<n; i++) remaining[i] = p[i].bt;
    
    while (done < n) {
        int progressed = 0;
        for (int i=0; i<n; i++) {
            if (remaining[i] > 0 && p[i].at <= time) {
                progressed = 1;
                int run = (remaining[i] < q) ? remaining[i] : q;
                time += run;
                remaining[i] -= run;
                if (remaining[i] == 0) {
                    done++;
                    p[i].ct = time;
                    p[i].tat = p[i].ct - p[i].at;
                    p[i].wt = p[i].tat - p[i].bt;
                }
            }
        }
        if (!progressed) time++;  // CPU idle: advance time
    }
}

Process Priority Inversion — Complete Example

Priority inversion is a subtle scheduling bug where a high-priority task is effectively blocked by a lower-priority task. This was famously discovered in the Mars Pathfinder mission (1997). The rover experienced unexpected system resets because a low-priority meteorological data collection task held a mutex that a high-priority bus management task needed. A medium-priority communications task kept preempting the low-priority task, keeping the mutex locked and starving the high-priority task — violating real-time deadlines.

Priority Inversion Scenario:
1. L (low priority) acquires mutex M.
2. H (high priority) tries to acquire M — blocks (correct behaviour).
3. K (medium priority) preempts L (since K > L in priority).
4. L is stuck behind K; M stays locked; H waits for M → H waits for K!
5. H (highest) effectively runs at K's (medium) priority — INVERSION!
Fix: Priority inheritance — OS temporarily raises L's priority to H's level while L holds M.
// Demonstrating priority inheritance (POSIX mutex with PI)
#include <pthread.h>

pthread_mutexattr_t attr;
pthread_mutex_t pi_mutex;

int setup_pi_mutex() {
    pthread_mutexattr_init(&attr);
    // Enable Priority Inheritance protocol
    pthread_mutexattr_setprotocol(&attr, PTHREAD_PRIO_INHERIT);
    pthread_mutex_init(&pi_mutex, &attr);
    return 0;
}

// → OS automatically boosts L's priority to H's level
// → L finishes critical section quickly
// → H unblocks and runs at full priority

CPU Affinity & NUMA

CPU Affinity is the property of binding a thread or process to run on specific CPU cores. On multi-core systems, migrating a process between cores causes cache pollution (the old core's warm cache is wasted). Setting CPU affinity prevents migration, improving cache performance for latency-sensitive applications like databases and real-time audio.

// Set CPU affinity — bind process to cores 0 and 1 only
#define _GNU_SOURCE
#include <sched.h>
#include <stdio.h>

int main() {
    cpu_set_t mask;
    CPU_ZERO(&mask);      // Clear all CPUs
    CPU_SET(0, &mask);    // Allow CPU 0
    CPU_SET(1, &mask);    // Allow CPU 1
    
    if (sched_setaffinity(0, sizeof(mask), &mask) != 0) {
        perror("sched_setaffinity failed");
        return 1;
    }
    printf("Process now runs only on CPU 0 or 1\n");
    
    // Verify
    cpu_set_t get_mask;
    sched_getaffinity(0, sizeof(get_mask), &get_mask);
    for (int i = 0; i < 4; i++)
        printf("CPU %d: %s\n", i, CPU_ISSET(i, &get_mask) ? "YES" : "NO");
    return 0;
}

2.18 Key Facts, Formulas & Code Bank

Context Switch Frequency: With n processes each running for average burst t, context switches per second ≈ n/(t + c) where c = context switch time. Minimise c to maximise efficiency.
Process States Mnemonic: Never Rush Recklessly — Wait Today → New, Ready, Running, Waiting, Terminated
CRITICAL: Only ONE process can be in Running state per CPU core at any time. Multiple processes can be in Ready and Waiting states simultaneously — those queues can be arbitrarily long.
CPU Efficiency with context switch overhead:
Efficiency = BT / (BT + switch_time) × 100%
If BT=10ms, switch=1ms: Efficiency = 10/11 ≈ 90.9%
If BT=1ms, switch=1ms: Efficiency = 1/2 = 50% — switch overhead dominates!
IPC Performance Ranking (fastest to slowest):
Shared Memory > Memory-Mapped File > Pipe > Message Queue > Socket (loopback) > Network Socket
Deadlock in IPC: Two processes each write to their pipe to the other, but both pipes are full (64KB buffer). Both block waiting for the other to read — deadlock! Solution: non-blocking I/O (O_NONBLOCK) or use select()/poll() to handle multiple FDs.
Thread Overhead vs Process:
Thread creation: ~10µs | Process creation: ~1000µs (100× slower)
Thread context switch: ~1µs (no TLB flush) | Process switch: ~5-10µs
Amdahl's Law — Multithreading Limit:
Speedup = 1 / (S + (1-S)/N) where S=serial fraction, N=threads
If 20% of code is serial: max speedup → 1/0.2 = 5× regardless of N threads
Scheduling Tie-Breaking Rule: When SJF/SRTF has ties (equal burst times), the problem statement should specify. Default assumption for GATE: break ties by arrival time (earlier arrival wins); if simultaneous arrival, use process ID order (P1 before P2).
RR Quantum Selection Rule:
Good quantum q: 80% of CPU bursts should complete within q.
Typical: q = 10ms–100ms. Linux CFS doesn't use fixed quantum — uses vruntime fairness.

// Simulate FCFS scheduling in C
#include <stdio.h>
struct Process { int pid, at, bt, ct, tat, wt; };

void fcfs(struct Process p[], int n) {
    // Sort by arrival time
    for (int i=0; i<n-1; i++)
        for (int j=i+1; j<n; j++)
            if (p[i].at > p[j].at) { struct Process t=p[i]; p[i]=p[j]; p[j]=t; }
    
    int time = 0;
    for (int i=0; i<n; i++) {
        if (time < p[i].at) time = p[i].at;  // CPU idle gap
        p[i].ct = time + p[i].bt;
        p[i].tat = p[i].ct - p[i].at;
        p[i].wt = p[i].tat - p[i].bt;
        time = p[i].ct;
        printf("P%d: CT=%d TAT=%d WT=%d\n", p[i].pid, p[i].ct, p[i].tat, p[i].wt);
    }
}
// Simulate Round Robin scheduling
void round_robin(struct Process p[], int n, int q) {
    int remaining[n], time=0, done=0;
    for (int i=0; i<n; i++) remaining[i] = p[i].bt;
    
    while (done < n) {
        int progressed = 0;
        for (int i=0; i<n; i++) {
            if (remaining[i] > 0 && p[i].at <= time) {
                progressed = 1;
                int run = (remaining[i] < q) ? remaining[i] : q;
                time += run;
                remaining[i] -= run;
                if (remaining[i] == 0) {
                    done++;
                    p[i].ct = time;
                    p[i].tat = p[i].ct - p[i].at;
                    p[i].wt = p[i].tat - p[i].bt;
                }
            }
        }
        if (!progressed) time++;  // CPU idle: advance time
    }
}

🎓 University Exams

Short Answer Questions (2 Marks)

Q1: What is a Zombie Process?
A zombie process has completed execution but still has an entry in the process table because its parent hasn't called wait() to read its exit status. It consumes virtually no CPU or memory, but it does occupy a PID. If a parent never calls wait(), zombies accumulate and can exhaust the PID space.
Q2: Differentiate Preemptive vs Non-Preemptive scheduling.
Non-preemptive: Once a process gets the CPU, it runs until it terminates or voluntarily blocks (e.g., FCFS, pure SJF). No context switch due to timer. Preemptive: OS can interrupt a running process via a timer interrupt and assign CPU to another process (e.g., RR, SRTF). Requires timer hardware and adds context switch overhead.
Q3: What is the Convoy Effect and which algorithm causes it?
The convoy effect occurs in FCFS when short, I/O-bound processes get stuck waiting behind one long CPU-bound process. Multiple short processes "convoy" (follow like a slow vehicle on a highway) behind the long job, causing high average waiting times and poor CPU and I/O device utilisation.
Q4: What does a PCB contain?
PCB (Process Control Block) contains: Process State (New/Ready/Running/Waiting/Terminated), Program Counter (next instruction address), CPU Registers (all GPRs, stack pointer, flags), Memory Info (page tables, segments), Scheduling Info (priority, nice), I/O Status (open files, pending I/O), Accounting Info (CPU time, start time, owner).
Q5: What is the difference between Shared Memory and Message Passing IPC?
Shared Memory: Processes map the same physical pages into their address spaces. Fastest IPC (no kernel involvement after setup), but requires explicit synchronisation (semaphores/mutexes). Message Passing: Data passes through kernel buffers (send/receive syscalls). Slower (kernel copy each time) but synchronisation is built-in. Message passing is safer for distributed systems (works across machines).

Long Answer Questions (10 Marks)

Q6: Explain the 5-State Process Model with transitions and diagram.

The five-state process model describes the complete lifecycle of a process:

  1. New: Process is being created; PCB is allocated; not yet in ready queue.
  2. Ready: Process is in main memory, waiting for CPU. Multiple processes can be ready simultaneously (ready queue).
  3. Running: Instructions are executing on CPU. Only one process per CPU core can be in this state.
  4. Waiting (Blocked): Process is waiting for an event (I/O completion, semaphore, child exit). Not using CPU.
  5. Terminated: Process has finished; PCB still exists until parent calls wait().

Key Transitions: New→Ready (admitted), Ready→Running (dispatched), Running→Ready (preempted/timer), Running→Waiting (I/O request), Waiting→Ready (I/O complete), Running→Terminated (exit/signal).

(Draw state boxes with directed arrows for each transition in exam)

Q7: Compare User-Level Threads (ULT) vs Kernel-Level Threads (KLT). Which is used in Linux?

ULT (Many-to-One): Managed by user-space library. Fast context switch (no mode switch). Disadvantage: one blocking system call blocks the entire process — no true parallelism on multi-core.

KLT (One-to-One): Each thread maps to a kernel thread. True multi-core parallelism. One blocked thread doesn't block others. Slower context switch (kernel involvement required).

Linux: Uses One-to-One model via NPTL (Native POSIX Thread Library). Every pthread_create() creates a kernel-scheduled task (a clone with shared address space). getpid() returns the thread group ID (same for all threads); gettid() returns the individual thread ID.

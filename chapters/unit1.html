1.1 What is an Operating System?

An Operating System (OS) is system software that acts as an intermediary between computer hardware and the user (or application programs). It manages hardware resources and provides a convenient environment for executing programs.

💡
Core Definition (Silberschatz): An OS is a program that runs at all times on the computer (usually called the kernel), with everything else being either a system program or an application program.

Two Roles of an OS:

  • Resource Allocator: Decides how CPU time, memory, files, and I/O devices are shared among competing processes — fairly and efficiently.
  • Control Program: Prevents errors and improper use of hardware by controlling program execution through interrupts, traps, privileged instructions, and protection mechanisms.
📌
OSTEP Framing: An OS provides three fundamental abstractions:
1. Virtualization — makes each process think it has its own CPU and memory
2. Concurrency — manages multiple activities happening simultaneously
3. Persistence — provides reliable, long-term storage via the file system

Distinguish carefully: Kernel (always-resident core) ≠ OS (kernel + system programs) ≠ Application programs (user software).
🎓 University Syllabus

Two Perspectives of an OS

1. Resource Manager View: The OS manages all hardware resources — CPU, memory, I/O devices, files — and allocates them efficiently among multiple processes.

2. Extended Machine View: The OS presents users and programs with an abstract, convenient view of the hardware, hiding the complexity of raw hardware operations. This is also called a virtual machine abstraction.

Goals of an OS

  • Convenience: Make the computer easy to use
  • Efficiency: Use hardware resources efficiently
  • Ability to evolve: Allow new system functions to be developed without interfering with old ones

Dual Mode Operation

Modern OSes operate in two modes to protect system integrity:

  • User Mode (Mode Bit = 1): Restricted access. User applications run here. Cannot directly execute privileged instructions.
  • Kernel Mode (Mode Bit = 0): Full access to all hardware instructions and memory. OS kernel runs here.

When a user program needs an OS service, it issues a system call, which causes a trap (software interrupt) that switches the CPU from User Mode to Kernel Mode.

⚠️
DU Exam Tip: The mode bit transition (0 ↔ 1) and the trap mechanism are frequently tested. Know which instructions are privileged (only executable in kernel mode): HLT, I/O instructions, interrupts enable/disable, memory management instructions.
⚡ GATE Focus

GATE Key Points — OS Fundamentals

  • OS is not required to run programs — programs can run without OS (embedded systems)
  • The kernel is always resident in memory (not literally executing every instant). Everything else is loaded as needed.
  • Privileged instructions cause a trap if executed in user mode → hardware protection mechanism
  • A timer interrupt is used by the OS to regain control from a running process (prevents infinite loops)
Mode Bit Rule
Mode Bit = 0 → Kernel Mode  |  Mode Bit = 1 → User Mode

Worked GATE-Style Question

MCQ — Privileged Instructions
Q. Which of the following instructions CANNOT be executed in user mode?
   (A) ADD R1, R2, R3
   (B) LOAD R1, [address]
   (C) HLT (halt the CPU)
   (D) CMP R1, R2

Answer: (C) HLT

Reasoning:
  ADD, LOAD, CMP — arithmetic and memory read in user's own
  address space: perfectly legal in user mode.

  HLT halts the entire CPU — a privileged instruction.
  If executed in user mode, hardware generates a General
  Protection Fault (trap) → OS catches it → kills the process.

Other privileged instructions to know for GATE:
  • I/O port instructions (IN/OUT on x86)
  • Interrupt enable/disable (STI/CLI)
  • Modifying page table base register (CR3 on x86)
  • Loading IDT/GDT (LIDT/LGDT)
  • Setting the mode bit itself

1.1c Interrupts & Hardware Protection

🎓 University

Interrupt Mechanism

Interrupt
An interrupt is a signal to the CPU that an event has occurred requiring immediate attention. The CPU suspends the current instruction sequence, saves its state, and jumps to a special routine called the Interrupt Service Routine (ISR) or interrupt handler.

Types of Interrupts

TypeSourceTriggerExamples
Hardware InterruptExternal deviceDevice sends signal via interrupt lineKeyboard press, disk I/O complete, timer tick, network packet arrival
Software Interrupt (Trap)Executing programExplicit instruction (INT, SYSCALL)System calls, breakpoints
ExceptionCPU errorFault during instruction executionDivision by zero, page fault, segmentation fault, overflow

Interrupt Handling Flow

Interrupt Processing Steps
1. Device/software raises interrupt signal
2. CPU completes current instruction
3. CPU checks for pending interrupts
4. CPU saves current state (PC, registers) → stack or PCB
5. CPU looks up interrupt number in IVT/IDT
6. CPU jumps to the ISR (Interrupt Service Routine)
7. ISR handles the event (e.g., reads data from device buffer)
8. ISR sends EOI (End of Interrupt) to interrupt controller
9. CPU restores saved state
10. CPU resumes the interrupted program

Interrupt Vector Table (IVT) / Interrupt Descriptor Table (IDT)

A table stored in memory that maps each interrupt number to the address of its ISR. On x86:

  • IVT (Real Mode): 256 entries × 4 bytes = 1KB at address 0x0000
  • IDT (Protected Mode): 256 entries × 8 bytes. Pointed to by IDTR register.
  • Entry 0 = divide-by-zero, Entry 14 = page fault, Entry 0x80 = Linux system call (legacy)

DMA — Direct Memory Access

DMA (Direct Memory Access)
A hardware mechanism that allows I/O devices to transfer data directly to/from RAM without CPU involvement. The CPU sets up the DMA controller (source, destination, byte count), then is free to do other work. The DMA controller interrupts the CPU only when the entire transfer is complete.
I/O MethodCPU InvolvementSpeedUse Case
Programmed I/OCPU polls device status in a loop (busy waiting)Slow — CPU wastedSimple embedded systems
Interrupt-Driven I/OCPU starts I/O, does other work, device interrupts when doneBetter — CPU freed between opsKeyboard, mouse
DMACPU sets up transfer, DMA controller handles it, one interrupt at endFast — bulk transfersDisk I/O, network, GPU

Hardware Protection Mechanisms

The OS relies on hardware support to enforce protection. Three types:

Protection TypeMechanismWhat It Prevents
CPU ProtectionTimer — generates interrupt after a set interval. OS resets timer on each context switch.Prevents a process from monopolizing the CPU (infinite loops). OS regains control via timer interrupt.
Memory ProtectionBase + Limit registers — checked on every memory access by MMU. Each process has its own pair.Prevents a process from accessing memory of other processes or the OS kernel.
I/O ProtectionAll I/O instructions are privileged — can only execute in kernel mode. User programs must use system calls for I/O.Prevents user programs from directly accessing hardware devices (disk corruption, security breach).
⚠️
DU Exam Tip: The three types of hardware protection (CPU, Memory, I/O) and dual-mode operation are frequently asked together. Know: Timer interrupt → CPU protection; Base/Limit → Memory protection; Privileged I/O → I/O protection; Mode bit → enforces all three.
⚡ GATE
  • Interrupt priority: Higher-priority interrupts can preempt lower-priority ISRs (nested interrupts).
  • DMA uses cycle stealing — borrows memory bus cycles from CPU. CPU is slightly slowed during DMA transfers.
  • Timer value is loaded by OS (privileged). If a user could modify the timer, they could disable preemption → security violation.
  • Trap (software interrupt) is synchronous — caused by the currently executing instruction. Hardware interrupt is asynchronous — can occur at any time.
  • GATE: "Which protection mechanism prevents a process from running forever?" → Timer
🔬 Advanced

Linux Interrupt Handling: Top Half vs Bottom Half

In Linux, interrupt handling is deliberately split into two halves to keep the system responsive. The top half (the ISR itself) runs with interrupts disabled and must complete as fast as possible — it does only the minimum necessary (acknowledges the hardware, reads data from device registers, enqueues work) and defers the rest. The bottom half handles the deferred, less time-critical work with interrupts re-enabled. This split is essential because a device like a network card can deliver thousands of packets per second, and blocking all other interrupts for the full processing time would be catastrophic.

Bottom-Half Mechanisms

MechanismContextCan Sleep?Use Case
SoftirqInterrupt context (softirq daemon)NoNetwork packet processing (NET_RX_SOFTIRQ), timer processing. Statically defined; can run on multiple CPUs simultaneously.
TaskletInterrupt context (built on softirq)NoGeneral device driver deferred work. Easier to use than softirqs; a given tasklet runs on only one CPU at a time.
WorkqueueProcess context (kernel thread)YesWork that needs to sleep (e.g., allocate memory with GFP_KERNEL, perform I/O). Runs in a kernel worker thread (kworker).

NAPI — New API (Network Interrupt Mitigation)

Under high network load, interrupt-driven packet processing causes an interrupt storm — the network ISR fires so frequently that the CPU spends all its time in interrupt handlers and cannot run any processes. Linux's NAPI solves this with a hybrid approach: after the first packet arrives (interrupt), the driver disables further interrupts and switches to polling mode, processing packets in batches. Only when the queue empties does it re-enable interrupts. This provides low latency at low load (interrupt-driven) and high throughput at high load (polling).

IRQ Affinity (SMP Systems)

On multi-core systems, each hardware IRQ line can be configured to be serviced by a specific CPU or set of CPUs. This is called IRQ affinity and is controlled via /proc/irq/N/smp_affinity (a bitmask). Distributing IRQs across CPUs prevents one CPU from becoming a bottleneck for interrupt processing. Tools like irqbalance do this automatically. High-performance networking often pins network card IRQs to the same NUMA node as the CPU running the application (reducing cross-node memory traffic).

1.2 History & Evolution of OS

Operating Systems have evolved through several generations, each driven by new hardware capabilities and user demands. Understanding this evolution helps grasp why modern OS features exist.

🎓 University

Four Generations of OS

1st Generation · 1940s–1955
Vacuum Tubes — No OS
Programs were wired directly into the machine. No OS existed. Programmers operated computers themselves. Output was on punch cards. Key: One job at a time, no sharing.
2nd Generation · 1955–1965
Transistors — Batch Processing
Jobs collected on cards/tape and submitted in batches. A Resident Monitor (primitive OS) automatically loaded the next job. Problem: CPU idle during I/O operations.
3rd Generation · 1965–1980
ICs — Multiprogramming & Time-Sharing
Multiprogramming: Multiple jobs kept in memory; CPU switches to another job when one is blocked on I/O. Time-sharing (CTSS, MULTICS): CPU time sliced among users for interactive use. UNIX born here (1969).
4th Generation · 1980–Present
VLSI — Personal Computers & Beyond
MS-DOS → Windows, Macintosh (GUI). Linux (1991). GUI, networking, distributed systems, mobile OS (Android, iOS). Modern OSes: cloud-native, containerized, real-time.

OS Evolution as a Research Arc

Every generation of OS design was a direct response to the dominant hardware constraint of its era. Batch systems existed because human operator time was the bottleneck; multiprogramming arose because CPU cycles were expensive relative to I/O speeds; time-sharing emerged when interactive terminals made latency the user-visible metric. Understanding this pattern is critical for research-level OS thinking: OS design is always a function of the hardware cost ratio at the time.

The UNIX Philosophy as a Design Principle

UNIX, designed at Bell Labs by Thompson and Ritchie (1969–1973), codified a design philosophy still governing modern systems: write programs that do one thing well, make programs work together via a uniform I/O abstraction (the file), and prefer plain text as a universal interface. The decision to rewrite UNIX in C in 1973 was radical — it made the OS portable across hardware for the first time, seeding the entire subsequent ecosystem. This is why Linux, macOS, Android, and iOS all share UNIX DNA fifty years later.

Missing a deadline is a catastrophic failure. Used in flight control, nuclear reactors, pacemakers. Must guarantee worst-case execution time (WCET).

Real-Time Scheduling: RMS vs EDF

To guarantee that RTOS tasks meet their deadlines, specific scheduling algorithms are used. The two most fundamental real-time scheduling algorithms are Rate Monotonic Scheduling (RMS) and Earliest Deadline First (EDF).

1. Rate Monotonic Scheduling (RMS)

A static priority algorithm where priorities are assigned based on the period of the task.

  • Rule: Shorter period = Higher priority. (The more frequently a task needs to run, the higher its priority).
  • Type: Static priority, preemptive.
  • Schedulability Test (Liu & Layland Bound): A set of n periodic tasks can definitely be scheduled by RMS if their total CPU utilization U satisfies:
    U = Σ (Ci / Ti) ≤ n(21/n - 1)
    (For n=2, limit is ~0.828. As n→∞, limit is ln(2) ≈ 0.693). If utilization is ≤ 69.3%, RMS is guaranteed to find a schedule.

2. Earliest Deadline First (EDF)

A dynamic priority algorithm where priorities are assigned based on the absolute deadline.

  • Rule: The task with the earliest deadline at any given moment gets the highest priority.
  • Type: Dynamic priority, preemptive.
  • Schedulability Test: EDF is an optimal scheduling algorithm on a single processor. It can schedule any task set as long as total utilization is ≤ 1 (100%).
    U = Σ (Ci / Ti) ≤ 1

Comparison: EDF handles higher CPU utilization (up to 100%) but is more complex to implement because priorities change dynamically. RMS is simpler and has very low overhead, but can only guarantee schedules up to ~69% utilization.

Beyond the Textbook Taxonomy

The traditional OS type classification (batch, time-sharing, RTOS, distributed) reflects hardware constraints of the 1970s–1990s. Modern OS research has introduced fundamentally new categories driven by cloud economics, security requirements, and heterogeneous hardware.

Unikernels — Library Operating Systems

A unikernel is a single-purpose, single-address-space OS image compiled from a library OS (LibOS) and an application. There is no user/kernel separation — the entire stack runs in ring 0. This eliminates syscall overhead, mode switching, and attack surface. Unikernels boot in milliseconds and are typically kilobytes to a few megabytes in size, making them ideal for cloud function deployments. Examples: MirageOS (OCaml, from Cambridge), IncludeOS (C++), Nanos, OSv. The tradeoff is limited functionality and difficult debugging (no shell, no dynamic loading).

Real-Time OS — Deeper Scheduling Theory

The Liu & Layland (1973) paper that introduced Rate Monotonic Scheduling established the foundational result that fixed-priority preemptive scheduling is optimal in the class of fixed-priority algorithms when task periods equal deadlines. Later work by Lehoczky, Sha, and Ding extended this to show that even when the utilization bound exceeds 69.3%, RMS may still succeed — the bound is sufficient but not necessary. The exact schedulability analysis requires computing the response time of each task, not just total utilization. This is tested in advanced systems courses and OS research contexts.

📚
Key Papers:
• Liu & Layland — "Scheduling Algorithms for Multiprogramming in a Hard-Real-Time Environment" (JACM 1973) — introduced RMS and EDF
• Madhavapeddy et al. — "Unikernels: Library Operating Systems for the Cloud" (ASPLOS 2013) — MirageOS
• Bruns et al. — "Mixed-Criticality Scheduling" — real-time OS with tasks of different safety levels (automotive, aerospace)

1.4 OS Structure

🎓 University

An OS can be organized in multiple ways. The structure determines how the OS components interact and what privileges each component has.

1. Simple Structure (MS-DOS Style)

No clear separation between layers. Application programs can directly call device drivers. Designed for a single-user, single-process environment. No protection between levels.

⚠️
In MS-DOS, a buggy application can corrupt the OS. There is no memory protection or dual-mode operation. This is why it was vulnerable to malware.

2. Layered Structure

OS divided into N layers (Layer 0 = hardware, Layer N = user interface). Each layer only uses services of the layer directly below it.

LayerFunction
Layer 5 (top)User programs
Layer 4User I/O devices
Layer 3Operator-process communication
Layer 2Memory management
Layer 1CPU scheduling
Layer 0 (bottom)Hardware

Advantages: Modularity, easy debugging (one layer at a time). Disadvantages: Requests may pass through several layers, adding overhead; difficult to define layers cleanly.

3. Modular Structure (Modern Linux/Solaris)

Core kernel with dynamically loadable modules. Each module has defined interfaces and communicates with the core and other modules. This is widely used in modern general-purpose OSes because it balances performance and extensibility.

Linux uses a modular monolithic kernel: It's technically monolithic (everything in kernel space) but supports loadable kernel modules (LKMs) for drivers, filesystems, etc.

UNIX System Structure

UNIX uses a two-level structure:

  • Kernel: Everything below the system-call interface and above the hardware. Contains file system, CPU scheduling, memory management, device drivers. This is a large monolithic piece.
  • System programs: Shells, compilers, editors — run in user space.

Policy vs Mechanism Separation

A critical OS design principle: separate policy (what to do) from mechanism (how to do it). This idea originates from Dijkstra's THE OS (1968) and was formally articulated in the context of microkernels by Liedtke (1995).

  • Mechanism: Timer interrupt implementation, context switch code, virtual memory page table walk
  • Policy: How long a time slice should be, which process gets CPU next, which pages to evict

This separation allows changing policies (e.g., swapping from round-robin to CFS scheduling) without modifying the underlying hardware mechanisms. Microkernels push this to the extreme — minimum mechanism in kernel, maximum policy pushed to user-space servers. The practical tension: pushing policy too far into user space increases IPC overhead, which is why pure microkernels (Mach) were often slower than expected and why hybrid kernels (Windows NT, macOS XNU) became dominant.

The L4 Microkernel Family — Proof That Speed Is Possible

Early microkernels (Mach) were dismissed as too slow because every IPC call required multiple kernel crossings and memory copies. Jochen Liedtke's 1993 paper demonstrated that a carefully designed microkernel could perform IPC in under 5 µs — an order of magnitude faster than Mach. His key insight: Mach's IPC overhead was an implementation problem, not an architectural problem. L4 achieved this by: keeping the entire IPC path in the CPU's L1 cache, eliminating unnecessary copies, and having an extremely small kernel (fewer than 12,000 lines of code). This spawned the L4 microkernel family: L4/Fiasco, seL4, OKL4, Nova — which power modern security-critical systems (Qualcomm's QSEE, iOS's secure enclave, satellite systems).

seL4 — Formal Verification

seL4 (NICTA/UNSW, 2009) is the first OS kernel to have its complete implementation formally verified — that is, mathematically proved to satisfy its specification, and the specification proved to enforce confidentiality and integrity. The verification covers 8,700 lines of C code and took roughly 20 person-years. seL4 is now used in critical avionics, autonomous vehicles, and military systems where correctness guarantees are mandatory. The key insight from the seL4 project: formal verification of OS kernels is feasible at the 10K LOC scale, but the kernel must be deliberately small — which reinforces the microkernel design argument.

1.5 Kernel Types

The kernel is the core of the OS — the one program running at all times. How it is structured determines performance, security, and maintainability.

🎓 University

1. Monolithic Kernel

Monolithic Kernel
The entire OS runs as a single large process in kernel space. All OS services (scheduling, memory, file system, device drivers, IPC) run in the same address space.

Examples: Linux, traditional UNIX, MS-DOS

2. Microkernel

Microkernel
Only the most essential services run in kernel space: basic IPC, minimal memory management, basic scheduling. All other services (file system, device drivers, network stack) run as user-space processes.

Examples: Mach, Minix, QNX, L4

MINIX → Linux: Linus Torvalds started Linux (1991) as a monolithic kernel partly in response to Minix's microkernel design which he found too slow for personal use.

3. Layered Kernel

OS divided into hierarchical layers. Layer i uses services of layer i−1 only. Each layer is independently testable. Example: THE OS (Dijkstra, 1968), OpenVMS.

Problem: Strict layering is often impractical because some components (like the file system) need both memory management and device I/O simultaneously.

Core kernel plus dynamically loadable kernel modules (LKMs). Modules communicate via well-defined interfaces. Linux implements this — it is monolithic but supports runtime module loading.

A monolithic kernel that borrows some microkernel ideas (some services moved to user space). Examples: Windows NT kernel (combines monolithic speed with microkernel modularity), macOS XNU (Mach microkernel + BSD subsystem).

Common Wrong Statements — Exam Trap Alert!

⚠️
• "System call = function call" → WRONG. System call involves mode switch + trap; function call does not.
• "The kernel is always executing" → WRONG. The kernel is always resident in memory, but user processes execute most of the time.
• "Containers are lightweight VMs" → MISLEADING. Containers share the host kernel; VMs each have their own kernel. Containers = OS-level virtualization.
• "GPT is part of MBR" → WRONG. GPT replaces MBR; it's used by UEFI, not BIOS.
• "RMS guarantees schedulability if ∑(C/T) ≤ 1" → WRONG. That's EDF. RMS bound is n(2^(1/n)−1) ≈ 0.693.
• "Modular kernel = high reliability" → MISLEADING. Modules run in kernel space; a bad module can crash the OS. Modular improves extensibility, not fault isolation.

1.6 System Calls & copy_from_user

A system call is the programmatic way in which a user program requests a service from the OS kernel. It is the interface between the user space and kernel space.

🎓 University

How a System Call Works (Step by Step)

  1. User program calls a library function (e.g., read() in libc/glibc)
  2. Library puts the system call number in a register:
      • x86-32: EAX, args in EBX, ECX, EDX, ESI, EDI, trap via INT 0x80
      • x86-64: RAX, args in RDI, RSI, RDX, R10, R8, R9, trap via SYSCALL instruction
  3. CPU switches from User Mode → Kernel Mode (mode bit 1→0)
  4. OS jumps to system call entry point (x86-32: via IDT entry 0x80; x86-64: via LSTAR MSR — not IDT)
  5. Kernel dispatches to correct handler via syscall table indexed by syscall number
  6. Handler validates parameters, performs the service
  7. Result placed in RAX; CPU switches back to User Mode; on error, errno is set
  8. Library returns result to user program
💡
System Call Number: Every system call has a unique number. On Linux x86-64: read=0, write=1, open=2, close=3, fork=57, execve=59, exit=60.

copy_from_user & copy_to_user (Linux Kernel)

copy_from_user
A kernel function used to safely copy data from a user-space buffer to kernel space. It validates that the user-space pointer is valid and accessible before copying.

unsigned long copy_from_user(void *to, const void __user *from, unsigned long n);

Returns: 0 on success, number of bytes NOT copied on failure.
copy_to_user
Copies data from kernel space to user space. Essential for system call return data (e.g., read() returning file contents).

unsigned long copy_to_user(void __user *to, const void *from, unsigned long n);
🔐
Why not just use memcpy? A direct kernel memcpy() from a user pointer is unsafe — the user could pass a kernel address (privilege escalation attack) or an unmapped address (kernel crash). copy_from_user prevents both.

⚡ GATE

GATE System Call Quick Facts

  • System calls use trap (software interrupt) to switch to kernel mode
  • Parameters passed via registers, stack, or a block pointed to by a register
  • fork() returns 0 to child, child PID to parent
  • exec() replaces current process image — never returns (unless error)
  • wait() blocks parent until child terminates — prevents zombie processes
  • System call interface is standardized by POSIX — not OS-specific. Same code compiles on Linux, macOS, BSD.
System Call Overhead
Each syscall = trap + mode switch + handler + mode switch back ≈ 100–1000 ns on modern hardware

Worked GATE-Style Question — System Call Cost

Numerical — System Call vs Function Call Cost
Q. A user program calls write() 1,000 times to write individual
   characters to a file. Each system call takes 1 µs (including
   mode switch overhead). Each character is 1 byte.
   If instead the program buffers 1000 characters and calls
   write() once with a 1000-byte buffer, what is the time saved?

Solution:
  Without buffering: 1000 system calls × 1 µs = 1000 µs = 1 ms
  With buffering:    1 system call × 1 µs     = 1 µs

  Time saved = 1000 µs − 1 µs = 999 µs ≈ 1 ms

This is exactly why printf() uses internal buffering (stdio
buffer) and only calls write() when: buffer is full, '\n' is
printed (line-buffered mode), or fflush()/exit() is called.
The C library absorbs thousands of tiny writes into few syscalls.

Key insight: System calls are expensive relative to function
calls. Minimizing syscall frequency is a core OS/performance
principle. This also justifies read-ahead in file I/O.
💡
GATE Pattern — Identify the system call: Given a C library function, know which syscall it ultimately uses:
• printf() → write(fd=1, ...)
• scanf() → read(fd=0, ...)
• malloc() → brk() or mmap() (only when heap needs to grow)
• fopen() → open()
• exit() → _exit() (after calling atexit() handlers and flushing stdio buffers)
🔬 Advanced

vDSO — Virtual Dynamic Shared Object

Linux optimizes frequent, read-only syscalls (like gettimeofday(), clock_gettime()) using vDSO: a small shared library mapped into every process's address space by the kernel. The process can call it directly without a trap — effectively a zero-cost system call.

vDSO Mechanism
Without vDSO:   User → int 0x80/syscall → Kernel → read clock → return
With vDSO:      User → call vDSO page (mapped read-only) → return

Kernel updates the vDSO data page on each tick.
User reads it directly — no privilege transition needed.
Mapped at a randomized address (ASLR-compatible).

seccomp — System Call Filtering

Linux's seccomp (secure computing mode) allows a process to restrict which system calls it can make. A process installs a BPF filter that the kernel checks on every syscall.

seccomp-BPF Filter Actions
seccomp modes:
  SECCOMP_MODE_STRICT: only read(), write(), _exit(), sigreturn()
  SECCOMP_MODE_FILTER: install BPF program to filter syscalls

BPF filter actions per syscall:
  SECCOMP_RET_ALLOW    — permit the syscall
  SECCOMP_RET_KILL     — kill the process immediately (SIGSYS)
  SECCOMP_RET_ERRNO    — return an error code without executing
  SECCOMP_RET_TRACE    — notify a ptrace tracer
  SECCOMP_RET_LOG      — allow but log the syscall
  SECCOMP_RET_TRAP     — send SIGSYS to the process

Used by: Docker, Chrome renderer sandbox, Android apps, Firejail

static ssize_t my_dev_write(struct file *f, const char __user *buf,
                             size_t count, loff_t *off) {
    char kernel_buf[256];
    size_t to_copy = min_t(size_t, count, sizeof(kernel_buf) - 1);
    unsigned long not_copied;

    /* Safely copy from user space → kernel space */
    not_copied = copy_from_user(kernel_buf, buf, to_copy);
    if (not_copied) {
        return -EFAULT;  /* Bad user pointer; returns -14 */
    }
    kernel_buf[to_copy] = '\0';  /* Use to_copy, NOT count (avoids OOB!) */
    printk(KERN_INFO "Received: %s\n", kernel_buf);
    return to_copy;  /* Return bytes actually copied */
}

1.7 Linux Boot Process

When you press the power button, a precise sequence of events brings the OS from cold silicon to a running system. Understanding this is essential for both the DU exam and practical Linux knowledge.

🎓 University

The 6 Stages of Linux Boot

Stage 1
BIOS / UEFI — Power-On Self-Test (POST)
BIOS (Basic Input/Output System) or UEFI (Unified Extensible Firmware Interface) is firmware stored in ROM on the motherboard. It:
1. Performs POST — checks hardware (RAM, CPU, keyboard, storage)
2. Searches for a bootable device (in configured boot order)
3. Loads the first sector (512 bytes) of the bootable disk into memory at address 0x7C00

BIOS vs UEFI: BIOS uses MBR (max 2TB disks, 4 partitions). UEFI uses GPT (up to 9.4ZB disks, 128 partitions), supports Secure Boot.
Stage 2
MBR / GPT — Master Boot Record
Legacy BIOS Boot: BIOS loads the first 512 bytes of disk (MBR) to address 0x7C00.
The MBR is 512 bytes at sector 0 of the disk:
• Bytes 0–445: Bootstrap code (446 bytes) — loads stage-2 bootloader
• Bytes 446–509: Partition table (4 × 16-byte entries = 64 bytes)
• Bytes 510–511: Boot signature 0x55AA

UEFI Boot: UEFI firmware does NOT load MBR. Instead, it reads the GPT (GUID Partition Table), finds the EFI System Partition (FAT32), and loads an EFI bootloader (e.g., grubx64.efi or shimx64.efi) directly.

The MBR code (BIOS path) locates the active partition and loads its Volume Boot Record (VBR), which then loads the full bootloader.
Stage 3
GRUB — Grand Unified Bootloader
GRUB2 is the standard Linux bootloader. It:
1. Presents the boot menu (multiple OS choices, kernel versions)
2. Loads the selected Linux kernel (vmlinuz) into memory
3. Loads the initial RAM disk (initrd / initramfs) into memory
4. Passes kernel parameters (from /boot/grub/grub.cfg)

GRUB config: /boot/grub/grub.cfg | Rebuild: update-grub
Stage 4
Kernel Initialization
The decompressed kernel (vmlinuz) takes control:
1. Sets up CPU architecture (GDT, IDT, paging)
2. Initializes scheduler, memory allocator, VFS, device model, interrupt subsystem
3. Initializes hardware: interrupts, timers, device drivers
4. Mounts initramfs as a temporary root filesystem
5. Creates PID 0 (swapper/idle task — kernel internal) and PID 1 (/sbin/init — first user-space process)

PID 0 vs PID 1: PID 0 is the kernel idle task (runs when nothing else needs CPU). PID 1 is the first user-space process and becomes the ancestor of all processes. It also reaps orphaned processes (when a parent exits before its child).

Key files: /boot/vmlinuz-<version> (compressed kernel), /boot/initrd.img-<version>
Stage 5
initramfs — Initial RAM Filesystem
A temporary compressed cpio archive loaded into memory that the kernel mounts first. Purpose:
• Contains drivers needed to mount the real root filesystem (LVM, LUKS, RAID, NFS, NVMe)
• Runs /init script inside initramfs
• Once real root is mounted, switch_root pivots to it and discards initramfs

initrd (older): a compressed ext2 image. initramfs (modern): a cpio archive — faster, simpler, no filesystem driver needed.
Stage 6
systemd / init — User Space Initialization
systemd (PID 1) is the modern init system on most Linux distros (Ubuntu, Fedora, Arch):
1. Reads unit files from /etc/systemd/system/ and /lib/systemd/system/
2. Builds a dependency graph of units and starts services in parallel
3. Reaches the target (e.g., graphical.target or multi-user.target)
4. Starts login manager (getty / GDM)

SysV init (legacy): uses scripts in /etc/init.d/ and runlevel directories (/etc/rc3.d/).
Runlevels: 0=halt, 1=single-user, 3=multi-user, 5=graphical, 6=reboot.
systemd equivalents: rescue.target, multi-user.target, graphical.target.

Useful kernel parameters: single, systemd.unit=rescue.target, quiet, init=/bin/bash

1.8 Linux Kernel Modules (LKMs)

A Loadable Kernel Module (LKM) is a piece of code that can be loaded into the kernel at runtime — without rebooting or recompiling the kernel. LKMs extend kernel functionality dynamically.

🎓 University

Why LKMs?

  • Linux is a monolithic kernel but supports dynamic extension via LKMs
  • LKMs run in kernel space — full hardware access, no protection boundaries
  • Common uses: device drivers (GPU, USB, network), filesystem drivers (ext4, NTFS), network protocols
  • A buggy LKM can kernel panic the entire system (kernel crash)

Anatomy of a Kernel Module

Every LKM has two mandatory functions:

  • module_init(my_init) — Called when module is inserted (insmod)
  • module_exit(my_exit) — Called when module is removed (rmmod)
C — Minimal Hello World LKM
#include <linux/init.h>
#include <linux/module.h>
#include <linux/kernel.h>

MODULE_LICENSE("GPL");
MODULE_AUTHOR("Arpit");
MODULE_DESCRIPTION("Hello World Kernel Module");
MODULE_VERSION("1.0");

/* Called when module is loaded */
static int __init hello_init(void) {
    printk(KERN_INFO "Hello from kernel space!\n");
    return 0;   /* 0 = success; non-zero = failure (module not loaded) */
}

/* Called when module is removed */
static void __exit hello_exit(void) {
    printk(KERN_INFO "Goodbye from kernel space!\n");
}

module_init(hello_init);
module_exit(hello_exit);
💡
printk vs printf: printk() is the kernel's print function. It writes to the kernel ring buffer, readable via dmesg. Log levels: KERN_EMERG, KERN_ALERT, KERN_CRIT, KERN_ERR, KERN_WARNING, KERN_NOTICE, KERN_INFO, KERN_DEBUG (0 = highest priority).

Character Device Driver (LKM Structure)

The most common LKM type is a character device driver, which implements a file_operations struct:

C — Character Device Skeleton
#include <linux/fs.h>
#include <linux/cdev.h>

static struct file_operations my_fops = {
    .owner   = THIS_MODULE,
    .open    = my_open,       /* called on open() */
    .release = my_release,    /* called on close() */
    .read    = my_read,       /* called on read()  — uses copy_to_user */
    .write   = my_write,      /* called on write() — uses copy_from_user */
    .unlocked_ioctl = my_ioctl, /* called on ioctl() */
};

static int __init my_init(void) {
    /* 1. Allocate major/minor device number */
    alloc_chrdev_region(&dev_num, 0, 1, "mydev");
    /* 2. Initialize cdev and add to kernel */
    cdev_init(&my_cdev, &my_fops);
    cdev_add(&my_cdev, dev_num, 1);
    /* 3. Create /dev/mydev automatically */
    my_class = class_create(THIS_MODULE, "mydev_class");
    device_create(my_class, NULL, dev_num, NULL, "mydev");
    return 0;
}
📌
Device Files: Character devices appear as files in /dev/ (e.g., /dev/tty, /dev/null, /dev/random). User programs open()/read()/write() these files, which triggers the corresponding LKM functions via the VFS layer.

  • MODULE_LICENSE("GPL") — non-GPL modules "taint" the kernel (kernel marks itself as having non-free code)

Live Migration — Pre-Copy vs Post-Copy

VM Live Migration Strategies
Pre-Copy Migration (most common — VMware, KVM default):
  1. Copy all memory pages to destination while VM runs
  2. Re-copy pages that became dirty during transfer
  3. Iterate until dirty set is small enough
  4. Brief pause → copy remaining dirty pages + CPU state
  5. Resume VM on destination
  Downtime: typically 10-100 ms
  Risk: convergence — if write rate > transfer rate, never finishes

Post-Copy Migration:
  1. Pause VM, transfer only CPU state + minimal pages
  2. Resume VM on destination immediately
  3. Pages fetched on demand (network page faults)
  Downtime: very short (ms)
  Risk: if source crashes after migration, pages are lost
  Used when pre-copy doesn't converge (write-heavy workloads)

Dirty Page Tracking:
  Hypervisor write-protects pages → traps on write → marks dirty
  Dirty bitmap sent with each iteration

Nested Virtualization

Running a hypervisor inside a VM — a VM running VMs. Useful for testing hypervisors, cloud provider development, CI pipelines. Intel VT-x and AMD-V support nested virtualization via VMCS shadowing. Performance penalty is significant (triple-layer context switching). Example: Android emulator uses nested virtualization on cloud VMs.

Trap-and-Emulate, Binary Translation, Shadow Page Tables

Virtualization Techniques Deep Dive
Trap-and-Emulate:
  Guest runs in user mode → privileged instructions trap to VMM
  VMM emulates the instruction's effect → returns to guest
  Requires: all sensitive instructions ARE privileged (Popek-Goldberg)
  Problem: x86 has 17 sensitive-but-unprivileged instructions

Binary Translation (VMware classic approach):
  VMM scans guest code blocks at runtime
  Replaces problematic instructions with safe equivalents
  Translated blocks cached for reuse
  No hardware support needed, but ~10-20% overhead

Shadow Page Tables:
  Guest has its own page tables (virtual → guest-physical)
  VMM maintains shadow tables (virtual → host-physical)
  On guest PT update → VMM intercepts and updates shadow
  Overhead: frequent page faults for synchronization

Nested/Extended Page Tables (EPT/NPT — hardware approach):
  Hardware walks two levels: guest PT + EPT
  No shadow table maintenance needed
  TLB caches combined translations
  ~5% overhead vs ~20% for shadow page tables

MCQ — Process Creation via Shell
Q. Consider the following C program:
   int main() {
       fork();
       fork();
       printf("Hello\n");
       return 0;

Answer: 4

Reasoning:
  After 1st fork(): 2 processes (parent + child)
  After 2nd fork(): each of the 2 processes forks again → 4 processes
  Each of the 4 processes executes printf("Hello\n") once
  Total = 4

(assuming each fork() succeeds and is unconditional)

Q1. Define an Operating System. Explain its two main roles with examples. ›

Definition: An OS is system software that manages hardware resources and provides services for application programs. It acts as an intermediary between users/applications and hardware.

Role 1 — Resource Manager: The OS allocates CPU time, memory, I/O devices, and file storage among competing processes. Example: Linux kernel scheduler allocates CPU to processes using CFS (Completely Fair Scheduler).

Role 2 — Extended Machine / Virtual Machine: The OS hides hardware complexity and presents a clean, abstract interface. Example: A user calls write(fd, buf, n) without needing to know the disk controller's registers or DMA configuration.

Q3. What is dual-mode operation? Why is it necessary? Give examples of privileged instructions. ›

Dual-mode operation means the CPU can execute in two protection levels:

  • User Mode (mode bit = 1): Restricted. User programs run here. Cannot access hardware directly.
  • Kernel Mode (mode bit = 0): Full access. OS runs here. All instructions allowed.

Necessity: Without dual modes, a buggy or malicious user program could corrupt the OS, overwrite interrupt vectors, or directly control hardware — crashing the system.

Privileged Instructions (kernel-only):

  • HLT — halt the CPU
  • I/O port instructions (IN/OUT)
  • Interrupt enable/disable (STI/CLI)
  • Load/store to CR3 (page table base register)
  • LGDT/LIDT — load GDT/IDT registers

Executing a privileged instruction in user mode causes a General Protection Fault (hardware trap → OS handles it, usually terminating the offending process).

Q5. Explain the Linux boot process step by step from power-on to login prompt. ›
  1. BIOS Path: Firmware runs POST (hardware check). Finds bootable device. Loads first 512 bytes (MBR) into RAM at 0x7C00. MBR contains bootstrap code (446 bytes) + partition table + boot signature 0x55AA. Loads GRUB stage 1.
  2. UEFI Path: Modern firmware runs POST and loads UEFI bootloader from EFI system partition (FAT32). Supports GPT partition tables (no 512-byte limit). Directly loads GRUB stage 2 or kernel.
  3. GRUB2: Displays boot menu. Loads compressed kernel (vmlinuz) and initial RAM disk (initramfs) into memory. Passes kernel parameters.
  4. Kernel Init: Decompresses itself. Sets up CPU (GDT, IDT, paging). Initializes memory, interrupts, device drivers. Mounts initramfs as temporary root.
  5. initramfs: Contains drivers needed to mount real root filesystem (LVM, encrypted disks). Runs /init script. Pivots to real root via switch_root.
  6. systemd (PID 1): First user-space process. Reads unit files. Starts services in parallel. Reaches graphical.target. Launches login manager (GDM/getty).
Q6. What is a system call? How does it differ from a function call? Explain copy_from_user. ›

System call: A programmatic interface for user programs to request OS kernel services. Triggers a trap (mode switch: user → kernel).

Function call vs System call:

  • Function call stays in user mode, ~nanoseconds
  • System call requires mode switch (trap → handler → return), ~100–1000 ns
  • printf() is a library function that internally calls the write() system call

copy_from_user: A Linux kernel function that safely copies data from user-space memory to kernel-space memory. It validates the user pointer (ensures it's a legitimate user-space address, not a kernel address) before copying. Returns the number of bytes that could not be copied (0 on complete success). A syscall wrapper can then convert a non-zero return to -EFAULT for the user process. Direct memcpy() from a user pointer is unsafe — privilege escalation vulnerability.

Answer: B
The mode bit (part of the CPU status register) is 0 for kernel mode and 1 for user mode. It determines which instructions the CPU can execute — privileged instructions (I/O, interrupt control, halt) can only run in kernel mode. A system call sets mode bit to 0; return from kernel sets it to 1.
6. Linux is an example of which kernel architecture?
  1. Pure microkernel
  2. Monolithic kernel with loadable modules
  3. Layered kernel
  4. Exokernel
View Answer & Explanation
Answer: B
Linux is a monolithic kernel — all core services (scheduling, memory, file systems, drivers) run in kernel space. However, it supports Loadable Kernel Modules (LKMs) that can be inserted/removed at runtime, giving it microkernel-like modularity without the IPC overhead. This is sometimes called a "modular monolithic" design.

2.1 Process Concepts

A process is a program in execution — a dynamic entity with its own address space, registers, stack, and OS resources. A program is a passive, static file on disk; the moment the OS loads it into memory and creates a PCB (Process Control Block) for it, it becomes a process.

Program vs Process — Core Differences

💡 Key Insight: Running the same program twice creates two separate processes — each with its own PCB, address space, and independent execution state. They share the code segment (read-only) but have separate stack and heap.
🎓 University Perspective: A process requires four fundamental resources: CPU time (scheduling), memory (address space), files (open file descriptors), and I/O devices. The OS manages allocation and deallocation of all four across the process lifetime.
Q: Which of the following is TRUE about a process?
(a) A process is a program stored on disk
(b) Two processes from the same program share heap memory
(c) A process is a program in execution with its own resources
(d) A process exists permanently
Answer(c) A process is a program in execution. It has its own PCB, address space, CPU registers, and OS-allocated resources. Two processes from the same program do NOT share heap (only read-only code may be shared).
⚡ GATE Trap: "A process is a program" is WRONG. A process is an instance of a program in execution. A single program can spawn multiple processes (e.g., Firefox opens 10 tabs → 10+ processes in some models).
GATE 2018: Consider a program with 3 global variables. If we create 2 processes from this program using fork(), how many copies of the global variables exist?
(a) 3   (b) 6   (c) 2   (d) 1
Answer(b) 6 — After fork(), the child gets a complete copy of the parent's address space (copy-on-write). Both processes have their own 3 global variables = 6 total.
🔬 Advanced: Copy-On-Write (COW)
Modern OS uses copy-on-write — after fork(), parent and child share the same physical pages marked read-only. Only when either process writes to a page does the OS create a private copy. This makes fork() O(1) instead of O(address space size). Linux implements this via page fault handler + MMU page table duplication.

Q: Which PCB field is restored when a process is dispatched after a context switch?
(a) PID   (b) Program Counter   (c) UID   (d) File descriptor table
Answer(b) Program Counter (PC) — The CPU registers including PC, SP, and GPRs are saved to the PCB on preemption and restored when the process is dispatched again. The PID and UID don't change.
GATE 2016: The information about a process that is NOT part of the PCB is:
(a) I/O status information   (b) CPU registers   (c) Scheduling information   (d) Contents of the stack
Answer(d) The PCB stores a pointer to the stack (stack pointer register) but not the actual contents of the stack. The stack contents reside in the process's own address space in memory.
⚡ GATE Key: PCB is stored in the kernel address space (protected memory) — not in the process's own user address space. A user process cannot directly read or modify its own PCB.
🔬 Linux task_struct: In Linux, the PCB is the task_struct (~700 fields, ~4KB). Key fields: pid, state, mm (memory descriptor), files (file descriptor table), signal, thread (CPU register save area). All task_structs are linked in a doubly-linked circular list. init_task (PID 0, the idle process) is the list head.

2.4 Context Switching & Process Operations

Every time the OS must run a different process — whether due to time quantum expiry, a blocking I/O call, or a higher-priority process becoming ready — it must perform a context switch. This operation is the heart of multitasking: it creates the illusion that multiple processes run simultaneously on a single-core machine, and enables true parallelism on multi-core machines.

The context switch has two distinct phases: save (capturing the current CPU state into the outgoing process's PCB) and restore (loading the incoming process's saved state from its PCB into the CPU). During this time, the CPU does no useful work — it's pure overhead. System designers aim to minimise both the frequency and duration of context switches. Modern CPUs help with fast register save/restore instructions (e.g., x86 FXSAVE/FXRSTOR for floating-point state).

A context switch is the mechanism by which the OS saves the state of one process and restores the state of another. It is pure overhead — no useful work happens during the switch.

Context Switch Steps
StepActionCost
1Save CPU registers of current process → PCB~µs
2Update process state (Running → Ready/Blocked)Negligible
3Move current PCB to appropriate queueNegligible

⚠️ Context Switch Cost: A typical context switch costs 1–10 µs. The major cost is TLB flush — the new process needs to reload its page translations, causing cache misses. Threads avoid this cost because they share the same address space (no TLB flush needed).

Process Operations

Unix/Linux process creation follows the fork-exec model — a deliberate separation of process cloning from program loading. fork() creates an exact copy of the parent (cheap due to copy-on-write), giving both parent and child the opportunity to configure their environment — redirect file descriptors, set signal handlers, close unnecessary handles. The child then calls exec() to replace its address space with a new program. This two-step model gives the shell extraordinary flexibility for I/O redirection, pipes, and job control — all configured between fork() and exec(), without modifying the parent.

The wait() system call prevents zombie accumulation. Every terminated child becomes a zombie — its PCB holds the exit code until the parent retrieves it via wait(). In production servers that fork thousands of connections, forgetting to call wait() exhausts the PID namespace and crashes the system. Signal-based zombie reaping (SIGCHLD handler calling waitpid(-1, NULL, WNOHANG)) is the standard solution for long-running daemons.

#include <stdio.h>
#include <unistd.h>
#include <sys/wait.h>

int main() {
    pid_t pid = fork();          // Creates child process

    if (pid < 0) {
        perror("fork failed");   // Error
    } else if (pid == 0) {
        // CHILD process (fork returns 0)
        printf("Child PID: %d\n", getpid());
        execlp("ls", "ls", "-la", NULL);  // Replace with 'ls'
        // exec() never returns on success
    } else {
        // PARENT process (fork returns child's PID)
        printf("Parent PID: %d, Child PID: %d\n", getpid(), pid);
        wait(NULL);              // Wait for child to finish
        printf("Child finished\n");
    }
    return 0;
}
Fork Bomb: :(){ :|:& };: — A function that calls itself twice, spawning exponentially many processes until the system runs out of PIDs and RAM. Prevented by limiting max processes per user (ulimit -u).
GATE 2014: Consider the code: fork(); fork(); fork(); How many total processes exist after all forks?
(a) 3   (b) 6   (c) 7   (d) 8
Answer(d) 8 — Each fork() doubles the number of processes. Start: 1. After fork 1: 2. After fork 2: 4. After fork 3: 8. Formula: 2^n where n = number of fork() calls (assuming sequential, unconditional forks).
GATE: After fork(), which of the following is NOT shared between parent and child?
(a) Code segment   (b) Open file descriptors   (c) Heap contents (initial copy)   (d) PID
Answer(d) PID — After fork(), the child gets a new unique PID. The code segment is shared (read-only). File descriptors are inherited (duplicated). The heap is COW-copied (initially shared, split on write).

2.5 Threads & Multithreading

Modern operating systems are built around the concept of concurrency — multiple tasks appearing to progress simultaneously. This is achieved through two fundamental abstractions: processes and threads. While processes provide strong isolation and independent address spaces, they are heavyweight and expensive to create and communicate between. Threads solve this by providing lightweight concurrent execution within a shared address space.

A thread is the smallest unit of CPU execution within a process — a lightweight process sharing the process's address space. A process with multiple threads runs concurrent tasks sharing code, data, and files but with independent stacks and registers.

Multithreading Models

The relationship between user-space threads (managed by the thread library) and kernel threads (managed by the OS) determines whether true parallelism is possible. In the Many-to-One model, N user threads map to a single kernel thread — simple but no parallelism, and one blocking call stalls all threads. The One-to-One model (Linux pthreads, Windows) maps each user thread to its own kernel thread — true multi-core parallelism, but higher per-thread overhead. The Many-to-Many model multiplexes N user threads onto M kernel threads (M≤N), balancing parallelism with overhead, but its complexity makes it rare in modern systems. Linux uses One-to-One exclusively via the clone() system call, which creates kernel-scheduled tasks sharing configurable resources.

Q: In the Many-to-One threading model, if one thread makes a blocking system call:
(a) Only that thread blocks   (b) All threads in the process block   (c) The kernel creates a new thread   (d) The process terminates
Answer(b) In Many-to-One, all N user threads are multiplexed onto a single kernel thread. When the kernel thread blocks (e.g., on I/O), all user threads block too — the kernel sees only one thread.
⚡ GATE Key: Linux uses the One-to-One model — each pthread maps to a kernel thread via clone() syscall. The kernel schedules kernel threads, so true parallelism is possible on multi-core CPUs.
GATE 2020: Which of the following is shared between threads of the same process?
(a) Stack pointer   (b) Program counter   (c) Heap memory   (d) CPU registers
Answer(c) Heap memory — Threads share the heap (and code, data, globals, files). Each thread has its own stack pointer, PC, and register set.
🔬 Thread-Local Storage (TLS) & errno: The global errno variable in C is thread-local — each thread has its own copy (implemented via TLS). Without TLS, concurrent system calls in different threads would corrupt each other's error codes. Linux uses the __thread keyword for TLS.

2.6 Inter-Process Communication (IPC)

The fundamental design principle of modern OS process isolation — each process has its own address space that cannot be directly accessed by other processes — is a crucial security and stability feature. However, real-world applications often require cooperation between processes. A web server needs to communicate with a database. A shell pipeline connects ls to grep to wc. A GUI application spawns worker processes for computation. All these require Inter-Process Communication.

IPC mechanisms vary widely in their design philosophy, performance characteristics, and use cases. The OS designer faces a fundamental trade-off: shared memory offers raw speed but requires manual synchronisation; message passing is safer and simpler but involves data copying overhead. Modern systems provide both, allowing programmers to choose the right tool for each situation.

Processes are isolated — they cannot directly access each other's memory. IPC mechanisms allow controlled communication and synchronisation between processes.

Shared Memory vs Message Passing

These two paradigms represent fundamentally different IPC philosophies. Shared memory is a memory-access model: after one-time setup via shm_open() or mmap(), processes read and write to the shared region directly — zero OS involvement per operation, giving maximum throughput. The programmer bears full responsibility for synchronisation using semaphores or mutexes to prevent concurrent corruption. Message passing is a data-exchange model: the sender packages data into a message and invokes send(); the kernel copies it to a buffer; the receiver calls receive() to get a private copy. Synchronisation is implicit — send/receive are atomic operations. The kernel copy per message is the performance cost. For bulk high-bandwidth data (video processing, shared database buffers), shared memory wins. For structured control messages and commands, message passing provides cleaner code with built-in isolation.

2.7 CPU Scheduling — Concepts & Criteria

CPU scheduling is one of the most critical functions of an operating system. In a multiprogramming system, several processes reside in memory simultaneously. When one process is waiting for I/O, another can use the CPU — this is the fundamental basis of CPU utilisation. The degree of multiprogramming is the number of processes in memory at any time.

The long-term scheduler (job scheduler) decides which processes are admitted into the ready queue from the job pool — it controls the degree of multiprogramming. The short-term scheduler (CPU scheduler) decides which process in the ready queue gets the CPU next — it is invoked very frequently (milliseconds). The medium-term scheduler handles swapping — temporarily removing processes from memory to reduce the degree of multiprogramming.

The CPU scheduler selects which Ready-queue process gets the CPU next. Scheduling decisions happen at four events: (1) process switches to Waiting, (2) process terminates, (3) new process arrives (preemptive), (4) time quantum expires (preemptive).

// Calculate scheduling metrics from Gantt chart
// Given: CT values for each process
int AT[] = {0, 2, 4, 5};   // Arrival times
int BT[] = {8, 4, 9, 5};   // Burst times
int CT[] = {8,12,26,17};    // Completion times (from Gantt — FCFS example)
int n = 4;

float sum_tat=0, sum_wt=0;
for(int i=0; i<n; i++) {
    int tat = CT[i] - AT[i];   // Turnaround = Completion - Arrival
    int wt  = tat  - BT[i];   // Wait = TAT - Burst
    sum_tat += tat; sum_wt += wt;
    printf("P%d: TAT=%d, WT=%d\n", i+1, tat, wt);
}
printf("Avg TAT=%.2f, Avg WT=%.2f\n", sum_tat/n, sum_wt/n);

2.8 FCFS, SJF & Non-Preemptive Scheduling

FCFS — First Come First Served

FCFS implements the most natural fairness intuition — first come, first served, exactly as a single-line queue at a service counter. Every arriving process joins the tail of the FIFO ready queue, and the CPU processes from the head without interruption. Its implementation requires only a simple queue data structure and zero scheduling decisions. However, this apparent simplicity masks a fundamental unfairness: the length of a process\'s wait depends entirely on what arrived before it, not on its own resource needs. A 1ms calculation job can be held for 100ms behind a single CPU-intensive job, wasting the CPU\'s potential and degrading system responsiveness.

FCFS is the most intuitive scheduling algorithm — it mirrors how humans naturally queue for service. Every process that arrives is added to the tail of the ready queue, and the CPU always serves the process at the head. Once a process gets the CPU, it runs to completion (or until it blocks for I/O) — there is no preemption. This simplicity makes FCFS easy to implement using a simple FIFO queue, but this same simplicity leads to its key weakness: the convoy effect.

The simplest algorithm — processes are served in arrival order. Non-preemptive. Suffers from the Convoy Effect where short processes wait behind long ones.

2.9 SRTF, Round Robin & Priority Scheduling

SRTF — Shortest Remaining Time First (Preemptive SJF)

SRTF extends SJF's core insight — that serving shorter jobs first minimises total waiting — into the preemptive domain. At every moment a new process arrives, the scheduler compares the newly arrived process's burst time against the remaining burst time of the currently running process. If the newcomer has a shorter remaining burst, it immediately preempts the running process, which is moved back to the ready queue to wait. This aggressive preemption strategy ensures that the CPU is never wasted on a long job when a short one is available.

The theoretical optimality of SRTF comes with a practical cost: it requires continuous knowledge of each process's remaining burst time — information that is unknown in advance. Real implementations must use burst time estimation (exponential averaging). Additionally, very frequent preemptions for marginal differences increase context switch overhead. Despite these limitations, SRTF is invaluable as a theoretical benchmark — it defines the minimum achievable average waiting time for any preemptive scheduling algorithm on a given workload.

Preemptive version of SJF. At each time unit, the process with the smallest remaining burst time runs. If a new process arrives with shorter remaining time, it preempts the current process. Optimal for minimising average WT among all preemptive algorithms.

Round Robin (RR)

Round Robin was designed specifically for time-sharing systems where fairness and responsiveness matter more than raw throughput. The fundamental insight is simple: no process should monopolise the CPU. By giving each process a fixed quantum of CPU time, RR ensures that every ready process gets CPU access within (n-1)×q time units. This bounded wait makes RR unique — no other algorithm provides a hard upper bound on how long a process waits.

The quantum size profoundly shapes RR behaviour. A large quantum degenerates to FCFS. A tiny quantum approaches true processor sharing, but context-switch overhead dominates. Systems choose 10–100ms so that most I/O-bound processes finish their short CPU bursts within one quantum (behaving like FCFS), while CPU-bound processes are preempted regularly, keeping the system responsive.

Each process gets a fixed time quantum (q) of CPU time. After q expires, the process is preempted and moved to the tail of the ready queue. Designed for time-sharing systems — no starvation.

2.10 Multilevel Queue & MLFQ

Multilevel Queue (MLQ)

As operating systems evolved to serve increasingly diverse workloads — from batch payroll processing to interactive terminals to real-time control systems — a single scheduling algorithm proved inadequate. A batch job and an interactive login shell have fundamentally different requirements: the batch job wants maximum throughput; the shell needs sub-100ms response time. Multilevel Queue scheduling addresses this by partitioning processes into separate queues based on their characteristics, with each queue implementing the algorithm best suited to its workload.

Divides the ready queue into multiple sub-queues, each with its own scheduling algorithm. Processes are permanently assigned to a queue based on type.

MLFQ — Multilevel Feedback Queue

Unlike MLQ, MLFQ allows processes to move between queues based on behaviour. CPU-bound processes move to lower queues; I/O-bound or recently interactive processes move to higher queues.

MLFQ Rules (Ousterhout's Definition):
Rule 1: If Priority(A) > Priority(B) → A runs.
Rule 2: If Priority(A) = Priority(B) → A & B run in RR.
Rule 3: New jobs start at highest priority queue.
Rule 4a: If job uses full quantum → priority demoted (move to lower queue).
Rule 4b: If job gives up CPU before quantum → priority retained.
Rule 5 (Boost): After period S, all jobs moved to top queue (prevents starvation).

I/O-bound processes that frequently give up the CPU stay in high priority queues — this naturally favours interactive/I-O-bound workloads, which is desirable for user responsiveness.
GATE: Which problem does the periodic "boost" in MLFQ solve?
(a) High context switch overhead   (b) Starvation of low-priority processes   (c) Priority inversion   (d) Memory leaks
Answer(b) Starvation — Without the boost, long-running CPU-bound processes may get stuck in the lowest queue and never run if higher queues are always busy. The periodic boost moves all processes to the top queue, giving them a fresh chance.
🔬 MLFQ Gaming: A malicious process could voluntarily give up CPU just before its quantum expires (Rule 4b), maintaining high priority indefinitely. Fix: Rule 4 (improved) — track total CPU time per queue period; demote if total allocation exceeded, regardless of voluntary/involuntary release.
MLQ Rule: A process in a higher-priority queue ALWAYS preempts lower-priority queue processes. The lower queue only runs when ALL higher queues are empty.

⚠️ Race Condition Example:
// Shared counter — race condition!
int count = 0;  // Shared between Thread A and Thread B

// Thread A:                  // Thread B:
count++;  // NOT atomic!       count++;
// count++ expands to:
// LOAD  R1, count    (R1 = 0)
// ADD   R1, 1        (R1 = 1)
// STORE count, R1    (count = 1)

// If threads interleave at machine code level:
// A: LOAD R1=0       B: LOAD R2=0
// A: ADD  R1=1       B: ADD  R2=1  
// A: STORE count=1   B: STORE count=1  ← count=1, NOT 2!
// Lost update! Expected: count=2, Got: count=1

Peterson's Solution (2-process):
// Process i (i=0 or 1, j=1-i)
flag[i] = true;      // "I want to enter"
turn = j;            // "But you go first (be polite)"
while (flag[j] && turn == j)
    ;                // Wait if j wants in AND it's j's turn
// === CRITICAL SECTION ===

// === REMAINDER SECTION ===

// Satisfies all 3 requirements:
// Mutual exclusion: Only one can pass the while loop
// Progress: If j doesn't want in (flag[j]=false), i enters immediately

⚠️ Why Synchronisation is Needed: Threads share the heap — without synchronisation, concurrent writes create race conditions. The OS cannot prevent this in user space; the programmer must use mutual exclusion primitives.

        shared_counter++;              // Critical section

}

// With mutex: result is always exactly 200000

Q: Which synchronisation primitive is most appropriate for a producer-consumer problem where the producer and consumer are different threads?
(a) Mutex only   (b) Counting semaphore   (c) Spinlock   (d) Memory barrier
Answer(b) Counting semaphore — The counting semaphore tracks available buffer slots. Producer calls sem_post() (increments), consumer calls sem_wait() (decrements, blocks if 0). Any thread can call either operation — unlike mutex which requires the same thread to unlock.
🔬 Priority Inversion: A low-priority thread L holds a mutex that high-priority thread H needs. H blocks waiting for L. But medium-priority thread M preempts L (since M has higher priority than L). Now H waits for L which is stuck behind M — H is effectively at M's priority. Solution: Priority inheritance — temporarily boost L's priority to H's level while it holds the mutex. Used in Mars Pathfinder (1997) — a real OS bug!

2.21 Key Facts, Formulas & Code Bank

Context Switch Frequency: With n processes each running for average burst t, context switches per second ≈ n/(t + c) where c = context switch time. Minimise c to maximise efficiency.
Process States Mnemonic: Never Rush Recklessly — Wait Today → New, Ready, Running, Waiting, Terminated
CRITICAL: Only ONE process can be in Running state per CPU core at any time. Multiple processes can be in Ready and Waiting states simultaneously — those queues can be arbitrarily long.
CPU Efficiency with context switch overhead:
Efficiency = BT / (BT + switch_time) × 100%
If BT=10ms, switch=1ms: Efficiency = 10/11 ≈ 90.9%
If BT=1ms, switch=1ms: Efficiency = 1/2 = 50% — switch overhead dominates!
IPC Performance Ranking (fastest to slowest):
Shared Memory > Memory-Mapped File > Pipe > Message Queue > Socket (loopback) > Network Socket
Deadlock in IPC: Two processes each write to their pipe to the other, but both pipes are full (64KB buffer). Both block waiting for the other to read — deadlock! Solution: non-blocking I/O (O_NONBLOCK) or use select()/poll() to handle multiple FDs.
Thread Overhead vs Process:
Thread creation: ~10µs | Process creation: ~1000µs (100× slower)
Thread context switch: ~1µs (no TLB flush) | Process switch: ~5-10µs
Amdahl's Law — Multithreading Limit:
Speedup = 1 / (S + (1-S)/N) where S=serial fraction, N=threads
If 20% of code is serial: max speedup → 1/0.2 = 5× regardless of N threads
Scheduling Tie-Breaking Rule: When SJF/SRTF has ties (equal burst times), the problem statement should specify. Default assumption for GATE: break ties by arrival time (earlier arrival wins); if simultaneous arrival, use process ID order (P1 before P2).
RR Quantum Selection Rule:
Good quantum q: 80% of CPU bursts should complete within q.
Typical: q = 10ms–100ms. Linux CFS doesn't use fixed quantum — uses vruntime fairness.

// Simulate FCFS scheduling in C
#include <stdio.h>
struct Process { int pid, at, bt, ct, tat, wt; };

void fcfs(struct Process p[], int n) {
    // Sort by arrival time
    for (int i=0; i<n-1; i++)
        for (int j=i+1; j<n; j++)
            if (p[i].at > p[j].at) { struct Process t=p[i]; p[i]=p[j]; p[j]=t; }
    
    int time = 0;
    for (int i=0; i<n; i++) {
        if (time < p[i].at) time = p[i].at;  // CPU idle gap
        p[i].ct = time + p[i].bt;
        p[i].tat = p[i].ct - p[i].at;
        p[i].wt = p[i].tat - p[i].bt;
        time = p[i].ct;
        printf("P%d: CT=%d TAT=%d WT=%d\n", p[i].pid, p[i].ct, p[i].tat, p[i].wt);
    }
}
// Simulate Round Robin scheduling
void round_robin(struct Process p[], int n, int q) {
    int remaining[n], time=0, done=0;
    for (int i=0; i<n; i++) remaining[i] = p[i].bt;
    
    while (done < n) {
        int progressed = 0;
        for (int i=0; i<n; i++) {
            if (remaining[i] > 0 && p[i].at <= time) {
                progressed = 1;
                int run = (remaining[i] < q) ? remaining[i] : q;
                time += run;
                remaining[i] -= run;
                if (remaining[i] == 0) {
                    done++;
                    p[i].ct = time;
                    p[i].tat = p[i].ct - p[i].at;
                    p[i].wt = p[i].tat - p[i].bt;
                }
            }
        }
        if (!progressed) time++;  // CPU idle: advance time
    }
}

AnswerSIGTERM (15): Graceful termination request. The process can catch this signal, perform cleanup (flush buffers, close files, release resources), and then exit voluntarily. Processes can ignore SIGTERM.
SIGKILL (9): Immediate forced termination by the kernel. Cannot be caught, blocked, or ignored. No cleanup possible — data loss may occur. Always effective unless process is in an uninterruptible sleep (D state).
🔬 Linux Scheduling Classes (sched_class):
Linux implements a modular scheduling framework with scheduling classes:
stop_sched_class → highest priority (migration, stopmachine)
dl_sched_class → SCHED_DEADLINE (EDF)
rt_sched_class → SCHED_FIFO, SCHED_RR (real-time)
fair_sched_class → SCHED_NORMAL (CFS — for most processes)
idle_sched_class → SCHED_IDLE (lowest, idle thread)
Each class implements a common interface: pick_next_task(), enqueue_task(), dequeue_task(). The main scheduler loop iterates from highest to lowest class.

2.22 Process & Thread Deep Dive

This section consolidates advanced process and thread knowledge essential for competitive exams and real-world OS development. Understanding how the OS bridges the gap between hardware resources and user-space programs requires mastering these inter-connected concepts.

The relationship between the hardware scheduler, OS scheduler, and user application is layered: the hardware timer interrupt fires at a configurable frequency (e.g., 250Hz in Linux = every 4ms), triggering the OS interrupt handler which invokes the CPU scheduler. The scheduler consults the ready queue data structure, applies the scheduling algorithm, selects the next process, performs the context switch, and transfers control to user space — all within a few microseconds.

Key Insight — Scheduler Invocation:
The Linux scheduler is called in four scenarios: (1) explicitly when a process blocks (schedule()), (2) on return from a system call if need_resched flag is set, (3) on return from an interrupt handler, (4) when a higher-priority process wakes up (WAKEUP preemption). The need_resched flag is set by the timer interrupt handler if the current process has exhausted its virtual runtime allocation.

Process Priority Inversion — Complete Example

Priority inversion is a subtle scheduling bug where a high-priority task is effectively blocked by a lower-priority task. This was famously discovered in the Mars Pathfinder mission (1997). The rover experienced unexpected system resets because a low-priority meteorological data collection task held a mutex that a high-priority bus management task needed. A medium-priority communications task kept preempting the low-priority task, keeping the mutex locked and starving the high-priority task — violating real-time deadlines.

Priority Inversion Scenario:
1. L (low priority) acquires mutex M.
2. H (high priority) tries to acquire M — blocks (correct behaviour).
3. K (medium priority) preempts L (since K > L in priority).
4. L is stuck behind K; M stays locked; H waits for M → H waits for K!
5. H (highest) effectively runs at K's (medium) priority — INVERSION!
Fix: Priority inheritance — OS temporarily raises L's priority to H's level while L holds M.
// Demonstrating priority inheritance (POSIX mutex with PI)
#include <pthread.h>

pthread_mutexattr_t attr;
pthread_mutex_t pi_mutex;

int setup_pi_mutex() {
    pthread_mutexattr_init(&attr);
    // Enable Priority Inheritance protocol
    pthread_mutexattr_setprotocol(&attr, PTHREAD_PRIO_INHERIT);
    pthread_mutex_init(&pi_mutex, &attr);
    return 0;
}

// high-priority thread H blocks on it:
// → OS automatically boosts L's priority to H's level
// → L finishes critical section quickly
// → H unblocks and runs at full priority

CPU Affinity & NUMA

CPU Affinity is the property of binding a thread or process to run on specific CPU cores. On multi-core systems, migrating a process between cores causes cache pollution (the old core's warm cache is wasted). Setting CPU affinity prevents migration, improving cache performance for latency-sensitive applications like databases and real-time audio.

// Set CPU affinity — bind process to cores 0 and 1 only
#define _GNU_SOURCE
#include <sched.h>
#include <stdio.h>

int main() {
    cpu_set_t mask;
    CPU_ZERO(&mask);      // Clear all CPUs
    CPU_SET(0, &mask);    // Allow CPU 0
    CPU_SET(1, &mask);    // Allow CPU 1
    
    if (sched_setaffinity(0, sizeof(mask), &mask) != 0) {
        perror("sched_setaffinity failed");
        return 1;
    }
    printf("Process now runs only on CPU 0 or 1\n");
    
    // Verify
    cpu_set_t get_mask;
    sched_getaffinity(0, sizeof(get_mask), &get_mask);
    for (int i = 0; i < 4; i++)
        printf("CPU %d: %s\n", i, CPU_ISSET(i, &get_mask) ? "YES" : "NO");
    return 0;
}

Q: What is the primary benefit of setting CPU affinity for a thread?

GATE: In a multiprocessor system with N CPUs, the maximum speedup achievable for a program that is 50% parallelisable is (by Amdahl's Law, as N→∞):
(a) N   (b) 2   (c) 0.5   (d) N/2
Answer(b) 2× — Amdahl's Law: Speedup = 1/(S + (1-S)/N). With S=0.5 (50% serial), as N→∞: Speedup → 1/0.5 = 2. The serial portion limits maximum speedup regardless of how many processors you add.
🔬 Work-Stealing Scheduler: Modern parallel runtimes (Go, Java ForkJoin, Cilk) use work-stealing. Each worker thread has its own deque of tasks. When a thread's deque is empty, it steals tasks from the tail of another thread's deque (victim). This achieves excellent load balancing with minimal synchronisation — workers steal only when idle, so the hot path (local task execution) is lock-free.

2C.1 Introduction to Deadlocks

🎓 University
Deadlock
A set of processes is in a deadlocked state when every process in the set is waiting for an event that can only be caused by another process in the set. Since all processes are waiting, none can ever proceed — the system is permanently stuck.

Real-World Analogy

Two cars on a one-lane bridge approaching from opposite ends. Each car is waiting for the other to back up — neither can proceed. This is deadlock: circular dependency, no external rescue.

System Model — Resources

A system consists of a finite number of resources distributed among competing processes. Resources include:

Resource TypeExamplesInstances
PhysicalCPU cores, printers, disk drives, memoryMultiple (e.g., 3 printers)
LogicalFiles, semaphores, mutex locks, socketsUsually 1 (binary)
I/O DevicesScanner, tape drive, GPU1 or few

Resource Usage Protocol

Every process must follow this 3-step protocol:

  1. Request: Process requests the resource. If not available, it waits.
  2. Use: Process operates on the resource (critical section).
  3. Release: Process frees the resource for others.

Approaches to Handle Deadlock

ApproachStrategyCostUsed by
PreventionEnsure at least one of the 4 conditions never holdsLow utilization, possible starvationSpecial-purpose systems
AvoidanceDynamically grant resources only if safe state maintainedRequires advance resource infoBatch systems (Banker's)
Detection & RecoveryAllow deadlock, detect it, then recoverOverhead of detection + recovery costDatabases, general OS
Ignore (Ostrich Algorithm)Pretend deadlocks never occurNone (until crash)Most desktop OS (Windows, Linux)
💡
Ostrich Algorithm: Most general-purpose OS (including Linux and Windows) simply ignore deadlocks! The reasoning: deadlocks are rare, and the overhead of preventing/detecting them would degrade performance for the common case. Users just reboot if a deadlock occurs.

Livelock — The Active Cousin

Livelock is similar to deadlock but processes are NOT blocked — they keep changing state in response to each other but make no progress. Example: two people in a corridor, each stepping aside to let the other pass, but both step to the same side repeatedly. Livelock wastes CPU (unlike deadlock which is idle). Fix: add randomization or backoff.

⚡ GATE
  • Deadlock requires ALL four conditions simultaneously — removing even ONE prevents deadlock.
  • Deadlock can involve software resources (locks, semaphores) not just hardware devices.

Starvation vs Deadlock — Key Distinction

💡
GATE Trick: "Every deadlock is starvation, but not every starvation is deadlock." In deadlock, processes are starved permanently. In starvation, the process might eventually get the resource.

Multi-Instance RAG: Cycle ≠ Deadlock!

RAG with Multiple Instances
Single-instance resources:
  Cycle in RAG → DEADLOCK (guaranteed)

Multi-instance resources:
  Cycle in RAG → POSSIBLE deadlock, NOT guaranteed!

Example (NO deadlock despite cycle):
  Resource R has 2 instances, Resource S has 1 instance.
  P1 holds R(instance 1) and requests S.
  P2 holds S and requests R.
  → Cycle: P1→S→P2→R→P1
  But R has a FREE instance → P2 gets R, completes, releases S.
  Then P1 gets S, completes. NO DEADLOCK.

Example (DEADLOCK with cycle, multi-instance):
  Resource R has 2 instances. S has 1 instance.
  P1 holds R(1), requests S.
  P2 holds R(2), requests S.
  P3 holds S, requests R.
  All 2 R instances held. S held. No free instances anywhere.
  → DEADLOCK: P1↔S↔P3↔R↔P1, and P2 also stuck.

Rule for GATE:
  • Single instance: cycle = deadlock ✅
  • Multiple instances: use Banker's/detection algorithm ✅
  • Cycle is necessary but NOT sufficient for multi-instance

2C.2 Four Necessary Conditions for Deadlock

🎓 University

Deadlock can arise if and only if all four of the following conditions hold simultaneously. These are known as the Coffman Conditions (1971).

01
Mutual Exclusion
At least one resource must be held in a non-sharable mode. Only one process at a time can use the resource. If another requests it, the requesting process must wait.
02
Hold and Wait
A process is holding at least one resource and is waiting to acquire additional resources currently held by other processes.
03
No Preemption
Resources cannot be forcibly taken from a process. A resource can only be released voluntarily by the process holding it after it has completed its task.
04
Circular Wait
A set of processes {P0, P1, …, Pn} where P0 is waiting for a resource held by P1, P1 waiting for P2, …, and Pn waiting for P0. A closed chain of dependency.
⚠️
All Four Must Hold: Deadlock is impossible unless ALL four conditions hold simultaneously. This is the foundation of all prevention strategies — if you can guarantee that even one condition can never hold, deadlock is impossible.

Classic Deadlock Trace

C — All Four Conditions Present
pthread_mutex_t A, B;

/* Thread 1 */                    /* Thread 2 */
lock(A);  ← holds A              lock(B);  ← holds B
lock(B);  ← waits for B          lock(A);  ← waits for A
// DEADLOCK: T1 holds A, waits B. T2 holds B, waits A.
// Condition 1: ME (only one can hold each mutex)
// Condition 2: Hold-and-Wait (each holds one, waits for other)
// Condition 3: No Preemption (OS won't force-release the lock)
// Condition 4: Circular Wait (T1→B→T2→A→T1)
📌 Section 4.1–4.3 Deadlock — Key Takeaways
  • HEMP mnemonic: Hold & Wait | Exclusion (Mutual) | Missing Preemption (No Preemption) | Path (Circular Wait). ALL 4 needed simultaneously
  • Preventing deadlock: Eliminate ANY ONE of the 4 conditions. Circular Wait (resource ordering) is the most practical
  • RAG single-instance: Cycle ⟺ Deadlock (biconditional). Multi-instance: Cycle is necessary but NOT sufficient
  • Safe vs Unsafe: Safe → no deadlock guaranteed. Unsafe → MAY deadlock (not certain). Deadlocked ⊂ Unsafe ⊂ All states
  • Banker's formula: Need[i] = Max[i] − Allocation[i]. Grant only if resulting state is SAFE (safe sequence exists). O(m×n²)
Active Recall: Coffman Conditions
1. Without looking above, what are the four Coffman conditions?
  • Mutual Exclusion (non-sharable resources)
  • Hold and Wait (holding a resource while requesting another)
  • No Preemption (resources cannot be forcibly taken)
  • Circular Wait (closed chain of dependency)
2. Are all 4 required for a deadlock?
Yes, they are necessary conditions. If even one is broken, deadlock is impossible.

3. A system is in an unsafe state. Is it deadlocked?
Not necessarily. Unsafe means there is no safe sequence — deadlock is possible, not guaranteed. A process may still finish without requesting max resources.
⚡ GATE
🧠 HEMP Mnemonic — Remember All 4 Coffman Conditions:
Hold and Wait  |  Exclusive Access (Mutual Exclusion)  |  Make Circular (Circular Wait)  |  Preemption — None
OR alternate: Hold-wait, Exclusive, Mutual-wait-circle (circular), Preemption-free
All 4 must be HEMP-resent at the same time for deadlock → break any one to prevent deadlock.
  • These are necessary but not sufficient conditions — all four together don't guarantee deadlock, but deadlock is impossible without all four.
  • Circular wait ⊇ Hold-and-Wait — Circular wait implies hold-and-wait, but not vice versa. If you prevent circular wait, you automatically prevent deadlock without needing to prevent hold-and-wait separately.
  • GATE often asks: "Which condition is easiest to prevent in practice?" → Circular wait (via resource ordering).
  • GATE trap: "Deadlock requires 4 conditions" — True. "4 conditions guarantee deadlock" — False (they are necessary, not sufficient for multi-instance resources).

2C.3 Resource Allocation Graph (RAG)

🎓 University
Resource Allocation Graph (RAG)
A directed graph G = (V, E) that precisely describes the deadlock state of a system. Vertices V = Processes (circles) ∪ Resources (rectangles). Edges E = Assignment edges + Request edges.

Graph Components

ElementNotationMeaning
ProcessCircle: ○ PiA process in the system
Resource TypeRectangle: □ Rj (with dots = instances)A resource with one or more instances
Request EdgePi → Rj (process to resource)Pi is requesting an instance of Rj
Assignment EdgeRj → Pi (resource to process)One instance of Rj is assigned to Pi

Deadlock Detection Using RAG

Example 1: Deadlock Scenario (Single-Instance Resources)
Cycle: P1 → R1 → P2 → R2 → P1 (DEADLOCK!)
Example 2: No Deadlock (Multi-Instance Resource)
  • R has 2 instances, S has 1 instance.
  • P1 holds R(1) and requests S; P2 holds S and requests R.
  • This forms a cycle: P1 → S → P2 → R → P1.
  • But one instance of R is still free, so another process (or P2) can get it, complete, and release S. Then P1 gets S and completes.
→ Cycle exists but no deadlock (extra instance breaks the wait).

RAG Rules for Deadlock Detection

Resource InstancesCycle in RAGConclusion
Single instance per typeCycle exists✅ Deadlock definitely exists
Single instance per typeNo cycle✅ No deadlock
Multiple instances per typeCycle exists⚠️ Deadlock may exist (not certain)
Multiple instances per typeNo cycle✅ No deadlock
📌
Key Rule: For single-instance resources, RAG cycle detection is equivalent to deadlock detection. For multi-instance resources, you need the more complex Detection Algorithm (Section 2C.9).
WFG vs RAG: The Wait-For Graph is derived from the Resource Allocation Graph by collapsing resource nodes — if P→R→Q in RAG (P waits for a resource held by Q), then P→Q directly in WFG. WFG is simpler and only exists for single-instance resources.

Wait-For Graph (WFG)

A simplified version of RAG used when all resources have exactly one instance. Resource nodes are removed — only processes remain. An edge Pi → Pj means "Pi is waiting for Pj to release a resource." A cycle in the WFG = deadlock.

⚡ GATE
  • RAG has a cycle → necessary condition for deadlock. Deadlock → cycle always exists in RAG.
  • Cycle in RAG with single-instance resources ⟺ Deadlock (iff). With multi-instance: cycle is necessary but not sufficient.
  • GATE often draws a RAG and asks whether deadlock exists — check for cycles and count instances carefully.
  • In a WFG with n processes, deadlock exists iff a cycle exists (detect with DFS in O(n²)).
Active Recall: Resource Allocation Graphs (RAG)
1. If a Resource Allocation Graph has a cycle, is there definitely a deadlock?
If all resources have only a single instance, YES. If resources have multiple instances, NO — a cycle is necessary but not sufficient, because an outside process could release an instance to break the cycle.

2. What is a Wait-For Graph (WFG)?
A simplified RAG where resource nodes are removed, leaving only processes. An edge P1 → P2 means P1 is waiting for a resource held by P2. It is only valid for single-instance resources.

2C.4 Deadlock Prevention

🎓 University

Prevention works by ensuring that at least one of the four Coffman conditions can never hold. This is done statically — the system is designed so that deadlock is structurally impossible.

Prevention Cheat — Break ANY ONE condition: Easiest to break: Circular Wait (impose resource ordering). Hardest: Mutual Exclusion (can't always share). Breaking Hold & Wait hurts utilisation; breaking No-Preemption risks data loss.

1. Deny Mutual Exclusion

Make resources sharable. Problem: Many resources are inherently non-sharable (e.g., a printer, a mutex). You can't print two documents simultaneously on one printer. This condition generally cannot be denied.

2. Deny Hold and Wait

Two protocols:

  • Protocol 1 (All-or-Nothing): A process must request ALL resources it will ever need before starting. If any are unavailable, it waits holding none. Problem: Low resource utilization. A process holds resources it won't need for hours.
  • Protocol 2 (Release-before-Request): Before requesting a new resource, a process must release all currently held resources. Problem: Starvation — a process may never get all needed resources simultaneously.

3. Allow Preemption

If a process holding resources requests another that cannot be immediately allocated, the OS preempts (forcibly takes away) all resources from the holding process. The process restarts when it can get all needed resources.

Problem: Only works for resources whose state can be easily saved and restored (CPU registers, memory). Does NOT work for printers, tape drives (mid-job interruption is destructive).

4. Deny Circular Wait (Most Practical!)

Impose a total ordering on all resource types. Assign a unique integer to each resource type (e.g., R1=1, R2=2, …). A process can only request resources in strictly increasing order.

Example — Resource Ordering
Resources: Mutex_A=1, Mutex_B=2, File=3, Printer=4

VALID sequence:   lock(Mutex_A) → lock(Mutex_B) → open(File)
INVALID sequence: lock(Mutex_B) → lock(Mutex_A)  ← would cause circular wait
                  (B=2 already held; can't request A=1 < 2)

If ALL processes follow this rule:
→ Circular wait is IMPOSSIBLE by construction
→ No deadlock can ever form
💡
Why ordering works (formal proof by contradiction): Assume circular wait exists: P0 holds Ra and waits for Rb (held by P1), P1 holds Rb and waits for Rc (held by P2), …, Pn holds Rz and waits for Ra (held by P0). By the ordering rule: a < b < c < … < z < a. But a < a is a contradiction! ∴ Circular wait cannot form under total resource ordering.

Comparison of Prevention Strategies

Release before RequestHold & WaitStarvation, inefficiency⚠️ Sometimes

⚡ GATE
  • Resource ordering (deny circular wait) is the most commonly used prevention technique in practice — used in database lock ordering, Linux kernel lock hierarchies.
  • All-or-Nothing leads to low utilization and starvation — GATE often asks about these side effects.
  • Preemption-based prevention only works for preemptable resources (CPU, memory). Printers, tape drives are non-preemptable.
  • Prevention is conservative — it may unnecessarily deny resources even when no deadlock would occur.
Active Recall: Deadlock Prevention
1. Which Coffman condition does resource ordering (linear ordering) break?
Circular Wait. If all processes must acquire resources in a fixed global order (R1 before R2 before R3...), a circular chain is impossible — proved by contradiction (a < b < ... < a is impossible).

2. What is the side effect of "All-or-Nothing" (request all resources upfront)?
Low resource utilization and possible starvation — a process waits for all resources even when it only needs some currently. Resources held but not yet used are wasted.

3. Why can't you prevent deadlock by denying Mutual Exclusion for all resources?
Some resources are inherently non-sharable by nature (e.g., printers, write access to a file). Mutual exclusion cannot be denied for these — it's a physical constraint, not a design choice.

2C.5 Deadlock Avoidance & Safe State

🎓 University

Unlike prevention (which statically removes a condition), avoidance is dynamic — the OS makes resource allocation decisions at runtime using additional information: the maximum future needs of each process.

Safe State
A state is safe if there exists a safe sequence — an ordering of all processes ⟨P1, P2, …, Pn⟩ such that for each Pi, the resources it still needs can be satisfied by the currently available resources plus the resources held by all Pj where j < i (processes that execute before Pi).
⚠️ GATE Trap — Safe ≠ Not Deadlocked immediately: Unsafe means no guaranteed safe sequence exists. Processes may still complete without deadlock if they don't request their maximum. Every deadlocked state is unsafe, but not every unsafe state is deadlocked. The hierarchy is: Safe ⊂ Unsafe ⊂ All states; Deadlocked ⊂ Unsafe.

Safe vs Unsafe vs Deadlocked

StateDefinitionDeadlock?
SafeA safe sequence exists — all processes can complete❌ No
UnsafeNo safe sequence exists — deadlock may occur⚠️ Possible
DeadlockedProcesses are stuck — none can proceed✅ Yes
⚠️
Critical Insight: An unsafe state does NOT guarantee deadlock — it means deadlock is possible depending on future requests. Avoidance algorithms keep the system in a safe state at all times by refusing grants that would lead to an unsafe state.

Safe Sequence — Worked Example

System has 12 tape drives. 3 processes:

ProcessMax NeedCurrently HoldingStill Needs
P01055
P1422
P2927

Currently available: 12 − (5+2+2) = 3 drives free.

Is this a safe state?

  • P1 needs 2. Available=3 ≥ 2. ✅ P1 runs to completion, releases 2 → Available = 3+2 = 5
  • P0 needs 5. Available=5 ≥ 5. ✅ P0 runs, releases 5 → Available = 5+5 = 10
  • P2 needs 7. Available=10 ≥ 7. ✅ P2 runs.

Safe sequence: ⟨P1, P0, P2⟩ → State is SAFE. No deadlock possible.

The Avoidance Decision Rule

When a process makes a resource request: grant it only if the resulting state is still safe. If the new state would be unsafe, deny the request and make the process wait (even if resources are physically available).

⚡ GATE
  • Safe state → No deadlock. Deadlock → Unsafe. But Unsafe ↛ Deadlock necessarily.
  • Avoidance requires processes to declare their maximum resource needs in advance — this is its main limitation in practice.
  • A state being unsafe means the system might deadlock — not that it will. An unsafe state reached by a bad sequence could still end safely if processes don't actually use their maximum declared resources.
  • GATE shortcut for safe sequence: at each step, find any process whose "still needs" ≤ available. Run it, release its resources, repeat.

2C.6 Banker's Algorithm — Single Resource Type

🎓 University
Banker's Algorithm (Dijkstra, 1965)
Named after a bank that grants loans only if it can still satisfy all customers' maximum possible demands. The OS grants a resource request only if the resulting state has a safe sequence. Designed for avoidance with multiple instances of resources.

Single Resource Type — Algorithm

Given: n processes, 1 resource type with m total instances.

Data structures: Max[i], Allocated[i], Need[i] = Max[i] − Allocated[i], Available.

Step 1: Check if Request is Valid

Pseudocode — Request Handling
When Process Pi requests Request_i units:

1. If Request_i > Need_i → ERROR: exceeded maximum claim
2. If Request_i > Available → WAIT: resources not available yet
3. Tentatively grant: simulate allocation
      Available -= Request_i
      Allocated[i] += Request_i
      Need[i] -= Request_i
4. Run Safety Algorithm on new state
5. If SAFE → grant permanently. If UNSAFE → rollback, Pi waits.

Step 2: Safety Algorithm

Pseudocode — Safety Check
Work = Available
Finish[i] = false for all i

Loop:
    Find i such that: Finish[i] == false AND Need[i] <= Work
    If found:
        Work += Allocated[i]  // simulate Pi finishing
        Finish[i] = true
    Else:
        Break

If all Finish[i] == true → SAFE STATE
Else → UNSAFE STATE

Single Resource Worked Example

Total = 10 instances. State:

ProcessMaxAllocatedNeed
P0725
P1330
P2927
P3220
P4404

Available = 10 − (2+3+2+2+0) = 1

Safety check (Work=1):

  1. P1: Need=0 ≤ Work=1. ✅ Work = 1+3 = 4
  2. P3: Need=0 ≤ Work=4. ✅ Work = 4+2 = 6
  3. P4: Need=4 ≤ Work=6. ✅ Work = 6+0 = 6
  4. P0: Need=5 ≤ Work=6. ✅ Work = 6+2 = 8
  5. P2: Need=7 ≤ Work=8. ✅ Done!

Safe sequence: ⟨P1, P3, P4, P0, P2⟩ ← SAFE!

Can P4 request 3 more? Need[P4]=4, Request=3 ≤ Need ✅. Available=1, Request=3 > Available=1 → WAIT (not enough resources right now).

⚡ GATE
  • Banker's algorithm time complexity: O(n² × m) where n = processes, m = resource types.
  • A process with Need=0 can always run immediately (it needs nothing more). Always pick these first in the safety sequence.
  • Multiple safe sequences may exist — GATE may ask if a specific sequence is valid, not for the unique one.
  • Banker's is not used in practice (requires advance maximum declarations, processes must declare needs upfront — impractical for general systems). Used in specialized batch systems.
🏦 Banker's Algorithm — Mental Model (Dijkstra's Original Analogy):
Think of the OS as a bank with a fixed amount of cash (resources). Processes are customers who have declared their maximum loan (Max). The bank will only grant a loan if, after granting it, it can still satisfy every customer's maximum possible future demand by recycling repaid loans. If granting would leave the bank unable to guarantee this → bank refuses (unsafe state). This is exactly Banker's Algorithm.

Variable mapping to the analogy:
Available = Cash in vault  |  Max = Maximum loan each customer declared
Allocation = Loans currently outstanding  |  Need = Max − Allocation (more they might borrow)

Finding ALL Safe Sequences (GATE asks this!)

G-004 Fix: Safe Sequence Enumeration Method
At each step of the safety algorithm, there may be multiple eligible processes (multiple Pi where Need[i] ≤ Work). Each choice leads to a different valid safe sequence. To find ALL safe sequences:

Algorithm (recursive/backtracking):
1. Find all processes where Need[i] ≤ Work and Finish[i]=false
2. For each such process Pi: tentatively "run" Pi, update Work, mark Finish[Pi]=true, recurse
3. If all Finish[i]=true → record this sequence
4. Backtrack and try the next candidate

Example with the §2C.6 state (Available=1, 5 processes):
Step 1: Eligible processes (Need ≤ Work=1): P1 (Need=0), P3 (Need=0)
→ Branch 1: Pick P1 first → Work=4 → P3 runs → Work=6 → P4 → Work=6 → P0 → Work=8 → P2 ✅
  Sequence: ⟨P1, P3, P4, P0, P2⟩
→ Branch 2: Pick P3 first → Work=3 → P1 runs → Work=6 → same as above
  Sequence: ⟨P3, P1, P4, P0, P2⟩
Both are valid safe sequences!
⚠️ GATE Trap: "Which Coffman condition does Banker's Algorithm break?"
Answer: NONE! Banker's is an avoidance algorithm, not a prevention algorithm. It does not break any Coffman condition — all 4 conditions can still hold. Instead, it prevents the system from entering an unsafe state where deadlock might occur.

2C.7 Banker's Algorithm — Multiple Resource Types

🎓 University

Data Structures (n processes, m resource types)

VariableTypeMeaning
Available[m]VectorAvailable[j] = free instances of resource type Rj
Max[n][m]MatrixMax[i][j] = max instances of Rj that Pi may request
Allocation[n][m]MatrixAllocation[i][j] = instances of Rj currently given to Pi
Need[n][m]MatrixNeed[i][j] = Max[i][j] − Allocation[i][j] (remaining need)

Full Worked Example

5 processes (P0–P4), 3 resource types A(10), B(5), C(7):

ProcessAlloc A B CMax A B CNeed A B C
P00 1 07 5 37 4 3
P12 0 03 2 21 2 2
P23 0 29 0 26 0 0
P32 1 12 2 20 1 1
P40 0 24 3 34 3 1

Available = Total − ΣAllocated = (10,5,7) − (7,2,5) = (3, 3, 2)

Safety Algorithm Trace:

Work = (3,3,2). Find processes where Need[i] ≤ Work (component-wise):

  1. P1: Need(1,2,2) ≤ Work(3,3,2) ✅ → Work = (3,3,2)+(2,0,0) = (5,3,2)
  2. P3: Need(0,1,1) ≤ Work(5,3,2) ✅ → Work = (5,3,2)+(2,1,1) = (7,4,3)
  3. P4: Need(4,3,1) ≤ Work(7,4,3) ✅ → Work = (7,4,3)+(0,0,2) = (7,4,5)
  4. P0: Need(7,4,3) ≤ Work(7,4,5) ✅ → Work = (7,4,5)+(0,1,0) = (7,5,5)
  5. P2: Need(6,0,0) ≤ Work(7,5,5) ✅ → DONE!

Safe sequence: ⟨P1, P3, P4, P0, P2⟩ — System is SAFE.

Request Handling Example:

P1 requests (1,0,2):

  1. Request(1,0,2) ≤ Need[P1](1,2,2)? ✅
  2. Request(1,0,2) ≤ Available(3,3,2)? ✅
  3. Simulate: Available=(2,3,0), Alloc[P1]=(3,0,2), Need[P1]=(0,2,0)
  4. Run safety on new state (Work=(2,3,0)):
    • P1: Need(0,2,0) ≤ (2,3,0) ✅ → Work=(2+3,3+0,0+2)=(5,3,2)
    • P3: Need(0,1,1) ≤ (5,3,2) ✅ → Work=(7,4,3)
    • P4: Need(4,3,1) ≤ (7,4,3) ✅ → Work=(7,4,5)
    • P0: Need(7,4,3) ≤ (7,4,5) ✅ → Work=(7,5,5)
    • P2: Need(6,0,0) ≤ (7,5,5) ✅ → DONE!
    Safe sequence: ⟨P1, P3, P4, P0, P2⟩ → GRANT the request.
⚡ GATE
  • Vector comparison: Need[i] ≤ Work means every component of Need[i] ≤ corresponding component of Work.
  • GATE trick: if a row in Need matrix is all zeros, that process can always run — it has everything it needs.
  • After granting tentatively: always re-run the full safety check. If new state fails safety → rollback the allocation, make process wait.
  • Common exam pattern: given a state + a new request, determine if the request can be granted safely.
Active Recall: Banker's Algorithm
1. What is the fundamental difference between Deadlock Prevention and Deadlock Avoidance?
Prevention statically breaks one of the four Coffman conditions (e.g., ordering resources). Avoidance (like Banker's) dynamically assesses resource requests at runtime and only grants them if the resulting state is safe.

2. How is the Need matrix calculated?
Need[i, j] = Max[i, j] - Allocation[i, j].

2C.8 Deadlock Detection — Single-Instance Resources

🎓 University

When prevention and avoidance are not used, the OS must be able to detect when deadlock has occurred and take recovery action.

Wait-For Graph (WFG)
A directed graph derived from the RAG by removing resource nodes. An edge Pi → Pj exists in WFG if Pi is waiting for a resource currently held by Pj. A cycle in the WFG = deadlock (for single-instance resources).

Algorithm

  1. Maintain the Wait-For Graph continuously (add edges on request, remove on release).
  2. Periodically invoke a cycle-detection algorithm on the WFG.
  3. If a cycle is found → deadlock exists. The processes in the cycle are deadlocked.
WFG Example — Deadlock Detected
Processes: P1, P2, P3, P4

Wait-For edges:
  P1 → P2  (P1 waits for resource held by P2)
  P2 → P3  (P2 waits for resource held by P3)
  P3 → P1  (P3 waits for resource held by P1)
  P4 → P1  (P4 waits for P1, but not in cycle)

Cycle detected: P1 → P2 → P3 → P1
→ DEADLOCK: {P1, P2, P3} are deadlocked
→ P4 is NOT deadlocked (it can proceed if P1 is freed)

2C.9 Deadlock Detection — Multiple-Instance Resources

🎓 University

For multi-instance resources, the WFG is not sufficient. We use a detection algorithm similar in structure to Banker's safety algorithm, but without the Max/Need matrices — only current state matters.

Data Structures

VariableMeaning
Available[m]Number of free instances of each resource type
Allocation[n][m]Current allocation matrix
Request[n][m]Current pending requests from each process
⚠️ Detection vs Avoidance Difference: Detection algorithm uses Request[i] (current pending requests) and starts Finish[i]=true if Alloc[i]==0. Avoidance (Banker's Safety) uses Need[i] (maximum future need) and starts Finish[i]=false for all. These are different algorithms!

Detection Algorithm

Pseudocode — Multi-Instance Detection
Work = Available.copy()
Finish[i] = (Allocation[i] == 0) ? true : false  // processes with no resources can finish

Loop:
    Find i such that: Finish[i] == false AND Request[i] <= Work
    If found:
        Work += Allocation[i]  // assume Pi will finish and release
        Finish[i] = true
    Else:
        Break

If any Finish[i] == false → DEADLOCK
Processes with Finish[i] == false are DEADLOCKED

Worked Example

5 processes, 3 resource types A(7), B(2), C(6). Available = (0,0,0).

ProcessAlloc A B CRequest A B C
P00 1 00 0 0
P12 0 02 0 2
P23 0 30 0 0

  2. P2: Request(0,0,0) ≤ Work(0,1,0) ✅ → Finish[P2]=true, Work=(3,1,3)
  3. P3: Request(1,0,0) ≤ Work(3,1,3) ✅ → Finish[P3]=true, Work=(5,2,4)
  4. P1: Request(2,0,2) ≤ Work(5,2,4) ✅ → Finish[P1]=true, Work=(7,2,4)
  5. P4: Request(0,0,2) ≤ Work(7,2,4) ✅ → Finish[P4]=true

Detection vs Avoidance: The detection algorithm is similar to Banker's safety algorithm but uses Request (current requests) instead of Need (maximum future needs). Detection is optimistic — it assumes any unblocked process will complete and release resources.
⚡ GATE
  • Key difference from Banker's: uses Request (actual current requests) not Need (maximum future claims).
  • Finish[i] starts as true if Allocation[i] is all zeros (process holds nothing — can't be in a deadlock contributing cycle).
  • Processes with Finish[i]=false at the end are the deadlocked set.
  • Time complexity: O(m × n²) — same as Banker's.

2C.10 Deadlock Recovery

🎓 University

Once a deadlock is detected, the OS must break it. Two main recovery strategies:

Strategy 1: Process Termination

Kill one or more deadlocked processes to break the cycle and free resources.

MethodHowProsCons
Abort AllKill all deadlocked processes at onceGuaranteed to break deadlock immediatelyVery costly — all work is lost
Abort One-at-a-TimeKill processes one by one; re-run detection after eachMinimizes killed processesHigh detection overhead per step

Victim Selection Criteria

When choosing which process to terminate (the "victim"), minimize cost. Factors considered:

  • Process priority: Kill lower-priority processes first
  • CPU time used: Kill processes that have used the least CPU (lose less work)
  • Resources held: Kill processes holding the most resources (break more locks)
  • Resources needed: Kill processes needing most to complete (least likely to finish soon anyway)
  • Number of processes to kill: Minimize the number of victims
  • Interactive vs batch: Prefer killing batch processes over interactive ones
Victim Selection Priority Order (GATE):
1. Lowest priority process first
2. Process with least CPU time used (least work lost)
3. Process holding most resources (breaks more locks)
4. Process furthest from completion
⚠️ Track rollback count to prevent starvation of victims — same process shouldn't always be chosen.

Strategy 2: Resource Preemption

Forcibly take resources away from some processes and give them to others, until the deadlock cycle is broken.

Three issues to handle:

  1. Victim Selection: Which process/resource to preempt? (same cost-minimization criteria)
  2. Rollback: The preempted process loses its resources and must be rolled back to a safe state (checkpoint). It will need to re-acquire resources later.
  3. Starvation: The same process might always be chosen as victim. Fix: include number of times a process has been rolled back in the cost function.

Recovery Comparison

ApproachSpeedWork LostStarvation Risk
Abort AllFastAll deadlocked processesNone
Abort One-by-OneSlow (repeated detection)MinimalLow
Resource Preemption + RollbackMediumPartial (to checkpoint)Possible (mitigated by rollback count)
📌
Real-World: Databases use transaction rollback for deadlock recovery — the DBMS detects a deadlock via WFG on transactions, picks a victim transaction (usually the youngest/cheapest), rolls it back completely, and restarts it. MySQL, PostgreSQL, Oracle all use this approach.
⚡ GATE
  • Recovery methods: (1) Process termination removes the terminated process entirely from the wait-for graph, breaking cycles; (2) Resource preemption temporarily takes resources from a process, freeing them for others — can break circular-wait cycles. Both require careful victim selection to avoid starvation.
  • Starvation in recovery: if same process is always victim → it never completes. Solution: include "number of times rolled back" in victim cost function.
  • Preemption only works for resources whose state can be saved/restored (CPU, memory pages). Non-preemptable: printers, tape drives.

RAG Claim Edges (Deadlock Avoidance)

Claim Edges in Resource Allocation Graph
For single-instance resource avoidance using RAGs:
  Request edge:  Pi → Rj  (Pi is requesting Rj)
  Assignment edge: Rj → Pi  (Rj is assigned to Pi)
  Claim edge:    Pi ⟶ Rj  (dashed — Pi MAY request Rj in future)

Algorithm:
  When Pi requests Rj:
    Convert claim edge Pi⟶Rj to request edge Pi→Rj
    Check if resulting graph has a CYCLE
    If cycle exists → UNSAFE → deny request
    If no cycle → SAFE → grant and convert to assignment edge

  This is the single-instance version of Banker's Algorithm.
  Cycle detection: O(n²) via DFS.
  Limitation: processes must declare all future resource needs upfront.

Wait-Die / Wound-Wait (Database Deadlock Prevention)

Timestamp-Based Deadlock Prevention
Each transaction gets a timestamp (older = smaller timestamp).
When Ti requests a lock held by Tj:

Wait-Die (non-preemptive):
  If Ti is older (Ti < Tj): Ti WAITS for Tj to release.
  If Ti is younger (Ti > Tj): Ti DIES (rolled back, restarted
    with SAME timestamp to ensure eventual progress).
  → Only older transactions wait for younger ones.
  → No circular wait possible.

Wound-Wait (preemptive):
  If Ti is older (Ti < Tj): Ti WOUNDS Tj (Tj is rolled back).
  If Ti is younger (Ti > Tj): Ti WAITS for Tj.
  → Older transactions preempt younger ones.
  → No circular wait possible.

Comparison:
  Wait-Die:   older waits, younger dies → fewer rollbacks for old txns
  Wound-Wait: older wounds, younger waits → faster for old txns
  Both prevent deadlock without detection overhead.
  Both avoid starvation: restarted txn keeps original timestamp.
  Used by: Oracle, MySQL InnoDB (configurable).

🎓 University Exams

Short Answer Questions (2–5 Marks)

Q1: State the four necessary conditions for deadlock.
The four Coffman conditions: (1) Mutual Exclusion — at least one resource is non-sharable. (2) Hold and Wait — a process holds one resource while waiting for more. (3) No Preemption — resources cannot be forcibly taken. (4) Circular Wait — a closed chain of processes each waiting for the next's resource. All four must hold simultaneously for deadlock to occur.
Q2: Differentiate between Deadlock Prevention and Deadlock Avoidance.
Prevention is static — it structurally ensures at least one Coffman condition can never hold (e.g., resource ordering to prevent circular wait). No runtime overhead but may cause low utilization. Avoidance is dynamic — at runtime, each resource request is evaluated and granted only if the resulting state is safe (has a safe sequence). Requires processes to declare maximum needs in advance. Banker's Algorithm is the key avoidance algorithm.
Q3: What is a safe state? Give an example.
A state is safe if there exists a safe sequence ⟨P1, P2, …, Pn⟩ such that each Pi's remaining resource needs can be satisfied by currently available resources plus resources released by all Pj (j < i). Example: 12 tape drives, P0 needs 5 more (holds 5), P1 needs 2 (holds 2), P2 needs 7 (holds 2). Available=3. Safe sequence: ⟨P1, P0, P2⟩ → safe state, no deadlock possible.
Q4: What is a Wait-For Graph? How is it used for deadlock detection?
A Wait-For Graph (WFG) is a simplified form of RAG used when all resources have single instances. Resource nodes are removed — only process nodes remain. An edge Pi → Pj means Pi is waiting for a resource currently held by Pj. A cycle in the WFG = deadlock. The OS periodically runs DFS (O(n²)) on the WFG to detect cycles. Processes forming the cycle are the deadlocked set. Example: P1→P2→P3→P1 = deadlock among {P1, P2, P3}.
Q5: Explain the deadlock detection algorithm for multiple-instance resources.
Similar to Banker's safety algorithm but uses Request (current pending requests) instead of Need (maximum future claims). Data: Available[m], Allocation[n][m], Request[n][m]. Algorithm: Work = Available. Finish[i] = true if Allocation[i] is all zeros. Loop: find i where Finish[i]=false AND Request[i] ≤ Work → Work += Allocation[i], Finish[i] = true. Repeat. If any Finish[i]=false at end → those processes are deadlocked. Time complexity: O(m × n²).
Q6: Compare process termination and resource preemption as deadlock recovery methods.
Process Termination: Kill one or all deadlocked processes. Abort-all is fast but loses all work. Abort-one-by-one re-runs detection after each kill. Victim selection uses priority, CPU time, resources held. Resource Preemption: Forcibly take resources from a process, roll it back to a checkpoint. Risk: starvation if same process always chosen — mitigated by including rollback count in cost function. Preemption only works for preemptable resources (CPU, memory), not printers.

Long Answer Questions (10 Marks)

Q7: Explain Banker's Algorithm for multiple resource types with a numerical example.
Banker's uses four data structures: Available[m], Max[n][m], Allocation[n][m], Need[n][m]. Need = Max − Allocation. On a request from Pi: (1) check Request ≤ Need, (2) check Request ≤ Available, (3) tentatively grant, (4) run safety algorithm. Safety: simulate processes completing using Work vector, find any process whose Need ≤ Work, run it, add its allocation to Work, mark finished. If all processes can finish → SAFE. [Students should demonstrate with the 5-process, 3-resource worked example from section 2C.7].
Q8: Compare all four deadlock handling strategies with their trade-offs.
Prevention: Static — deny at least one Coffman condition. Resource ordering is most practical. Drawback: low utilization. Avoidance: Dynamic — Banker's checks safe state before granting. Requires max declarations — impractical for general OS. Detection & Recovery: Allow deadlocks, detect via WFG or detection algorithm, recover via termination or preemption. Used in databases. Ignore (Ostrich): Assume deadlocks are rare. Used by Linux, Windows. No overhead but freezes require reboot.

Q (DU 2023): Which of the following correctly distinguishes livelock from deadlock?
(a) Livelock uses no CPU; deadlock wastes CPU cycles   (b) Deadlock processes are blocked; livelock processes keep changing state but make no progress   (c) Both are caused by circular wait   (d) Livelock requires all 4 Coffman conditions
Answer(b) — In deadlock, processes are permanently blocked (no state change, no CPU). In livelock, processes actively respond to each other (state changes, CPU consumed) but make zero forward progress. Livelock is caused by over-polite synchronisation, not necessarily circular wait.
⚡ GATE Key: Livelock is NOT the same as busy-waiting (spinlock). In a spinlock, one thread is actively waiting and another IS making progress. In livelock, ALL involved threads are changing state but NONE make progress. Livelock is always mutual; spinlock is one-sided.
🔬 Real-World Livelock — Ethernet CSMA/CD: Original Ethernet used CSMA/CD — if two stations transmitted simultaneously (collision), both would back off and retry. Without randomised backoff (exponential backoff), both stations would retry at exactly the same time, causing infinite collisions — a livelock. IEEE 802.3 mandates truncated binary exponential backoff: after kth collision, retry delay is random in [0, 2^min(k,10) - 1] slot times. This probabilistically eliminates livelock.

2C.12 Real-World Deadlock Patterns & Code

Deadlocks appear in production systems daily — databases, OS kernels, and networking code are all vulnerable. Recognising and preventing these patterns is what separates competent systems programmers from experienced ones.

Pattern 1: Lock Ordering (Most Common Fix)

// DEADLOCK: Thread A acquires lock1 then lock2; Thread B does the reverse
pthread_mutex_t lock1 = PTHREAD_MUTEX_INITIALIZER;
pthread_mutex_t lock2 = PTHREAD_MUTEX_INITIALIZER;

void* thread_A(void* arg) {
    pthread_mutex_lock(&lock1);    // A holds lock1
    sleep(1);                       // Gives B time to grab lock2
    pthread_mutex_lock(&lock2);    // A waits for lock2 → DEADLOCK
    // critical section
    pthread_mutex_unlock(&lock2);
    pthread_mutex_unlock(&lock1);
    return NULL;
}

void* thread_B(void* arg) {
    pthread_mutex_lock(&lock2);    // B holds lock2
    sleep(1);
    pthread_mutex_lock(&lock1);    // B waits for lock1 → DEADLOCK
    pthread_mutex_unlock(&lock1);
    pthread_mutex_unlock(&lock2);
    return NULL;
}

// FIX: Always acquire locks in the SAME GLOBAL ORDER
// Both threads acquire lock1 THEN lock2 — circular wait impossible
void* thread_A_fixed(void* arg) {
    pthread_mutex_lock(&lock1);    // Always lock1 first
    pthread_mutex_lock(&lock2);    // Then lock2
    // critical section
    pthread_mutex_unlock(&lock2);  // Release in reverse order
    pthread_mutex_unlock(&lock1);
    return NULL;
}

Pattern 2: trylock with Backoff

// Use trylock to avoid blocking indefinitely
int acquire_both(pthread_mutex_t* a, pthread_mutex_t* b) {
    while(1) {
        pthread_mutex_lock(a);             // Acquire first lock
        if(pthread_mutex_trylock(b) == 0)  // TRY second — non-blocking
            return 0;                       // Success: hold both
        pthread_mutex_unlock(a);           // Failed: release first
        usleep(rand() % 1000);            // Random backoff (prevent livelock)
    }
}

// This breaks "Hold and Wait" — we either hold both or none
// Never in the state of holding one while waiting for another

Pattern 3: Database Deadlock (Transactions)

-- SQL Deadlock: Classic two-transaction scenario
-- Transaction T1:                 -- Transaction T2:
BEGIN;                              BEGIN;
UPDATE accounts                     UPDATE accounts
  SET balance = balance - 100       SET balance = balance + 100
  WHERE id = 1;    -- locks row 1   WHERE id = 2;    -- locks row 2

UPDATE accounts                     UPDATE accounts  
  SET balance = balance + 100       SET balance = balance - 100
  WHERE id = 2;    -- waits!        WHERE id = 1;    -- waits!
-- T1 waits for T2's lock on row 2
-- T2 waits for T1's lock on row 1
-- DEADLOCK! Database detects and rolls back one transaction

-- FIX: Always access rows in same order (by primary key)
-- T1 and T2 both: lock row 1 first, then row 2

🔬 Linux Kernel Deadlock Prevention: The Linux kernel uses lock dependency tracking (lockdep) in debug builds. Every time a lock is acquired, lockdep records the acquisition order. Q: 4 processes each need maximum 3 units of a single resource type. What is the minimum number of resource units to guarantee no deadlock?

GATE 2014: A system has 4 processes and 5 identical resources. Each process can request at most 2 resources. The minimum number of resources that ensures no deadlock is:
(a) 4   (b) 5   (c) 6   (d) 7
Answer(b) 5 — Using formula: R ≥ n(k−1)+1 = 4(2−1)+1 = 4+1 = 5. With 5 resources: worst case — each process holds 1 (total 4 held), 1 free. That 1 free resource lets at least one process complete (needs at most 2, has 1, 1 more available). It finishes, releases 2, chain continues.
GATE 2019: Consider a system with 3 resource types and 4 processes. Which is TRUE about a system in unsafe state?
(a) It is already deadlocked   (b) Deadlock will definitely occur   (c) Deadlock may or may not occur   (d) The system has a cycle in the RAG
Answer(c) Deadlock may or may not occur — Unsafe means there is NO guaranteed safe sequence, but processes may still complete without deadlock if they don't all request their maximum simultaneously. Safe state guarantees no deadlock; unsafe state only means the safety guarantee is lost.
🔬 Distributed Deadlock Detection: In distributed systems, each node maintains a local Wait-For Graph. A coordinator periodically collects all local WFGs and constructs a global WFG. A cycle in the global WFG indicates deadlock. Challenge: the global WFG snapshot may be inconsistent (different nodes queried at different times). Chandy-Misra-Haas algorithm handles this by propagating probe messages along wait-for edges — if a probe returns to its origin, a deadlock cycle is confirmed. Time complexity: O(e) where e = edges in WFG.

3.1 Concurrency & Race Conditions

🎓 University
Concurrency
Concurrency is the execution of multiple processes or threads in overlapping time periods. On a single-core CPU, concurrency is achieved via rapid context switching (interleaving). On multi-core CPUs, true parallelism is possible — multiple threads literally execute simultaneously.

Why Concurrency Creates Problems

When multiple threads share data (global variables, heap memory), they can interfere with each other in unpredictable ways. The result depends on the exact order of execution — called non-determinism.

Race Condition — The Core Problem

Race Condition
A race condition occurs when two or more threads access shared data concurrently and the final result depends on the timing / order of execution. The outcome is non-deterministic — it "races" to see which thread wins.

Classic Example: counter++

Suppose two threads both execute counter++ simultaneously. This single C statement compiles into 3 machine instructions:

Thread T1
LOAD R1, counter   ; R1 = 5
ADD  R1, 1         ; R1 = 6
STORE counter, R1  ; counter = 6
Thread T2
LOAD R1, counter   ; R1 = 5 ← STALE!
ADD  R1, 1         ; R1 = 6
STORE counter, R1  ; counter = 6 ← LOST!

If T1 reads counter=5, then T2 reads counter=5 (before T1 stores), both compute 6 and store 6. One increment is lost! Counter should be 7 but is 6.

⚠️
Key Insight: The problem is that counter++ is not atomic. An atomic operation is indivisible — it executes completely without interruption. Non-atomic operations on shared data are vulnerable to race conditions.

Concurrent vs Parallel vs Sequential

ModeDefinitionExample
SequentialOne task at a time, complete before nextSingle-threaded program
ConcurrentMultiple tasks making progress, may interleaveOS time-sharing on 1 core
ParallelMultiple tasks literally running simultaneouslyMulti-core processor

Interleaving Trace — Why counter++ Fails

Step-by-Step Trace (counter initially = 5)
Step  T1 instruction        T2 instruction        counter  R1(T1)  R1(T2)
────  ───────────────────  ───────────────────  ─────────  ──────  ──────
 1    LOAD R1, counter                           5         5       —
 2                          LOAD R1, counter     5         5       5
 3    ADD R1, 1                                  5         6       5
 4                          ADD R1, 1            5         5       6
 5    STORE counter, R1                          6         6       6
 6                          STORE counter, R1    6         6       6

Expected: counter=7. Actual: counter=6. One increment LOST.

GATE-style: If 3 threads each execute x++ once (x initially 0),
what are the possible final values of x?
Answer: x ∈ {1, 2, 3}. Minimum=1 (all read 0 before any stores).
Maximum=3 (fully serialized).
⚡ GATE
  • A race condition requires: (1) shared data, (2) concurrent access, (3) at least one write operation.
  • Race conditions are non-deterministic — they may not appear every run, making them extremely hard to debug.
  • GATE often asks to identify the critical section in code — it is the section that accesses shared data.

3.2 The Critical Section Problem

🎓 University
Critical Section
A critical section is a segment of code that accesses shared resources (variables, files, devices) and must not be executed by more than one process simultaneously. The critical section problem is to design a protocol that processes can use to cooperate safely.

Structure of a Process with a Critical Section

Pseudocode — Process Structure
while (true) {
    /* ENTRY SECTION */
    acquire_lock();         // Request permission to enter CS

    /* CRITICAL SECTION */
    // Access shared data here — ONLY ONE PROCESS AT A TIME

    /* EXIT SECTION */
    release_lock();         // Signal that CS is free

    /* REMAINDER SECTION */
    // Non-critical work — can run concurrently
}

Three Requirements for a Valid Solution

01
Mutual Exclusion
If process Pi is in its critical section, no other process Pj can be in its critical section at the same time. This is the most fundamental requirement.
02
Progress
If no process is in the CS and some processes want to enter, the selection of which process enters next cannot be postponed indefinitely. Only processes NOT in the Remainder Section participate in this decision.
03
Bounded Waiting
There must exist a bound on the number of times other processes are allowed to enter the CS after a process has made a request to enter — before that request is granted. Prevents starvation.
💡
Memory Trick: Mutual exclusion = Max one in. Progress = Process chosen eventually. Bounded = Bounded waits before entry. MPB.

Naive (Broken) Solutions

Approach 1: Disable Interrupts

Disable all interrupts while in the critical section. This works on single-processor systems but fails on multi-core (disabling interrupts on one core doesn't stop other cores). Also, giving user processes the ability to disable interrupts is dangerous — a malicious process could never re-enable them.

Approach 2: Strict Alternation (Turn Variable)

Use a shared turn variable. Process 0 enters when turn == 0, Process 1 when turn == 1. Satisfies mutual exclusion but violates Progress: Process 0 cannot enter the CS twice in a row even if Process 1 is in the Remainder Section.

Summary of Broken Approaches

ApproachME?Progress?BW?Problem
Lock variable only❌✅✅Both can read lock=0 simultaneously → both enter CS
Strict alternation (turn)✅❌✅Process blocked if other is in remainder section
Flags only (no turn)✅❌❌Both set flag=true → both wait forever → deadlock
Disable interrupts✅✅✅Single-core only; dangerous for user processes
Peterson's✅✅✅✅ Correct for 2 processes (needs memory fences on modern HW)
⚡ GATE
  • Mutual exclusion alone is not sufficient — a solution that always blocks every process technically satisfies ME but violates Progress.
  • The strict alternation solution satisfies ME and Bounded Waiting but FAILS Progress — this is a classic GATE trick question.
  • A solution that disables interrupts satisfies all 3 on single-core only.

3.3 Peterson's Solution

🎓 University
Peterson's Solution
A classic software-only solution to the critical section problem for two processes. Uses two shared variables: flag[2] (indicates desire to enter) and turn (indicates whose turn it is when both want in). Proposed by Gary Peterson in 1981.

Shared Variables

  • int turn — Whose turn it is to enter the CS (0 or 1)
  • boolean flag[2] — flag[i] = true means Process i is ready to enter the CS

The Algorithm

C — Peterson's Solution for Process Pi (i=0 or 1)
/* j = 1 - i  (the OTHER process) */

/* ENTRY SECTION */
flag[i] = true;     // "I want to enter"
turn = j;           // "But I give priority to you first"
while (flag[j] == true && turn == j)
    ;               // Busy-wait (spin) until safe

/* CRITICAL SECTION */
// ... access shared data ...

/* EXIT SECTION */
flag[i] = false;    // "I'm done, others may enter"

/* REMAINDER SECTION */
// ... non-critical code ...

Why It Works — Proving All 3 Requirements

RequirementHow Peterson's Satisfies It
Mutual ExclusionBoth Pi and Pj can be in the while loop simultaneously only if turn == i AND turn == j — impossible since turn is a single integer. So at most one proceeds.
ProgressIf Pj is not ready (flag[j] == false), Pi exits the while loop immediately. No indefinite postponement.
Bounded WaitingAfter Pi sets turn = j and Pj finishes, Pj sets flag[j] = false, freeing Pi. Pi will enter on its very next attempt — wait bound = 1.
⚠️
Modern Hardware Caveat: Peterson's solution is correct in theory but may fail on modern CPUs that reorder memory operations (out-of-order execution). It requires memory barriers / fences to work correctly on real hardware. This is why hardware-level solutions are needed in practice.
⚡ GATE
  • Peterson's solution is a software solution — no special hardware instructions needed (in theory).
  • Works for exactly 2 processes. Generalizing to N processes requires the Bakery Algorithm (Lamport's Algorithm).
  • It uses busy-waiting (spinlock) — the process actively loops checking the condition. This wastes CPU time.
  • GATE tip: "Does Peterson's solution satisfy bounded waiting?" → Yes, bound = 1 (at most one other entry before you).
🔬 Advanced

Lamport's Bakery Algorithm (N processes)

Like a bakery — each process takes a numbered ticket. The process with the lowest number enters first. Two shared arrays: choosing[n] and number[n]. A process sets choosing[i] = true, picks number[i] = max(number[0..n-1]) + 1, then waits until all processes with lower numbers have finished. Satisfies all 3 requirements for N processes.

Memory Ordering, Barriers, and Compiler Reordering

Why Peterson's/Bakery Can Fail on Modern CPUs
Problem: Modern CPUs and compilers REORDER memory operations.

Compiler Reordering:
  Compiler may reorder loads/stores for optimization.
  flag = 1;          // may be moved AFTER critical section!
  critical_section();
  Fix: use volatile, atomic, or compiler barriers:
    asm volatile("" ::: "memory");  // GCC compiler fence
    atomic_store(&flag, 1);         // C11 atomic

CPU Memory Reordering:
  Out-of-order execution + store buffers can reorder operations.
  x86: Total Store Order (TSO) — stores are ordered, but
       store-load reordering can still occur.
  ARM/RISC-V: weaker memory model — almost anything can reorder.

Memory Barriers (Fences):
  mfence  (x86)     — full barrier: no load/store crosses
  lfence  (x86)     — load fence: loads before complete first
  sfence  (x86)     — store fence: stores before commit first
  dmb/dsb (ARM)     — data memory/synchronization barriers
  __sync_synchronize() (GCC) — full memory barrier

Rule for GATE: Peterson's and Bakery assume Sequential Consistency.
  On real hardware, you need memory barriers or atomic operations.
  This is why OS kernels use spinlock implementations with barriers.

3.4 Synchronization Hardware (TSL & CAS)

🎓 University

Modern CPUs provide special atomic hardware instructions that read and modify memory in a single, uninterruptible operation. These are the foundation of all practical synchronization primitives.

1. Test-and-Set (TSL)

Atomically reads a variable and sets it to true in one operation. No other process can interrupt between the read and the set.

C — TestAndSet (hardware-equivalent logic)
/* This entire function executes ATOMICALLY */
boolean TestAndSet(boolean *target) {
    boolean rv = *target;   // 1. Read current value
    *target = true;         // 2. Set to true
    return rv;              // 3. Return OLD value
}

/* Usage: Mutual exclusion with a shared lock variable */
boolean lock = false;  // shared, initially unlocked

/* Process Pi entry: */
while (TestAndSet(&lock))  // spin until lock was false (we got it)
    ;
/* CRITICAL SECTION */
lock = false;  // release

Analysis: If lock == false (unlocked), TestAndSet returns false (exits while loop) and sets lock to true. If lock == true (locked), TestAndSet returns true (keeps spinning). Satisfies Mutual Exclusion and Progress but may not satisfy Bounded Waiting (a process can be stuck if others keep taking turns).

2. Compare-and-Swap (CAS)

The modern standard. Atomically compares a memory location to an expected value, and if they match, swaps it with a new value.

C — Compare-and-Swap
/* Atomic operation */
int CompareAndSwap(int *value, int expected, int new_value) {
    int temp = *value;
    if (*value == expected)
        *value = new_value;
    return temp;             // return OLD value
}

/* Usage: lock = 0 means free, lock = 1 means held */
while (CompareAndSwap(&lock, 0, 1) != 0)
    ;   // spin until we swap 0→1 (meaning we got the lock)
/* CRITICAL SECTION */
lock = 0;  // release
FeatureTest-and-SetCompare-and-Swap
Works forBoolean locksGeneral integers
Modern usageLegacy (x86 XCHG)Standard (x86 CMPXCHG, ARM LDREX/STREX)
ABA ProblemNot applicablePossible; solved with version tags
⚡ GATE
  • TestAndSet and CAS both satisfy Mutual Exclusion and Progress.
  • Simple TSL-based locks do NOT guarantee Bounded Waiting alone. To add bounded waiting, pair with a waiting array (see OS textbooks).
  • Hardware atomic instructions are the basis for mutex locks and semaphores in OS kernels.
  • ABA Problem (CAS): Value changes A→B→A; CAS sees A and assumes nothing changed, but it did. Solution: use a version counter alongside the value.

3.5 Mutex Locks & Spinlocks

🎓 University
Mutex Lock (Mutual Exclusion Lock)
A mutex is a synchronization primitive that provides mutual exclusion. Only the process/thread that acquired the lock can release it. Provides two operations: acquire() and release(). Built on top of hardware atomic instructions.
C — Mutex Usage (POSIX)
#include <pthread.h>

pthread_mutex_t lock = PTHREAD_MUTEX_INITIALIZER;
int shared_counter = 0;

void *increment(void *arg) {
    for (int i = 0; i < 1000000; i++) {
        pthread_mutex_lock(&lock);     // ACQUIRE
        shared_counter++;              // CRITICAL SECTION (safe!)
        pthread_mutex_unlock(&lock);   // RELEASE
    }
    return NULL;
}
/* Without the mutex, shared_counter would be < 2,000,000 due to races */

Spinlock vs Blocking Mutex

PropertySpinlockBlocking Mutex (Sleeping)
Waiting strategyBusy-waits (spins in loop)Blocks process — puts it to sleep
CPU usage while waiting100% (wastes CPU)0% (process is descheduled)
Wake-up latencyImmediate (no context switch)Higher (context switch needed)
Best forShort CS, multi-core, OS kernelsLong CS, user-space applications
ExampleLinux kernel spinlock, TSL-based lockPOSIX pthread_mutex_t
💡
Golden Rule: Use a spinlock when you expect the lock to be held for a very short time (microseconds) and you're on multi-core. Use a blocking mutex when the lock might be held for a longer time — spinning would waste too much CPU.

Deadlock with Mutexes

Mutexes can cause deadlock if two threads acquire locks in opposite order:

C — Classic Deadlock Scenario
/* Thread 1 */                    /* Thread 2 */
lock(mutex_A);                    lock(mutex_B);
lock(mutex_B); // WAITS           lock(mutex_A); // WAITS
// → DEADLOCK! Both threads waiting for each other forever

Fix: Always acquire locks in the same order across all threads. Or use trylock() with rollback.

⚡ GATE
  • A mutex is owned — only the thread that locks it can unlock it. A semaphore (next section) has no ownership.
  • A spinlock always satisfies Mutual Exclusion (via hardware atomics) but wastes CPU. A blocking mutex is more CPU-efficient for long waits.
  • Spinlocks are inappropriate on single-core — if one thread is spinning, it prevents the lock holder (on the same core) from running and releasing the lock → infinite spin!

Priority Inversion — GATE Classic

⚠️
Priority Inversion: High-priority task H blocks waiting for a mutex held by low-priority task L. Medium-priority task M preempts L. Now M runs before H — priority is effectively inverted.
Fix — Priority Inheritance: When H blocks on L's mutex, L temporarily inherits H's priority. M cannot preempt L. L finishes, releases mutex, H runs.
Famous example: Mars Pathfinder (1997) experienced repeated system resets until NASA remotely enabled priority inheritance on the shared bus mutex.

3.6 Semaphores — Binary & Counting

🎓 University
Semaphore (E.W. Dijkstra, 1965)
A semaphore is a synchronization tool that is an integer variable accessed only through two atomic operations: wait() (originally called P — from Dutch proberen, "to test") and signal() (originally called V — from verhogen, "to increment").

Operations

Pseudocode — Semaphore P & V Operations
/* WAIT (P) — Decrement; if negative, block */
wait(S) {
    S.value--;
    if (S.value < 0) {
        add this process to S.list;
        block();   // put process to sleep
    }
}

/* SIGNAL (V) — Increment; if processes sleeping, wake one */
signal(S) {
    S.value++;
    if (S.value <= 0) {
        remove a process P from S.list;
        wakeup(P); // resume the sleeping process
    }
}
📌
Key Insight: When S.value < 0, the absolute value |S.value| tells you exactly how many processes are currently blocked waiting on this semaphore.

Types of Semaphores

TypeInitial ValueRangeUse Case
Binary Semaphore (Mutex Semaphore)1API view: 0 or 1; textbook view may become negative when waiters queueMutual exclusion — similar to a mutex lock (but no ownership)
Counting SemaphoreN (a positive integer)API view: 0..N; textbook view may become negativeControl access to a pool of N identical resources (e.g., 5 printers, 3 DB connections)
⚠️
Semaphore Value Convention — Two Views:
Abstract/API view (POSIX): S.value is always ≥ 0. Blocked waiters are tracked in a separate queue, not reflected in the value.
Textbook/Dijkstra view: S.value can go negative. When S < 0, |S| = number of processes blocked on this semaphore. The implementation shown above (lines 511–527) uses this convention.
For GATE: follow whichever convention the question uses. If it says "value after 3 waits on S=2", the textbook answer is S = −1 (one process blocked). The API answer is S = 0 (one process queued but value stays at 0).

Binary Semaphore Usage (Mutual Exclusion)

Pseudocode — Binary Semaphore for CS
Semaphore mutex = 1;  // Initialize to 1 (unlocked)

Process Pi:
    wait(mutex);      // P — if 1, becomes 0. If 0, block.
    /* CRITICAL SECTION */
    signal(mutex);    // V — becomes 1, wakes one waiter

Semaphore vs Mutex

PropertySemaphoreMutex
OwnershipNone — any thread can signalOwned — only locker can unlock
Value rangeAny integer (can be >1)Binary (locked/unlocked)
Use caseResource counting, orderingMutual exclusion only
SignallingNo ownership — depending on OS/API, a different thread or context may post/signal itMust be unlocked by the owning thread only
⚡ GATE
  • A binary semaphore initialized to 1 can be used for mutual exclusion, but unlike a true mutex, it lacks ownership semantics — any process can call signal() to release it, even if that process didn't call wait() first.
  • A counting semaphore initialized to 0 is used for ordering / signalling — one thread waits, another signals it when an event occurs.
  • Classic GATE question: "What is the maximum number of processes that can be in the CS simultaneously with a semaphore initialized to N?" → N processes.
  • Semaphore operations must be atomic — implemented using TSL/CAS internally by the OS.
  • |S.value| when negative = number of blocked processes. Very common computation question.

3.7 Classical Problem: Producer–Consumer

🎓 University
Bounded-Buffer (Producer–Consumer) Problem
A producer process creates data items and places them in a fixed-size shared buffer. A consumer process removes items from the buffer. The challenge: producer must wait if buffer is full; consumer must wait if buffer is empty; both cannot access the buffer simultaneously.

Three Semaphores Needed

SemaphoreInitial ValueMeaning
mutex1Ensures mutual exclusion on buffer access
emptyN (buffer size)Counts empty slots — producer waits when 0
full0Counts filled slots — consumer waits when 0
Pseudocode — Complete Solution
Semaphore mutex = 1, empty = N, full = 0;
int buffer[N], in = 0, out = 0;  // circular buffer

/* PRODUCER */
while (true) {
    item = produce_item();
    wait(empty);        // wait for an empty slot
    wait(mutex);        // enter critical section
    buffer[in] = item;
    in = (in + 1) % N;
    signal(mutex);      // exit critical section
    signal(full);       // notify consumer: one more full slot
}

/* CONSUMER */
while (true) {
    wait(full);         // wait for a filled slot
    wait(mutex);        // enter critical section
    item = buffer[out];
    out = (out + 1) % N;
    signal(mutex);      // exit critical section
    signal(empty);      // notify producer: one more empty slot
    consume_item(item);
}
⚠️
Critical Ordering: Always wait(empty/full) BEFORE wait(mutex). If you swap these, a deadlock can occur. Example: Producer calls wait(mutex) first when buffer is full, then tries wait(empty) — but Consumer is blocked waiting for wait(mutex) which the Producer holds. Deadlock!

Real-World Examples

  • Printer spooler: Applications (producers) submit print jobs; printer daemon (consumer) processes them.
  • Web server: Network I/O fills a request queue (producer); worker threads process requests (consumers).
  • Unix pipes: ls | grep .c — ls is the producer, grep is the consumer.
⚡ GATE
  • The order of wait() calls matters critically: resource semaphore before mutex always.
  • With N buffer slots, at most N producers can have produced without consuming — tracked by the full semaphore.
  • GATE often gives code with swapped semaphore order and asks: "Does this cause deadlock?" → Yes if mutex is acquired before empty/full.

3.8 Classical Problem: Readers–Writers

🎓 University
Readers–Writers Problem
A shared database is accessed by two types of processes: Readers (only read data) and Writers (read and modify data). Multiple readers can read simultaneously, but a writer needs exclusive access — no readers or other writers while writing.

First Readers–Writers Problem (Readers Priority)

No reader shall be kept waiting unless a writer already has access. Writers may starve.

Pseudocode — First RW Solution
Semaphore rw_mutex = 1;  // Exclusive access for writers
Semaphore mutex   = 1;  // Protect read_count
int read_count    = 0;  // Number of active readers

/* READER */
wait(mutex);
    read_count++;
    if (read_count == 1)  // First reader blocks writer
        wait(rw_mutex);
signal(mutex);

/* --- READ DATA --- */

wait(mutex);
    read_count--;
    if (read_count == 0)  // Last reader releases writer
        signal(rw_mutex);
signal(mutex);

/* WRITER */
wait(rw_mutex);       // Need exclusive access
/* --- WRITE DATA --- */
signal(rw_mutex);

Second Readers–Writers Problem (Writers Priority)

Once a writer is ready, no new readers are allowed in. Readers may starve. More complex — requires additional semaphores to track waiting writers.

Variants Comparison

VariantPriorityStarvation RiskConcurrency
First RW (Readers Priority)Readers go firstWriters may starveHigh (many readers simultaneously)
Second RW (Writers Priority)Writers go firstReaders may starveLower (queued readers wait for writer)
Fair RW (No starvation)FIFO orderNoneMedium

Real-World Examples

  • Database system: Multiple SELECT queries run simultaneously; UPDATE needs exclusive access.
  • File system: Many processes can read a file; write operations need exclusive access.
  • Linux kernel: rwlock_t — read-write spinlock used in the kernel for exactly this pattern.
⚡ GATE
  • In the First RW solution, the first reader to arrive acquires rw_mutex (blocking all writers). The last reader to leave releases it.
  • Maximum readers simultaneously: Unlimited (only bounded by system resources) — as long as no writer holds rw_mutex.
  • GATE often asks about read_count and when exactly rw_mutex is acquired/released — trace carefully.
  • A writer waiting for all readers to finish → writers starve in First RW if readers keep arriving.

3.9 Classical Problem: Dining Philosophers

🎓 University
Dining Philosophers Problem (Dijkstra, 1965)
Five philosophers sit at a circular table. Between each pair of adjacent philosophers is one chopstick (5 chopsticks total). A philosopher alternates between thinking and eating. To eat, they need both the left and right chopstick. The challenge: design a protocol that prevents deadlock and starvation.

Why It's Hard

If every philosopher simultaneously picks up their left chopstick, all are waiting for the right one — deadlock. This is a model for any resource allocation problem with multiple processes and multiple resources.

Naive Broken Solution

Pseudocode — Deadlock-Prone Solution
Semaphore chopstick[5] = {1,1,1,1,1};

Philosopher i:
    while (true) {
        think();
        wait(chopstick[i]);          // pick up LEFT
        wait(chopstick[(i+1) % 5]);  // pick up RIGHT → DEADLOCK possible!
        eat();
        signal(chopstick[i]);
        signal(chopstick[(i+1) % 5]);
    }
⚠️
Deadlock Scenario: All 5 philosophers pick up their left chopstick simultaneously. Now each waits for right (= another's left). Circular wait → deadlock! All five are stuck forever.

Three Correct Solutions

Solution 1: Allow at most 4 philosophers at the table

Use a counting semaphore room = 4. At most 4 philosophers sit at once — by pigeonhole, at least one always gets both chopsticks.

Pseudocode — Room Semaphore Fix
Semaphore chopstick[5] = {1,1,1,1,1};
Semaphore room = 4;   // at most 4 at table

Philosopher i:
    wait(room);                      // enter table (max 4)
    wait(chopstick[i]);
    wait(chopstick[(i+1) % 5]);
    eat();
    signal(chopstick[(i+1) % 5]);
    signal(chopstick[i]);
    signal(room);                    // leave table

Solution 2: Asymmetric Solution

Odd-numbered philosophers pick up left chopstick first; even-numbered pick up right first. This breaks the circular wait condition.

Pseudocode — Asymmetric Pick Order
Philosopher i:
    if (i % 2 == 0) {              // Even: right first
        wait(chopstick[(i+1) % 5]);
        wait(chopstick[i]);
    } else {                       // Odd: left first
        wait(chopstick[i]);
        wait(chopstick[(i+1) % 5]);
    }
    eat();
    signal(chopstick[i]);
    signal(chopstick[(i+1) % 5]);

Solution 3: Monitor-Based (All-or-Nothing)

A philosopher only picks up chopsticks if both are available simultaneously. Uses a monitor with condition variables (covered next section).

States in Monitor Solution

StateMeaning
THINKINGNot hungry, not eating
HUNGRYWants to eat, waiting for both chopsticks
EATINGHas both chopsticks, currently eating

A philosopher transitions HUNGRY → EATING only when both neighbors are not EATING. This prevents deadlock (no circular wait — philosopher only holds chopsticks when both available). Starvation freedom is NOT automatically guaranteed — it requires a fair condition-variable queue or an explicit turn/priority rule.

⚡ GATE
  • The naive solution deadlocks when all pick left chopstick simultaneously — this is a circular wait.
  • The room semaphore (max 4) solution is the simplest correct fix — deadlock free by pigeonhole principle.
  • The asymmetric solution breaks deadlock by ensuring at least one philosopher picks in the opposite order, preventing circular wait.
  • GATE tip: "For N dining philosophers, initialize room semaphore to N−1 to prevent deadlock." For 5 philosophers: room = 4.
🔬 Advanced

Resource Allocation Graph Analogy

Each chopstick is a resource, each philosopher is a process. Deadlock = cycle in the Resource Allocation Graph. The dining philosophers problem is the canonical model for studying deadlock and resource allocation strategies in OS design.

Which Coffman Condition Does Each Solution Break?

SolutionCoffman Condition BrokenHow

Solution 4: Resource Hierarchy (Lower-Numbered First)

Each philosopher always picks up the lower-numbered chopstick first, then the higher-numbered one. Since resource ordering is total, circular wait cannot form. This is a general deadlock-prevention strategy used in database lock ordering.

Barrier Synchronization

Barrier, RCU, and Futex
Barrier Synchronization:
  All N threads must reach the barrier before any can proceed.
  Implementation: shared counter + mutex + condition variable.
  barrier_wait():
    lock(mutex);
    count++;
    if (count == N) { count = 0; broadcast(cv); }
    else { wait(cv, mutex); }
    unlock(mutex);
  Used in: parallel scientific computing, MapReduce phases.
  POSIX: pthread_barrier_init(), pthread_barrier_wait().

Read-Copy-Update (RCU) — Linux Kernel:
  Lock-free synchronization for read-mostly data structures.
  Readers: no locks at all (just read the pointer).
  Writers: create a new copy, update pointer atomically,
           wait for all current readers to finish (grace period),
           then free old copy.
  Performance: reads are essentially free; writes pay copy cost.
  Used for: routing tables, module lists, /proc data.

Futex (Fast Userspace muTEX) — Linux:
  Hybrid approach: fast path in userspace, slow path in kernel.
  If lock is uncontended: atomic compare-and-swap in userspace.
  If contended: futex() syscall to sleep in kernel wait queue.
  POSIX mutexes and pthread_cond on Linux are built on futexes.
  Avoids syscall overhead for the common uncontended case.

3.10 Classical Problem: Sleeping Barber

🎓 University
The Sleeping Barber Problem
A barbershop has 1 barber, 1 barber chair, and N waiting chairs.
• If there are no customers, the barber goes to sleep in the barber chair.
• When a customer arrives and the barber is sleeping, the customer wakes the barber up.
• If a customer arrives and the barber is busy cutting hair, the customer sits in an empty waiting chair.
• If all N waiting chairs are full, a newly arriving customer leaves the shop.

Why It's Hard

This problem tests the ability to coordinate multiple processes (customers) competing for limited resources (waiting chairs) and synchronizing with a consumer (the barber). Race conditions can lead to the barber sleeping forever, or customers sitting on the same chair, or customers leaving when chairs are actually empty.

Solution using Semaphores

We need three semaphores and one shared variable:

  • Semaphore customers = 0; — Counts waiting customers (signals the barber).
  • Semaphore barbers = 0; — Indicates barber is ready to cut hair (signals the customer).
  • Semaphore mutex = 1; — Protects the shared waiting counter.
  • int waiting = 0; — Shared counter of currently occupied waiting chairs.
Pseudocode — Sleeping Barber
int CHAIRS = N;       // Total waiting chairs
int waiting = 0;      // Customers currently waiting

Barber Process:
    while (true) {
        wait(customers);     // Go to sleep if no customers
        wait(mutex);         // Lock the waiting counter
        waiting--;           // Take one customer from waiting chairs
        signal(barbers);     // Tell the customer the barber is ready
        signal(mutex);       // Unlock the waiting counter
        cut_hair();          // Barber is cutting hair (outside mutex)
    }

Customer Process:
    wait(mutex);             // Lock the waiting counter
    if (waiting < CHAIRS) {
        waiting++;           // Take a waiting chair
        signal(customers);   // Wake up the barber (or add to queue)
        signal(mutex);       // Unlock the waiting counter
        wait(barbers);       // Wait for the barber to be ready
        get_haircut();       // Sit in the barber chair
    } else {
        signal(mutex);       // Unlock the waiting counter
        leave_shop();        // No chairs available, leave
    }
⚠️
Common Pitfall: The waiting counter is a shared variable and MUST be protected by mutex. If two customers check waiting < CHAIRS simultaneously without mutual exclusion, they might both increment it and exceed the physical chair limit.

3.11 Monitors & Condition Variables

🎓 University
Monitor (C.A.R. Hoare, 1974)
A monitor is a high-level synchronization construct — a programming language feature that bundles shared data, procedures that operate on it, and synchronization into one module. Only one process can be active inside the monitor at any time. The compiler automatically inserts mutual exclusion code — programmers don't manage locks manually.

Monitor Structure

Pseudocode — Monitor Syntax
monitor MonitorName {
    /* Shared variables — PRIVATE, only accessible via procedures */
    int shared_var;
    condition cond_var;   // condition variable for waiting/signalling

    /* Initialization */
    initialization_code() { shared_var = 0; }

    /* Procedures — only ONE can execute at a time inside monitor */
    procedure P1() {
        if (/* can't proceed */)
            cond_var.wait();    // release monitor, sleep
        // ... do work ...
        cond_var.signal();      // wake one waiting process
    }

    procedure P2() {
        // ...
    }
}

Condition Variables

Monitors use condition variables for processes to wait within the monitor without blocking other processes permanently. Two operations:

OperationEffect
x.wait()Suspends the calling process; releases the monitor lock so another can enter
x.signal()Resumes exactly ONE process waiting on x. If none waiting, has no effect (unlike semaphore V)
⚠️
Key Difference from Semaphores: condition.signal() on a condition variable with no waiters does nothing. Semaphore signal() always increments the value, which is remembered. Condition variables have no memory.

Hoare vs Mesa Semantics

SemanticsOn signal()Used in
Hoare (Signal-and-Wait)Signaller immediately yields to woken process; guarantees condition still holdsOriginal monitors
Mesa (Signal-and-Continue)Signaller continues; woken process re-checks condition in a while loopJava synchronized, POSIX condition vars, most real systems

Java Synchronized (Monitor Implementation)

Java — Built-in Monitor via synchronized
public class BoundedBuffer {
    private int[] buffer;
    private int count = 0, in = 0, out = 0;
    private int N;

    public BoundedBuffer(int size) {
        N = size; buffer = new int[N];
    }

    public synchronized void produce(int item) throws InterruptedException {
        while (count == N) wait();         // buffer full — sleep
        buffer[in] = item;
        in = (in + 1) % N;
        count++;
        notifyAll();                       // wake consumers
    }

    public synchronized int consume() throws InterruptedException {
        while (count == 0) wait();         // buffer empty — sleep
        int item = buffer[out];
        out = (out + 1) % N;
        count--;
        notifyAll();                       // wake producers
        return item;
    }
}

Note the while loops (not if) — this is Mesa semantics: condition must be re-checked after wakeup.

Monitor vs Semaphore

Error-prone?Very (wrong order = deadlock/race)Less (ME guaranteed by structure)
signal() with no waiterRemembered (value increments)Lost (no effect)

⚡ GATE
  • Monitors ensure mutual exclusion automatically — only one process inside the monitor at a time, guaranteed by the language/compiler.
  • condition.signal() wakes at most one process. If none waiting, nothing happens. This is the biggest conceptual difference from semaphore signal.
  • In Mesa semantics, always use while (not if) to re-check condition after wakeup — another process may have changed the condition before the woken process runs.
  • Java's synchronized + wait()/notify() is a direct implementation of monitors with Mesa semantics.

3.11 Sleeping Barber & Cigarette Smokers

🎓 University

The Sleeping Barber Problem (Dijkstra)

Sleeping Barber Problem
A barbershop has one barber, one barber chair, and N waiting chairs. If there are no customers, the barber sleeps. When a customer arrives: if the barber is sleeping, the customer wakes the barber; if the barber is busy and waiting chairs are available, the customer sits; if all chairs are full, the customer leaves.

Semaphore-Based Solution

Sleeping Barber — Semaphore Solution
Semaphores:
  customers = 0      // # customers waiting (counting sem)
  barbers   = 0      // # barbers ready to cut (0 or 1)
  mutex     = 1      // protects 'waiting' variable (binary sem)

Shared variable:
  int waiting = 0    // # customers in waiting room

BARBER process:                     CUSTOMER process:
  while (true) {                      wait(mutex);
    wait(customers);  // sleep if     if (waiting < N) {
                      // no customers   waiting++;
    wait(mutex);                        signal(customers); // wake barber
    waiting--;                          signal(mutex);
    signal(barbers);  // ready to cut   wait(barbers);     // wait for barber
    signal(mutex);                      get_haircut();
    cut_hair();                       } else {
  }                                     signal(mutex);
                                        leave();  // shop full!
                                      }
💡
Key Insight: The customers semaphore represents waiting customers. When 0, the barber sleeps on wait(customers). A new customer does signal(customers) to wake the barber. The barbers semaphore synchronizes: customer waits for barber to be ready.
🔬 Advanced

Cigarette Smokers Problem (Patil, 1971)

Three smokers and one agent. Each smoker has an infinite supply of one ingredient (tobacco, paper, or matches). To smoke, a smoker needs all three. The agent randomly places two ingredients on the table. The smoker who has the missing ingredient picks them up and smokes.

Cigarette Smokers — Solution Sketch
Agent:    signal(ingredient_A); signal(ingredient_B);
                      // places two of three ingredients

Smoker_C: // has ingredient C, needs A and B
  wait(ingredient_A);
  wait(ingredient_B);
  smoke();
  signal(agent);      // signal done

Challenge: Cannot solve with just three semaphores!
Need helper threads ("pushers") to detect which pair was placed
and signal the correct smoker. This is a subtle coordination problem.

This problem demonstrates the limitations of semaphores — some coordination patterns are extremely difficult to express correctly with semaphores alone.

Hoare vs Mesa Monitor Semantics — Worked Trace

signal() Behavior Comparison
Setup: monitor with condition variable cv.
  Process A holds the monitor, calls signal(cv).
  Process B is waiting on cv.

Hoare Semantics (original Hoare 1974):
  signal(cv) immediately transfers control to B.
  A is suspended (pushed onto a separate "signaler" queue).
  B runs inside the monitor. When B exits or waits, A resumes.
  Consequence: condition guaranteed true when B resumes.
  B can use:   if (condition_false) wait(cv);
  Pro: simple reasoning. Con: extra context switch.

Mesa Semantics (Lampson & Redell 1980, used in Java/POSIX):
  signal(cv) is a HINT — B is moved to ready queue.
  A continues running inside the monitor.
  B runs later, but condition may have changed!
  Consequence: B must RE-CHECK condition after waking.
  B MUST use: while (condition_false) wait(cv);
  Pro: simpler implementation. Con: spurious wakeups possible.

Worked Trace (Mesa, bounded buffer, size=1):
  Producer: lock(); while(count==1) wait(not_full); produce(); count++;
            signal(not_empty); unlock();
  Consumer: lock(); while(count==0) wait(not_empty); consume(); count--;
            signal(not_full); unlock();

  Step 1: Consumer calls wait(not_empty)  [count=0, consumer sleeps]
  Step 2: Producer produces, count=1, signals not_empty
  Step 3: Producer CONTINUES (Mesa!), unlocks
  Step 4: Another producer could run, fill buffer again
  Step 5: Consumer wakes, re-checks while(count==0) → count=1, proceeds
  Without while-loop: consumer might try to consume from empty buffer!

GATE rule: always use while() with Mesa monitors (Java/POSIX).
⚡ GATE

GATE Semaphore Numericals

Problem 1 — Semaphore Value Tracking
Q: S is a binary semaphore initialized to 1.
   Three processes P1, P2, P3 execute:
     P1: wait(S); CS1; signal(S);
     P2: wait(S); CS2; signal(S);
     P3: wait(S); CS3; signal(S);

   At most how many processes can be in CS simultaneously?

Answer: 1 (binary semaphore = mutex → mutual exclusion)

─────────────────────────────────────────
Q: S is a counting semaphore initialized to 3.
   5 processes execute wait(S) simultaneously.
   How many will proceed? How many will block?

Answer: 3 proceed (S goes 3→2→1→0), 2 block (S would go negative)
After two signal(S) calls: 2 blocked processes wake up.
Problem 2 — Process Interleaving
Q: S1=0, S2=0. What is printed?

  P1:                    P2:
    print("A");            wait(S1);
    signal(S1);            print("C");
    print("B");            signal(S2);
    wait(S2);
    print("D");

  - P1 prints "A", signals S1 (S1=1), prints "B"
  - P2 was blocked on wait(S1), now proceeds → prints "C", signals S2
  - P1's wait(S2) now succeeds → prints "D"

Guaranteed output: A must come before C (S1 enforces this)
                   C must come before D (S2 enforces this)
Constraints: A < C < D, and A < B. No fixed order between B and C.
Valid outputs: ABCD or ACBD (D is always last).
Problem 3 — Bounded Buffer Semaphore Values
Q: Bounded buffer with N=5 slots.
   empty=5, full=0, mutex=1.
   Producer produces 3 items, consumer consumes 1.
   What are the semaphore values?

Solution:
  Initial: empty=5, full=0, mutex=1
  Produce 3 items: empty=5→4→3→2, full=0→1→2→3
  Consume 1 item:  full=3→2, empty=2→3

  Final: empty=3, full=2, mutex=1

GATE Formula:
  empty + full = N (always, in steady state)
  empty = N - items_in_buffer
  full  = items_in_buffer

★ Unit 3 PYQ & Exam Prep

🎓 University Exams

Short Answer Questions (2–5 Marks)

Q1: State and explain the three requirements for a critical section solution.
  1. Mutual Exclusion: If Pi is in its CS, no other process can be in its CS.
  2. Progress: If no process is in the CS and some wish to enter, selection cannot be postponed indefinitely. Only processes not in remainder section participate.
  3. Bounded Waiting: A bound must exist on the number of times others enter the CS after a process has requested entry — prevents starvation.
Q2: Differentiate between a Semaphore and a Monitor.

A semaphore is a low-level integer-based OS primitive requiring the programmer to manually call wait/signal in the correct order. Errors cause deadlock or race conditions. A monitor is a high-level language construct that automatically enforces mutual exclusion — only one process can be active inside the monitor at a time. Monitors are safer and less error-prone. The key difference: semaphore signal always increments the value; monitor condition signal has no effect if no process is waiting.

Q3: What is a race condition? Give a real-world example.

A race condition occurs when two or more threads access shared data concurrently and the final result depends on the timing of execution. Example: two bank threads both read balance=1000, both add 500, both write 1500 — instead of the correct 2000. One deposit is lost because the write-modify-write sequence was not atomic.

Q4: Explain Peterson's Solution and prove it satisfies mutual exclusion.

Peterson's solution uses two variables: flag[2] (intent to enter) and turn (tie-breaker). Process i sets flag[i]=true and turn=j, then spins while flag[j] && turn==j.

Mutual Exclusion proof: Suppose both Pi and Pj are in the CS. Pi is in CS means either flag[j]==false OR turn==i. Similarly for Pj: flag[i]==false OR turn==j. Since both flags must be true (both wanted to enter), we need turn==i AND turn==j — impossible. Contradiction. So at most one can be in CS.

Long Answer Questions (10 Marks)

Q5: Solve the Producer-Consumer problem using semaphores. Show complete code with explanation.

Three semaphores: mutex=1 (mutual exclusion), empty=N (empty slots), full=0 (filled slots).

Producer: wait(empty) → wait(mutex) → add_item → signal(mutex) → signal(full)

Consumer: wait(full) → wait(mutex) → remove_item → signal(mutex) → signal(empty)

Key rule: Resource semaphore (empty/full) must always be acquired BEFORE mutex. Reversing this order creates deadlock when buffer is full/empty.

Q6: Describe the Dining Philosophers problem. What are the different solutions to prevent deadlock?

Five philosophers share 5 chopsticks at a round table. Each needs two adjacent chopsticks to eat. Naive solution (pick left then right) causes deadlock when all pick left simultaneously.

Solutions: (1) Allow max 4 philosophers at table (room semaphore=4) — by pigeonhole, at least one always gets both. (2) Asymmetric: odd philosophers pick left first, even pick right first — breaks circular wait. (3) All-or-nothing: pick both chopsticks only when both are available (monitor-based).

⚡ GATE & Competitive Exams

GATE Master MCQs

1. Which of the following is NOT a requirement for a solution to the critical section problem?
  1. Mutual Exclusion
  2. Progress
  3. Bounded Waiting
  4. Fairness (strict alternation)
View Answer & Explanation
Answer: D (Fairness/strict alternation)
The three formal requirements are Mutual Exclusion, Progress, and Bounded Waiting. Strict alternation (fairness) is NOT one of them — in fact, it may violate Progress (if Pi must wait for Pj even when Pj is in its remainder section).
2. A semaphore is initialized to 7. After 12 wait() operations and 4 signal() operations, the semaphore value is:
  1. −1
  2. 3
  3. −5
  4. 1
View Answer & Explanation
Answer: A (−1)
Initial value = 7. After 12 waits: 7 − 12 = −5. After 4 signals: −5 + 4 = −1. Since value is negative, |−1| = 1 process is currently blocked on this semaphore.
3. In Peterson's solution, which variable is responsible for ensuring bounded waiting?
  1. flag[i]
  2. flag[j]
  3. turn
  4. Both flag and turn
View Answer & Explanation
Answer: C (turn)
The turn variable acts as the tie-breaker and is the key to bounded waiting. When Pi sets turn = j, it gives Pj priority. After Pj executes its CS and sets flag[j] = false, Pi is guaranteed to enter on its next attempt. Waiting bound = 1.
4. Which of the following statements about monitors is CORRECT?
  1. Multiple processes can execute inside a monitor simultaneously
  2. A condition.signal() with no waiters increments a counter for future use
  3. Monitors automatically ensure mutual exclusion
  4. Monitors are a low-level hardware primitive
View Answer & Explanation
Answer: C
Only ONE process can be active inside a monitor at any time — mutual exclusion is automatically enforced by the compiler/runtime. (A) is wrong — only one process at a time. (B) is wrong — condition signal with no waiters does nothing; only semaphore signal increments. (D) is wrong — monitors are a high-level language construct.
5. In the Dining Philosophers problem with 5 philosophers, a counting semaphore room is used to limit the number of philosophers attempting to pick up chopsticks simultaneously. To what value should room be initialized to prevent deadlock?
  1. 5
  2. 4
  3. 3
  4. 2

Answer: B (4 = N−1)
With at most 4 philosophers at the table and 5 chopsticks, by the pigeonhole principle at least one philosopher can always acquire both chopsticks, complete eating, and release them. This breaks the circular wait condition. Initializing to 5 allows all 5 to pick up one chopstick each → deadlock. Initializing to 3 or 2 is unnecessarily restrictive (correct but reduces concurrency).

4 Additional GATE-Style MCQs

G1. Semaphore S is initialized to 3. Operations: wait(), wait(), wait(), wait(), signal(), signal(). Value of S = ?
A. 1   B. 0   C. -1   D. 2
Answer
A. 1. S: 3→2→1→0(block)→signal→1→signal→2. Wait… actually: 3-1=2, 2-1=1, 1-1=0, 0-1=-1(one blocked). signal: -1+1=0(wake one). signal: 0+1=1. Final S=1 (textbook convention). API convention: S=1 as well (unblocked process runs).
G2. In the First Readers-Writers problem, what is the maximum number of readers that can be in the CS simultaneously?
A. 1   B. N   C. Unlimited   D. read_count
Answer
C. Unlimited (bounded only by system resources). As long as no writer holds rw_mutex, any number of readers can enter. The first reader acquires rw_mutex; subsequent readers just increment read_count.
G3. In a monitor, if signal() is called on a condition variable with no waiting processes, what happens?
A. Value incremented   B. Nothing   C. Error   D. Process blocks
Answer
B. Nothing. Unlike semaphore signal (which increments and remembers), monitor condition variable signal is LOST if no process is waiting. This is the #1 conceptual difference tested between semaphores and monitors.
G4. TestAndSet-based mutual exclusion satisfies which of the following?
A. ME only   B. ME + Progress   C. ME + Progress + Bounded Waiting   D. None
Answer
B. ME + Progress. Simple TSL lock satisfies Mutual Exclusion (atomic read-modify) and Progress (if lock is free, a process will acquire it). But it does NOT guarantee Bounded Waiting — a process could be perpetually unlucky while others keep acquiring. Adding a waiting array solves this.

3.1 Concurrency & Race Conditions

🎓 University
Concurrency
Concurrency is the execution of multiple processes or threads in overlapping time periods. On a single-core CPU, concurrency is achieved via rapid context switching (interleaving). On multi-core CPUs, true parallelism is possible — multiple threads literally execute simultaneously.
ModeDefinitionExample
SequentialOne task at a time, complete before nextSingle-threaded program
ConcurrentMultiple tasks making progress, may interleaveOS time-sharing on 1 core
ParallelMultiple tasks literally running simultaneouslyMulti-core processor
Race Condition
A race condition occurs when two or more threads access shared data concurrently and the final result depends on the timing / order of execution. The outcome is non-deterministic — it "races" to see which thread wins.

Classic Example: counter++

Two threads both execute counter++. This single C statement compiles into 3 machine instructions:

Thread T1
LOAD R1, counter   ; R1 = 5
ADD  R1, 1         ; R1 = 6
STORE counter, R1  ; counter = 6
Thread T2
LOAD R1, counter   ; R1 = 5 ← STALE!
ADD  R1, 1         ; R1 = 6
STORE counter, R1  ; counter = 6 ← LOST!

Both read counter=5 before either stores. Both compute 6 and store 6. One increment is lost! Counter should be 7 but is 6.

⚠️
Key Insight: The problem is that counter++ is not atomic. An atomic operation is indivisible — it executes completely without interruption. Non-atomic operations on shared data are vulnerable to race conditions.
⚡ GATE
Interleaving Trace — Why counter++ Fails
Step  T1 instruction        T2 instruction        counter  R1(T1)  R1(T2)
────  ───────────────────  ───────────────────  ─────────  ──────  ──────
 1    LOAD R1, counter                           5         5       —
 2                          LOAD R1, counter     5         5       5
 3    ADD R1, 1                                  5         6       5
 4                          ADD R1, 1            5         6       6
 5    STORE counter, R1                          6         6       6
 6                          STORE counter, R1    6         6       6

Expected: 7.  Actual: 6.  One increment LOST.

GATE question: If 3 threads each execute x++ once (x=0 initially),
what are the possible final values of x?
Answer: x ∈ {1, 2, 3}.
  Minimum = 1 (all three read 0 before any store)
  Maximum = 3 (fully serialized — each reads the updated value)
  • A race condition requires: (1) shared data, (2) concurrent access, (3) at least one write operation.
  • Two concurrent reads with no writes → no race condition.
  • Race conditions are non-deterministic — may not appear every run, making them extremely hard to debug.

3.2 The Critical Section Problem

🎓 University
Critical Section
A segment of code that accesses shared resources and must not be executed by more than one process simultaneously. The critical section problem is to design a protocol that processes can use to cooperate safely.
Pseudocode — Generic CS Structure
while (true) {
    /* ENTRY SECTION */
    acquire_lock();         // Request permission to enter CS

    /* CRITICAL SECTION */
    // Access shared data — ONLY ONE PROCESS AT A TIME

    /* EXIT SECTION */
    release_lock();         // Signal that CS is free

    /* REMAINDER SECTION */
    // Non-critical work — can run concurrently
}

Three Requirements for a Valid Solution

01
Mutual Exclusion
If Pi is in its CS, no other process Pj can be in its CS at the same time. The most fundamental requirement.
02
Progress
If no process is in the CS and some want to enter, the selection of which process enters next cannot be postponed indefinitely. Only processes NOT in the Remainder Section participate in this decision.
03
Bounded Waiting
There must be a bound on the number of times other processes are allowed to enter the CS after a process has made a request — before that request is granted. Prevents starvation.
💡
Memory Trick: MPB — Mutual exclusion (max one in), Progress (process chosen eventually), Bounded waiting (bounded waits before entry).
⚡ GATE
Broken Solutions Analysis — Classic GATE Traps
ApproachME?Progress?BW?Why It Fails
Lock variable only❌✅✅Both read lock=0 simultaneously → both enter CS
Strict alternation (turn)✅❌✅P0 blocked even if P1 is in remainder section

GATE Trap: Strict alternation satisfies ME and Bounded Waiting but FAILS Progress. This is the most common trick question on CS requirements.

3.3 Peterson's Solution

🎓 University
Peterson's Solution (Gary Peterson, 1981)
A classic software-only solution to the CS problem for two processes. Uses two shared variables: flag[2] (indicates desire to enter) and turn (indicates whose turn it is when both want in).
C — Peterson's Solution for Process Pi (i=0 or 1, j=1-i)
/* ENTRY SECTION */
flag[i] = true;     // "I want to enter"
turn = j;           // "But I give priority to you first"
while (flag[j] == true && turn == j)
    ;               // Busy-wait (spin) until safe to enter

/* CRITICAL SECTION */
// ... access shared data ...

/* EXIT SECTION */
flag[i] = false;    // "I'm done, others may enter"

/* REMAINDER SECTION */
// ... non-critical code ...

Proof — Why All 3 Requirements Are Satisfied

RequirementProof
Mutual ExclusionFor both Pi and Pj to be in CS, we'd need turn==i AND turn==j simultaneously — impossible since turn is a single integer. Contradiction. ✅
ProgressIf Pj is not ready (flag[j]==false), Pi exits the while loop immediately. No indefinite postponement. ✅
Bounded WaitingAfter Pi sets turn=j and Pj finishes, Pj sets flag[j]=false, freeing Pi. Pi enters on its very next attempt. Wait bound = 1. ✅
⚠️
Modern Hardware Caveat: Peterson's solution is correct in theory but may fail on modern CPUs that reorder memory operations (out-of-order execution). It requires memory barriers / fences to work correctly on real hardware.
⚡ GATE
  • Peterson's is a software solution — no special hardware instructions (in theory).
  • Works for exactly 2 processes. For N processes → Lamport's Bakery Algorithm.
  • Uses busy-waiting (spinlock) — wastes CPU time while spinning.
  • turn variable ensures bounded waiting (bound = 1). flag[] ensures progress.
🔬 Advanced

Lamport's Bakery Algorithm (N processes)

Bakery Algorithm — N Processes
bool choosing[n] = {false};  // is process picking a number?
int  number[n]   = {0};      // ticket number (0 = not interested)

/* ENTRY for process i */
choosing[i] = true;
number[i] = max(number[0..n-1]) + 1;  // take a ticket
choosing[i] = false;

for (int j = 0; j < n; j++) {
    while (choosing[j]);  // wait if j is still picking
    while (number[j] != 0 &&
           (number[j] < number[i] ||
            (number[j] == number[i] && j < i)));
    // Wait while j has a smaller ticket, or same ticket + lower PID
}

/* CRITICAL SECTION */

/* EXIT */
number[i] = 0;  // "done" — give up ticket

// Tie-breaking: (number[i], i) tuples compared lexicographically
// Satisfies all 3 requirements for N processes
// Based on real bakery number machines — First Come First Served

Memory Ordering & Why Peterson Fails on Modern CPUs

Memory Barriers — Practical Guide
Problem: Modern CPUs + compilers REORDER memory operations.
  // Compiler may move flag=1 AFTER the critical section!
  flag = 1;          ← could be reordered
  critical_section();

Fixes:
  volatile int flag;              // prevents compiler reordering
  atomic_store(&flag, 1);         // C11 atomic — full guarantee
  asm volatile("" ::: "memory"); // GCC compiler fence

CPU-level fences (x86):
  mfence — full barrier: no load/store crosses the fence
  lfence — load fence: all loads before complete first
  sfence — store fence: all stores before commit first

ARM/RISC-V need stronger barriers (weaker memory model than x86).

GATE Rule: Peterson's and Bakery assume Sequential Consistency.
  On real hardware → use atomic operations or explicit fences.
  This is why OS kernels use spinlock_t with embedded barriers.

3.4 Synchronization Hardware — TSL & CAS

🎓 University

Modern CPUs provide special atomic hardware instructions that read and modify memory in a single, uninterruptible operation. These are the foundation of all practical synchronization primitives.

1. Test-and-Set (TSL)

C — TestAndSet (atomic hardware equivalent)
/* Executes ATOMICALLY — no interrupt possible between read and set */
boolean TestAndSet(boolean *target) {
    boolean rv = *target;   // 1. Read current value
    *target = true;         // 2. Set to true
    return rv;              // 3. Return OLD value
}

/* Usage: Spinlock with TSL */
boolean lock = false;  // shared, initially unlocked

// ACQUIRE:
while (TestAndSet(&lock))  // spin until lock was false (we got it)
    ;
/* CRITICAL SECTION */
lock = false;  // RELEASE

// Analysis:
//   If lock=false: TSL returns false (exit while) and sets lock=true → acquired
//   If lock=true:  TSL returns true (keep spinning) → already held by someone

2. Compare-and-Swap (CAS)

C — Compare-and-Swap (modern standard)
/* Atomic: if *value == expected, set to new_value. Return old value. */
int CompareAndSwap(int *value, int expected, int new_value) {
    int temp = *value;
    if (*value == expected)
        *value = new_value;
    return temp;  // return OLD value
}

/* Usage: lock=0 means free, lock=1 means held */
while (CompareAndSwap(&lock, 0, 1) != 0)
    ;   // spin until we atomically swap 0→1
/* CRITICAL SECTION */
lock = 0;  // release

/* C11 standard: __atomic_compare_exchange_n() or atomic_compare_exchange_strong() */
/* x86 instruction: CMPXCHG   ARM instruction: LDREX/STREX (LL/SC) */
FeatureTest-and-SetCompare-and-Swap
Works forBoolean locksGeneral integers, pointers
HW instructionx86: XCHGx86: CMPXCHG, ARM: LDREX/STREX
Lock-free data structuresLimitedYes — CAS enables lock-free stacks, queues
ABA ProblemN/AYes — solved with version tags
⚡ GATE
  • TSL and CAS both satisfy ME + Progress but simple implementations do NOT guarantee Bounded Waiting alone.
  • To add bounded waiting: use a waiting[] array — process adds itself, checks in order.
  • ABA Problem: Value changes A→B→A; CAS sees A and assumes no change — but it changed! Fix: use a version counter (stamped reference) alongside the value.
GATE Q: TestAndSet-based lock satisfies which requirements?
(a) ME only   (b) ME + Progress   (c) ME + Progress + BW   (d) None
Answer(b) ME + Progress. ME is guaranteed by the atomic read-modify. Progress is guaranteed — if lock is free, some process will acquire it. But Bounded Waiting is NOT guaranteed without a waiting array — a process could be unlucky indefinitely.

3.5 Mutex Locks & Spinlocks

🎓 University
Mutex Lock (Mutual Exclusion Lock)
A mutex provides mutual exclusion with ownership — only the thread that acquired the lock can release it. Two operations: acquire() and release(). Built on hardware atomic instructions.
C — POSIX Mutex
#include <pthread.h>

pthread_mutex_t lock = PTHREAD_MUTEX_INITIALIZER;
int shared_counter = 0;

void *increment(void *arg) {
    for (int i = 0; i < 1000000; i++) {
        pthread_mutex_lock(&lock);     // ACQUIRE — blocks if held
        shared_counter++;              // CRITICAL SECTION (safe!)
        pthread_mutex_unlock(&lock);   // RELEASE
    }
    return NULL;
}
/* Without mutex: result < 2,000,000 due to race conditions */
/* With mutex: always exactly 2,000,000 */
Spinlock vs Blocking Mutex
PropertySpinlockBlocking Mutex (Sleeping)
Waiting strategyBusy-waits (spins in loop)Blocks process — puts it to sleep
CPU usage while waiting100% (wastes CPU)0% (process is descheduled)
Wake-up latencyImmediate (no context switch)Higher (context switch overhead)
Best forShort CS, multi-core, OS kernelsLong CS, user-space applications
ExampleLinux kernel spinlock_tPOSIX pthread_mutex_t
💡
Golden Rule: Use spinlock when you expect lock held for microseconds on multi-core. Use blocking mutex when lock might be held for milliseconds — spinning would waste too much CPU. Never spinlock on single-core — holder can't run while spinner occupies the CPU → infinite spin!

Deadlock with Mutexes

C — Classic Deadlock Scenario
/* Thread 1 */                    /* Thread 2 */
lock(mutex_A);                    lock(mutex_B);
lock(mutex_B); // WAITS for B     lock(mutex_A); // WAITS for A
// → DEADLOCK! Both threads waiting for each other forever

/* Fix: Always acquire locks in the SAME ORDER across all threads */
/* Thread 1 */                    /* Thread 2 */
lock(mutex_A);                    lock(mutex_A);  // same order!
lock(mutex_B);                    lock(mutex_B);
// OR: use trylock() with rollback on failure
⚡ GATE
⚠️
Priority Inversion — GATE Classic (Mars Pathfinder):
High-priority task H blocks waiting for a mutex held by low-priority task L. Medium-priority task M preempts L. Now M runs before H — priority is effectively inverted.
Fix — Priority Inheritance: L temporarily inherits H's priority while holding the mutex. M cannot preempt L. L finishes, releases mutex, H runs.
Real case: Mars Pathfinder (1997) — NASA uploaded a one-line fix to Mars enabling priority inheritance on the shared bus mutex. Resets stopped immediately.
  • A mutex is owned — only the locker can unlock. A semaphore has no ownership semantics.
  • Spinlocks are inappropriate on single-core — if one thread is spinning, it prevents the lock holder (on the same core) from running and releasing the lock.

3.6 Semaphores — Binary & Counting

🎓 University
Semaphore (E.W. Dijkstra, 1965)
An integer variable accessed only through two atomic operations: wait() (P — proberen, "to test") and signal() (V — verhogen, "to increment"). Can be used for both mutual exclusion and ordering/signalling.
Pseudocode — Semaphore P & V (Textbook/Blocking version)
wait(S) {           // P — Decrement; if negative, BLOCK
    S.value--;
    if (S.value < 0) {
        add this process to S.list;
        block();    // put process to sleep
    }
}

signal(S) {         // V — Increment; if processes sleeping, WAKE one
    S.value++;
    if (S.value <= 0) {
        remove process P from S.list;
        wakeup(P);  // resume the sleeping process
    }
}

Key: When S.value < 0, |S.value| = number of processes currently BLOCKED
TypeInitial ValueUse Case
Binary Semaphore1Mutual exclusion (like a mutex but without ownership)
Counting SemaphoreN > 1Control access to a pool of N identical resources
Ordering Semaphore0Thread A waits; Thread B signals when event occurs → ordering guarantee
⚠️
Two Value Conventions — Know Both:
Textbook/Dijkstra: S.value can go negative. |S| when negative = # blocked processes.
POSIX API: S.value ≥ 0 always. Blocked waiters tracked in separate queue, value stays at 0.
For GATE: follow whichever convention the question uses. "Value after 3 waits on S=2" → textbook: S=−1 (one blocked). POSIX: S=0 (one queued).

Semaphore vs Mutex — Key Differences

PropertySemaphoreMutex
OwnershipNone — any thread can signalOwned — only locker can unlock
Value rangeAny integer (can be >1)Binary (locked/unlocked)
Use caseResource counting, ordering, signallingMutual exclusion only
signal() with no waiterValue incremented (remembered)Does nothing (mutex stays unlocked)
⚡ GATE
  • Binary semaphore initialized to 1 → mutual exclusion (but no ownership — any process can signal).
  • Counting semaphore initialized to 0 → ordering/signalling (waiter blocks until another process signals).
  • Semaphore initialized to K → at most K processes in CS simultaneously.
  • |S.value| when negative = number of blocked processes. Very frequent GATE computation.
GATE Q: Semaphore S=3. Five processes execute wait(S) simultaneously. How many block?
Answer3 proceed (S: 3→2→1→0), then 2 block (S: 0−1=−1, −1−1=−2). Final S=−2. |−2|=2 processes blocked.

3.7 Classical Problem: Producer–Consumer

🎓 University
Bounded-Buffer (Producer–Consumer) Problem
A producer places data items into a fixed-size shared buffer. A consumer removes items. Challenge: producer must wait if buffer is full; consumer must wait if buffer is empty; both cannot access the buffer simultaneously.
SemaphoreInitial ValueMeaning
mutex1Mutual exclusion on buffer access
emptyN (buffer size)Counts empty slots — producer waits when 0
full0Counts filled slots — consumer waits when 0
Pseudocode — Complete Producer-Consumer Solution
Semaphore mutex = 1, empty = N, full = 0;
int buffer[N], in = 0, out = 0;  // circular buffer indices

/* PRODUCER */
while (true) {
    item = produce_item();
    wait(empty);        // wait for an empty slot (COUNT DOWN)
    wait(mutex);        // enter critical section
    buffer[in] = item;
    in = (in + 1) % N;
    signal(mutex);      // exit critical section
    signal(full);       // notify consumer: one more full slot
}

/* CONSUMER */
while (true) {
    wait(full);         // wait for a filled slot (COUNT DOWN)
    wait(mutex);        // enter critical section
    item = buffer[out];
    out = (out + 1) % N;
    signal(mutex);      // exit critical section
    signal(empty);      // notify producer: one more empty slot
    consume_item(item);
}
⚠️
Critical Ordering Rule — Never Swap! Always wait(empty/full) BEFORE wait(mutex).
If swapped: Producer holds mutex when buffer is full, calls wait(empty) — but Consumer is blocked on wait(mutex) which Producer holds. DEADLOCK!
⚡ GATE
Bounded Buffer Invariants
empty + full = N     (always, in steady state)
empty = N − items_in_buffer
full = items_in_buffer
GATE Q: N=5 buffer. Produce 3, consume 1. Final semaphore values?
AnswerInitial: empty=5, full=0, mutex=1.
Produce 3: empty=5→4→3→2, full=0→1→2→3.
Consume 1: full=3→2, empty=2→3.
Final: empty=3, full=2, mutex=1. Check: 3+2=5=N ✅
GATE Q: The following code — does it deadlock?
Producer: wait(mutex); wait(empty); add; signal(mutex); signal(full);
AnswerYes, deadlocks when buffer is full. Producer holds mutex, then waits on empty (which is 0). Consumer needs mutex to consume, but Producer holds it. Both stuck forever.

3.8 Classical Problem: Readers–Writers

🎓 University
Readers–Writers Problem
Multiple Readers (read-only) and Writers (read+write) share a database. Multiple readers can read simultaneously. A writer needs exclusive access — no readers or other writers during write.

First Solution — Readers Priority

No reader waits unless a writer already has access. Writers may starve.

Pseudocode — First Readers-Writers
Semaphore rw_mutex = 1;  // Exclusive access — writers wait here
Semaphore mutex   = 1;  // Protects read_count
int read_count    = 0;  // Number of ACTIVE readers

/* READER */
wait(mutex);
    read_count++;
    if (read_count == 1)   // First reader blocks ALL writers
        wait(rw_mutex);
signal(mutex);

/* --- READ DATA --- */

wait(mutex);
    read_count--;
    if (read_count == 0)   // Last reader releases writers
        signal(rw_mutex);
signal(mutex);

/* WRITER */
wait(rw_mutex);    // Exclusive access — waits for ALL readers to finish
/* --- WRITE DATA --- */
signal(rw_mutex);
Three Variants Compared
VariantPriorityStarvation RiskConcurrency
First RW (Readers Priority)Readers firstWriters may starveHigh (unlimited concurrent readers)
Second RW (Writers Priority)Writers firstReaders may starveLower (queued readers wait for writer)
Fair RW (No starvation)FIFO orderNoneMedium — Linux rwlock_t
⚡ GATE
  • First reader acquires rw_mutex (blocking all writers). Last reader releases it.
  • Maximum simultaneous readers: unlimited (as long as no writer holds rw_mutex).
  • Writers starve in First RW if readers keep arriving.
  • GATE often traces read_count and asks exactly when rw_mutex is acquired/released.
GATE Q: In First RW, 3 readers and 1 writer all arrive simultaneously. Which runs first and why?
AnswerIt depends on which reader executes first. Whichever reader gets mutex first will acquire rw_mutex (since read_count goes 0→1, first reader blocks the writer). The writer waits on rw_mutex until all 3 readers finish. Readers run concurrently; writer runs after all readers done.

3.9 Classical Problem: Dining Philosophers

🎓 University
Dining Philosophers Problem (Dijkstra, 1965)
Five philosophers at a circular table. Five chopsticks, one between each pair. To eat, a philosopher needs both adjacent chopsticks. Challenge: design a protocol that prevents deadlock and starvation.

Naive Broken Solution → Deadlock

Deadlock-Prone — All Pick Left Simultaneously
Semaphore chopstick[5] = {1,1,1,1,1};

Philosopher i:
    while (true) {
        think();
        wait(chopstick[i]);          // pick up LEFT
        wait(chopstick[(i+1) % 5]);  // pick up RIGHT → DEADLOCK possible!
        eat();
        signal(chopstick[i]);
        signal(chopstick[(i+1) % 5]);
    }
// DEADLOCK: All 5 pick up left → each waits for right (held by neighbour)
// → CIRCULAR WAIT → all stuck forever

Solution 1 — Room Semaphore (N−1)

Fix: Allow at most N−1 philosophers at table
Semaphore chopstick[5] = {1,1,1,1,1};
Semaphore room = 4;   // At most 4 at table (= N−1)

Philosopher i:
    wait(room);                      // enter table (max 4)
    wait(chopstick[i]);
    wait(chopstick[(i+1) % 5]);
    eat();
    signal(chopstick[(i+1) % 5]);
    signal(chopstick[i]);
    signal(room);                    // leave table

// By pigeonhole: 4 philosophers + 5 chopsticks → at least one always gets both

Solution 2 — Asymmetric Pickup Order

Fix: Break circular wait with opposite pickup order
Philosopher i:
    if (i % 2 == 0) {              // Even: pick RIGHT first
        wait(chopstick[(i+1) % 5]);
        wait(chopstick[i]);
    } else {                       // Odd: pick LEFT first
        wait(chopstick[i]);
        wait(chopstick[(i+1) % 5]);
    }
    eat();
    signal(chopstick[i]);
    signal(chopstick[(i+1) % 5]);

// Not all philosophers request in same cyclic order → circular wait impossible

Solution 3 — Resource Hierarchy (Lower # First)

Always pick up the lower-numbered chopstick first, then higher-numbered. Total resource ordering prevents circular chains. General deadlock-prevention strategy used in DB lock ordering.

⚡ GATE
  • Naive solution deadlocks via circular wait (one of the four Coffman conditions).
  • Room semaphore = N−1 for N philosophers. For 5 philosophers: room=4.
  • Deadlock freedom does not imply starvation freedom — a philosopher could be unlucky repeatedly.
🔬 Advanced

SolutionCoffman Condition BrokenHow

Barrier Synchronization, RCU & Futex

Advanced Sync Primitives
/* Barrier Synchronization */
// All N threads must reach barrier before any proceed
// pthread_barrier_init(&b, NULL, N); pthread_barrier_wait(&b);
// Used in: parallel scientific computing, MapReduce phases

/* Read-Copy-Update (RCU) — Linux Kernel */
// Lock-free for read-mostly data structures
// Readers: no locks — just read the pointer
// Writers: create copy, update pointer atomically, wait for grace period, free old
// Performance: reads are essentially free; writes pay copy cost
// Used for: routing tables, module lists, /proc data

/* Futex (Fast Userspace muTEX) — Linux */
// Hybrid: fast path in userspace, slow path in kernel
// Uncontended: atomic CAS in userspace — NO syscall
// Contended: futex() syscall to sleep in kernel wait queue
// POSIX mutexes and pthread_cond on Linux are built on futexes
// Benefit: avoids syscall overhead for the common uncontended case

3.10 Classical Problem: Sleeping Barber

🎓 University
Sleeping Barber Problem (Dijkstra)
One barber, one barber chair, N waiting chairs. If no customers: barber sleeps. Customer arrives: if barber sleeping → wake barber; if barber busy and chair available → sit; if all chairs full → leave.
Pseudocode — Semaphore Solution
Semaphore customers = 0;  // # customers waiting (counting)
Semaphore barbers   = 0;  // # barbers ready to cut (0 or 1)
Semaphore mutex     = 1;  // protects 'waiting' variable
int waiting = 0;          // shared — # customers in waiting room
int CHAIRS  = N;          // total waiting chairs

/* BARBER */                          /* CUSTOMER */
while (true) {                         wait(mutex);
    wait(customers);  // sleep if      if (waiting < CHAIRS) {
                      // no one          waiting++;
    wait(mutex);                         signal(customers); // wake barber
    waiting--;                           signal(mutex);
    signal(barbers);  // ready!          wait(barbers);     // await barber
    signal(mutex);                       get_haircut();
    cut_hair();                        } else {
}                                        signal(mutex);
                                         leave_shop();  // shop full!
                                       }
⚠️
Common Pitfall: The waiting counter MUST be protected by mutex. Without it, two customers could both check waiting < CHAIRS simultaneously and both increment — exceeding the physical chair count.
💡
Key Insight: customers semaphore = waiting customers. When 0, barber sleeps on wait(customers). New customer does signal(customers) to wake barber. barbers semaphore synchronises: customer wait(barbers) until barber is ready.
🔬 Advanced

Cigarette Smokers Problem (Patil, 1971)

Three smokers, one agent. Each smoker has infinite supply of one ingredient (tobacco, paper, or matches). Agent randomly places two ingredients on the table. The smoker with the missing ingredient picks them up and smokes.

Hoare vs Mesa Monitor Semantics — Worked Trace

signal() Behavior: Hoare vs Mesa
Setup: monitor with condition variable cv.
  Process A holds monitor, calls signal(cv).
  Process B is waiting on cv.

Hoare (original, 1974):
  signal() immediately transfers control to B. A is suspended.
  B runs inside monitor. When B exits or waits, A resumes.
  Guarantee: condition is still true when B resumes.
  B can safely use:   if (condition) { wait(cv); }
  Con: extra context switch overhead.

Mesa (Lampson & Redell 1980 — used in Java, POSIX):
  signal() is a HINT — B moves to ready queue.
  A continues running. B runs later (condition may have changed!)
  B MUST use:         while (condition) { wait(cv); }
  Pro: simpler implementation. Con: spurious wakeups possible.

GATE Rule: Always use while() with Mesa semantics (Java/POSIX).
  Java: synchronized + wait()/notify() = Mesa monitors.
  POSIX: pthread_cond_wait() = Mesa semantics.

3.11 Monitors & Condition Variables

🎓 University
Monitor (C.A.R. Hoare, 1974)
A high-level synchronization construct — a module that bundles shared data, procedures, and synchronization. Only one process can be active inside the monitor at any time. The compiler automatically enforces mutual exclusion — programmers don't manage locks manually.
Pseudocode — Monitor Syntax
monitor MonitorName {
    /* Shared variables — PRIVATE, only accessible via procedures */
    int shared_var;
    condition cond_var;  // condition variable for waiting/signalling

    initialization_code() { shared_var = 0; }

    /* Procedures — only ONE can execute inside monitor at a time */
    procedure P1() {
        while (/* can't proceed */)
            cond_var.wait();    // release monitor, sleep
        // ... do work ...
        cond_var.signal();      // wake ONE waiting process
    }
}
Condition Variable Operations
OperationEffect
x.wait()Suspends calling process; releases monitor lock so another can enter
x.signal()Resumes exactly ONE process waiting on x. If none waiting: does nothing (unlike semaphore V!)
x.broadcast()Wakes ALL processes waiting on x (used in Java notifyAll())
⚠️
Critical Difference from Semaphores: condition.signal() with no waiters does NOTHING. Semaphore signal() always increments the value (remembered). Condition variables have no memory.

Java Synchronized — Monitor in Practice

Java — Bounded Buffer with Monitor (Mesa semantics)
public class BoundedBuffer {
    private int[] buffer;
    private int count = 0, in = 0, out = 0, N;

    public BoundedBuffer(int size) { N = size; buffer = new int[N]; }

    public synchronized void produce(int item) throws InterruptedException {
        while (count == N) wait();   // buffer full — sleep (Mesa: use while!)
        buffer[in] = item;
        in = (in + 1) % N;
        count++;
        notifyAll();                 // wake consumers
    }

    public synchronized int consume() throws InterruptedException {
        while (count == 0) wait();   // buffer empty — sleep
        int item = buffer[out];
        out = (out + 1) % N;
        count--;
        notifyAll();                 // wake producers
        return item;
    }
}
// synchronized = only one thread inside any synchronized method at a time
// wait() releases the lock and sleeps; notifyAll() wakes all waiters
// while() (not if!) because Mesa semantics: condition must be re-checked

Barrier Synchronisation

A barrier is a synchronisation point where all threads in a group must arrive before any of them can proceed. Used extensively in parallel scientific computing (e.g., a parallel simulation where every thread must finish computing the current time step before any starts the next).

// Barrier Synchronization
// All N threads must reach the barrier before any can proceed.
// Implementation: shared counter + mutex + condition variable.

typedef struct {
    int       count;   // How many have arrived
    int       total;   // Total threads (N)
    pthread_mutex_t lock;
    pthread_cond_t  cv;
} Barrier;

void barrier_wait(Barrier* b) {
    pthread_mutex_lock(&b->lock);
    b->count++;
    if (b->count == b->total) {
        b->count = 0;                      // Reset for reuse
        pthread_cond_broadcast(&b->cv);    // Wake ALL threads
    } else {
        while (b->count != 0)
            pthread_cond_wait(&b->cv, &b->lock);  // Wait for broadcast
    }
    pthread_mutex_unlock(&b->lock);
}

// POSIX provides this as: pthread_barrier_wait()

RCU — Read-Copy-Update

RCU is a synchronisation mechanism optimised for read-heavy workloads. The core insight: readers never block — they run completely lock-free. Writers work by making a copy, modifying it, and atomically publishing the new version. Old versions are reclaimed only after all ongoing readers finish.

// RCU — Read-Copy-Update (Linux kernel mechanism)
// Readers: completely lock-free, zero overhead
// Writers: copy → modify → publish atomically

// READER:
rcu_read_lock();              // Mark start of read-side critical section
p = rcu_dereference(global_ptr);  // Safely load pointer
if (p) use(p->data);         // Use data — guaranteed not freed
rcu_read_unlock();            // Mark end of read-side

// WRITER:
new = kmalloc(sizeof(*new), GFP_KERNEL);
*new = *old;                  // Copy
new->field = new_value;       // Modify copy
rcu_assign_pointer(global_ptr, new);  // Publish atomically
synchronize_rcu();            // Wait for all active readers to finish
kfree(old);                   // NOW safe to free old version

// Key properties:
// - Readers: O(1), no locks, no cache line bouncing
// - Writers: O(readers) wait, but readers never stall writers
// - Used for: routing tables, module lists, file system dentries

Futex — Fast Userspace Mutex

// Futex (Fast Userspace muTEX) — Linux's secret weapon
// Key insight: In the UNCONTENDED case (common), NO syscall needed.
// Only goes to kernel when blocking is actually required.

// Userspace futex word (shared memory integer):
int futex_var = 1;  // 1=unlocked, 0=locked, -1=contended

void futex_lock(int* futex) {
    // Fast path: CAS from 1→0 (uncontended)
    if (__sync_val_compare_and_swap(futex, 1, 0) == 1)
        return;  // Got lock without syscall!
    
    // Slow path: must block — call kernel
    while (*futex != 1) {
        *futex = -1;
        syscall(SYS_futex, futex, FUTEX_WAIT, -1, NULL, NULL, 0);
    }
}

void futex_unlock(int* futex) {
    if (__sync_fetch_and_add(futex, 1) == -1) {
        // Someone was waiting — wake them
        *futex = 1;
        syscall(SYS_futex, futex, FUTEX_WAKE, 1, NULL, NULL, 0);
    }
}

// pthreads on Linux internally use futex for all mutex operations

Q: What is the key advantage of a Futex over a traditional mutex?
(a) It works across processes   (b) It requires no kernel involvement at all   (c) It avoids syscall overhead in the common uncontended case   (d) It automatically handles priority inversion
Answer(c) — A Futex performs a userspace CAS operation first. If the mutex is uncontended (common case), the lock is acquired without any system call. The kernel is only invoked on the slow path when a thread must actually block. Traditional mutexes always involve a syscall.
⚡ GATE Traps on Advanced Sync:
• Concurrent ≠ Parallel: A concurrent program on 1 CPU core is NOT parallel.
• RCU readers are NOT wait-free globally — they can stall writers via synchronize_rcu(), but readers themselves never block.
• Barrier reuse: After a broadcast, the barrier count resets. A barrier can be used for multiple phases — this is called a "reusable barrier" or "cyclic barrier".
🔬 Seqlock (Sequence Lock): A seqlock is a Linux kernel primitive for data that is read very frequently but written rarely (e.g., system time). Writers increment a sequence number (making it odd), update data, then increment again (making it even). Readers read the sequence number before and after — if it changed or was odd during their read, they retry. No reader lock needed. Used in gettimeofday() on Linux — reading the system clock takes ~4ns because no kernel lock is needed.

// Bounded Buffer using MONITOR (Java-style)
monitor BoundedBuffer {
    int buffer[N];
    int count = 0, in = 0, out = 0;
    condition not_full, not_empty;  // Condition variables

    void produce(int item) {
        if (count == N)
            wait(not_full);         // Auto releases monitor lock while waiting
        buffer[in] = item;
        in = (in + 1) % N;
        count++;
        signal(not_empty);         // Wake one consumer
    }

    int consume() {
        if (count == 0)
            wait(not_empty);
        int item = buffer[out];
        out = (out + 1) % N;
        count--;
        signal(not_full);
        return item;
    }
}

// Same using SEMAPHORES (3 semaphores needed):
semaphore mutex = 1;     // Mutual exclusion
semaphore empty = N;     // Count of empty slots
semaphore full = 0;      // Count of full slots

void producer() {
    P(empty);   // Wait for space
    P(mutex);   // Enter critical section
    // ... add item ...
    V(mutex);   V(full);  // Exit CS, signal consumer
}

void consumer() {
    P(full);    // Wait for item
    P(mutex);
    // ... remove item ...
    V(mutex);   V(empty);
}
⚠️ Critical Difference — Signal Semantics:
V() on semaphore: If no thread is waiting, the value is incremented and "stored" — the signal is NOT lost. A future P() will immediately succeed.
signal() on condition variable: If no thread is waiting on that condition, the signal is LOST — it has no effect. This is why a thread must re-check the condition after waking (use while, not if).
Q (DU 2023): What is the advantage of a monitor over a semaphore?
AnswerMonitors provide structured, compiler-enforced synchronisation. With semaphores, the programmer must manually ensure every P() has a matching V() and place them correctly — a single mistake causes deadlock or a race condition. Monitors automatically enforce mutual exclusion (only one thread inside at a time) and provide built-in condition variables for waiting. The monitor structure makes it impossible to forget to release the lock — the compiler handles it when the procedure returns.
Q: Why must condition variable wait() always be called inside a while() loop, not an if() statement?
AnswerBecause of spurious wakeups. A thread can wake from wait() without the condition being true (e.g., another thread was signalled, or the OS woke it for scheduling reasons). If you use if(), the thread assumes the condition is true and proceeds incorrectly. With while(), the thread re-checks the condition after every wakeup and only proceeds when it's genuinely satisfied. POSIX officially documents that pthread_cond_wait() may return spuriously.
GATE Concept: A semaphore with initial value 1 is used to protect a critical section. After one P() operation, what happens to its value, and can it go negative?
AnswerAfter one P() by Process A, value becomes 0. If Process B also does P() while value is 0, value becomes -1 (in the counting model). The absolute value of a negative semaphore represents the number of processes waiting. V() increments: from -1 to 0 (wakes one waiter), from 0 to 1 (no waiter, just increments). This is the correct model to use for GATE questions involving "how many processes are blocked on this semaphore?"

★ Unit 3 PYQ & Exam Prep

🎓 University Exams

Short Answer Questions (2–5 Marks)

Q1: State the three requirements for a critical section solution.
  1. Mutual Exclusion: If Pi is in its CS, no other Pj can be in its CS simultaneously.
  2. Progress: If no process is in CS and some wish to enter, selection cannot be postponed indefinitely. Only processes not in remainder section participate.
  3. Bounded Waiting: A bound must exist on the number of times other processes enter CS after a process requests entry — prevents starvation.
Q2: Differentiate Semaphore vs Monitor.

A semaphore is a low-level integer OS primitive. Programmer manually calls wait/signal in the correct order — errors cause deadlock or races. signal() always increments value (remembered even if no waiter).

A monitor is a high-level language construct. Mutual exclusion is enforced automatically by the compiler. condition.signal() with no waiter does nothing (not remembered). Monitors are safer and less error-prone.

Q3: What is a race condition? Give a real-world example.

A race condition occurs when the final result of concurrent access to shared data depends on the timing of execution. Example: two bank threads both read balance=1000, both add 500, both write 1500 — instead of the correct 2000. One deposit is lost because the read-modify-write sequence was not atomic.

Q4: Explain Peterson's Solution and prove it satisfies Mutual Exclusion.

Peterson's uses flag[i] (intent to enter) and turn (tie-breaker). Process i sets flag[i]=true, turn=j, then spins while flag[j] && turn==j.

ME proof: Suppose both Pi and Pj are in CS simultaneously. Pi in CS means flag[j]==false OR turn==i. Pj in CS means flag[i]==false OR turn==j. Since both entered CS, both flags were true. So we need turn==i AND turn==j — impossible. Contradiction. ∎

Q5: What is Priority Inversion? How is it solved?

Priority inversion: High-priority task H blocks waiting for mutex held by low-priority task L. Medium-priority M preempts L (since M > L). H is effectively stuck behind M — priority inverted.

Solutions: (1) Priority Inheritance — L temporarily gets H's priority while holding mutex. M can't preempt. (2) Priority Ceiling — mutex has ceiling priority = highest priority of any user. Lock holder always runs at ceiling. (3) Lock-free data structures using CAS — no locks, no inversion.

Famous example: Mars Pathfinder (1997) experienced repeated resets until NASA enabled priority inheritance on the bus mutex.

Long Answer Questions (10 Marks)

Q6: Solve Producer-Consumer with semaphores. Show complete pseudocode with explanation of each semaphore.

Three semaphores: mutex=1 (mutual exclusion on buffer), empty=N (count of empty slots), full=0 (count of filled slots).

Producer: wait(empty) → wait(mutex) → buffer[in]=item; in=(in+1)%N → signal(mutex) → signal(full)

Consumer: wait(full) → wait(mutex) → item=buffer[out]; out=(out+1)%N → signal(mutex) → signal(empty)

Critical rule: Resource semaphore (empty/full) MUST be acquired BEFORE mutex. Reversing causes deadlock when buffer is full (producer holds mutex, waits on empty, but consumer can't acquire mutex to consume).

Invariant: empty + full = N at all times.

Q7: Describe Dining Philosophers and all valid solutions to prevent deadlock.

Five philosophers share 5 chopsticks. Each needs both adjacent chopsticks to eat. Naive solution (pick left then right) deadlocks when all pick left simultaneously — circular wait.

Solution 1 — Room semaphore (N−1=4): At most 4 philosophers sit at once. By pigeonhole, at least one always gets both. Breaks circular wait.

Solution 2 — Asymmetric: Odd philosophers pick left first; even pick right first. Breaks circular ordering of resource requests.

Solution 3 — Resource hierarchy: Always pick lower-numbered chopstick first. Total ordering prevents circular chains.

Solution 4 — All-or-nothing (monitor): Philosopher picks up both chopsticks atomically or waits with none. Breaks hold-and-wait condition.

Q8: Explain First Readers-Writers problem. Who can starve and why?

Multiple readers can read simultaneously; a writer needs exclusive access. The first reader acquires rw_mutex (blocking all writers). Subsequent readers just increment read_count. The last reader (read_count → 0) releases rw_mutex.

Writers starve because: as long as at least one reader is active, rw_mutex is held. If readers keep arriving in a steady stream, read_count never hits 0, so rw_mutex is never released for the writer. The writer waits indefinitely.

Fix (Second RW): Writers block new readers from entering once a writer is waiting. This causes readers to starve instead.
